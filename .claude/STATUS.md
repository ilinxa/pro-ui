## Roadmap

Next candidates, ordered by team utility:

1. `data/stat-card` — value + label + delta + sparkline. Universal in dashboards.
2. `feedback/empty-state` — icon + title + body + primary action.
3. `forms/multi-select` — combobox with tag chips (shadcn has Command, no real multi-select).
4. `layout/page-header` — title + breadcrumbs + actions slot.
5. `feedback/notification-feed` — grouped, time-bucketed, read/unread.
6. `navigation/command-palette` — cmd+k, grouped results.
7. `media/dropzone` — drag-drop + progress + previews.

**`rich-card` v0.2 follow-ups** (deferred from v0.1.0, by phase per [rich-card-procomp-plan.md](../docs/procomps/rich-card-procomp/rich-card-procomp-plan.md)): v0.2 inline editor (add/edit/remove fields/cards/predefined-key elements, granular change events, dirty tracking, explicit selection); v0.3 drag-drop (2 scopes), bulk ops, per-level/per-card/per-key permissions, custom predefined-key registration, virtualization if needed; v0.4 sync validation hooks + per-commit undo/redo; v0.5 (deferred indefinitely, separate companion) markdown ↔ JSON adapter. The v0.1 API surface is designed so each phase adds props rather than changing them.

**`workspace` v0.2 follow-ups** (deferred from v0.1.0): detach-to-OS-window for multi-monitor users; touch / pen gesture support; aligned-edge linked resize (multiple edges move together); DnD between areas; cross-breakpoint transition animation. The v0.1 API surface is designed so v0.2 doesn't break it.

## Open decisions / TODOs

- **Chart palette still neutral:** `--chart-1..5` are greyscale. Define a lime-leaning analogous palette (e.g. lime → emerald → teal → cyan → blue) when the first chart-heavy component ships.
- **Site nav has only "Components":** add Docs / Blog / GitHub when those sections exist. Mobile menu (sheet/drawer) is currently unnecessary at this nav size — revisit when 4+ items.
- **Reserved-but-unrendered meta fields:** `RegistryEntry.examples`, `ComponentMeta.thumbnail`, `ComponentMeta.subcategory`, `ComponentMeta.related` exist on the types but render nowhere yet. Decide whether to wire any (each adds detail-page surface) or remove them.
- **`alpha` and `beta` badges look identical:** the hard-coded mapping in [components/page.tsx:78-83](../src/app/components/page.tsx#L78) and [[slug]/page.tsx:56-62](../src/app/components/[slug]/page.tsx#L56) only assigns unique colors to `stable` (default) and `deprecated` (destructive); both `alpha` and `beta` render `secondary`. If we want `beta` distinguishable, that's a 2-line tweak in two pages.
- **Footer version is hardcoded** (`v0.1.0`) in `site-footer.tsx`. Lift to a single source (`package.json` or a constant) when version starts changing.
- **Lime contrast pattern is dark-on-lime.** Both modes use near-black `--primary-foreground`. Don't switch to white — contrast fails. If a use case really wants white text on a green action, use a darker green (e.g. `oklch(0.45 0.18 142)` forest) and pair white text — but that becomes a separate token, not `--primary`.
- **MDX for usage docs:** currently `usage.tsx`. Consider migrating to `usage.mdx` once we cross ~5 components with prose-heavy guidance.
- **Public registry build:** no `pnpm dlx shadcn build` config or `public/r/*.json` output yet. Defer until publish phase.
- **NPM publish artifacts:** no `tsup`/`rollup`, no `package.json` exports map. One-day job at publish time.
- **Test debt — `workspace`:** workspace ships v0.1.0 without unit tests because no test runner is wired in this repo. The pure modules in `src/registry/components/layout/workspace/lib/` (`tree.ts`, `reducer.ts`, `geometry.ts`) are written to be testable in isolation when Vitest lands. Verification for v0.1.0 is demo-driven (the dummy registry's `counter` component proves Q3 state preservation across split visually). Land Vitest as a separate STATUS decision before workspace ships v0.2.
- **Test debt — `rich-card`:** same posture. The pure modules in `src/registry/components/data/rich-card/lib/` (`parse.ts`, `serialize.ts`, `classify-key.ts`, `infer-type.ts`, `reducer.ts`) are pure functions and trivially testable when Vitest lands. The single highest-value test is a **round-trip property test** — random `RichCardJsonNode` → `parse` → `serialize` → `parse` → assert tree-equal — which would lock down the Q3 fixed-point contract. v0.1.0 verification is demo-driven via the three demos in [`dummy-data.ts`](../src/registry/components/data/rich-card/dummy-data.ts) and the manual "Read canonical JSON" button in [demo.tsx](../src/registry/components/data/rich-card/demo.tsx).

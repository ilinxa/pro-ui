# todo-tree v0.1.3 — dnd-kit SSR hydration mismatch hotfix

## Summary

Same-day hotfix after v0.1.2 shipped. Browser console flooded with hydration warnings on every grip button:

```
+ aria-describedby="DndDescribedBy-2"
- aria-describedby="DndDescribedBy-1"
```

…with a clear `-N → +2N` ratio pattern across all 7 `<TodoTree>` demos on the detail page.

## Root cause

`@dnd-kit/core`'s `<DndContext>` auto-allocates its `DndDescribedBy-N` ID from a **module-level counter** (`uniqueId++` inside the package). In Next.js dev with React StrictMode enabled, client trees are double-rendered on hydration — so the CSR counter increments TWICE per `<DndContext>` mount while the SSR counter only increments ONCE. Every grip button's `aria-describedby` references an SSR-generated ID that doesn't exist in the CSR DOM.

The exact `-1 → +2, -2 → +4, -3 → +6` ratio in the hydration diff is the dead giveaway — module-level counter + StrictMode double-pass.

## Fix

One-line addition in [`todo-tree.tsx`](../../src/registry/components/data/todo-tree/todo-tree.tsx):

```tsx
const dndContextId = useId();
// ...
const dndProps: DndContextProps = {
  id: dndContextId,
  sensors: dndInternal.sensors,
  // ...
};
```

React's `useId()` is explicitly designed for this — it produces the same ID on server and client by tying generation to the React tree position rather than a mutable counter. `<DndContext>` accepts an explicit `id` prop that, when provided, bypasses dnd-kit's internal counter entirely. dnd-kit then derives all child IDs (the `DndDescribedBy-N` description elements, draggable announcements, etc.) from this stable seed instead of the module counter.

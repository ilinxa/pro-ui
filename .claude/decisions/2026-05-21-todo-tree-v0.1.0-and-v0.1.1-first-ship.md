## F-cross-13 carrier expansion — Checkbox.onCheckedChange

Path-b smoke against the published v0.1.0 Vercel artifact surfaced 1 F-cross-13 site:

| Site | Symptom | Fix |
|---|---|---|
| [`parts/todo-tree-checkbox.tsx:37`](../../src/registry/components/data/todo-tree/parts/todo-tree-checkbox.tsx) | `onCheckedChange={(next) => ...}` with `next === "indeterminate"` check. Radix Checkbox passes `boolean \| "indeterminate"`; Base UI Checkbox passes only `boolean`. The `===` comparison fails TS type-narrowing against the narrower signature: `'boolean' and 'string' have no overlap`. | Replaced with `typeof next === "boolean" ? next : true`. Gates through both backends without an `unknown` cast. Semantics unchanged — indeterminate (Radix-only) commits as `true` because that's the only state the row transitions to on partial check commit. |

**Adds Checkbox.onCheckedChange to the F-cross-13 carrier list** alongside the prior `Select.onValueChange` + `TooltipProvider.delayDuration` carriers. The defensive coercion pattern is now established at three primitive boundaries — Select callbacks (widen to `(v: string | null) => void`), Tooltip props (drop divergent prop names like `delayDuration`), and **Checkbox callbacks (typeof-guard instead of equality check for narrower callback args)**.

Path (b) producer-primitive refresh — re-running `pnpm dlx shadcn@4.6.0 add tooltip select checkbox ...` to align producer primitives with Base UI — remains a standalone hygiene task.

## Lessons reinforced

1. **Same-day F-cross-13 patch loop is the new default.** Three consecutive ships (todo-rich-card v0.1.0→v0.1.1; sibling; todo-tree v0.1.0→v0.1.1). Any future procomp using Select/Checkbox/Tooltip primitives should plan for it.
2. **F-S1 cross-procomp relative-paths lock is battle-tested.** Pre-application from C1 of any future cross-procomp procomp is the expected discipline.
3. **Pre-emptive F-cross-13 defensive coercion catches Select but missed Checkbox** — the C7 plan locked Select callbacks at `(v: string | null) => void` but did not preemptively rewrite the Checkbox `=== "indeterminate"` check. **Next plan should pre-emptively use `typeof next === "boolean" ? next : true` at every `onCheckedChange` boundary** as the project default.
4. **Permissions surfaces need end-to-end audit before declaring them on public types.** Declaring `permissions` on `TodoTreeProps` without threading them through keyboard handlers shipped a Medium gap to consumers; spotcheck caught it but a pre-commit "wire-all-the-things" audit on the C8 keyboard handler would have closed it before push.

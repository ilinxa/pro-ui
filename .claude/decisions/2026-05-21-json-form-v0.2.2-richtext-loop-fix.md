# json-form v0.2.2 — richtext controlled-mode echo loop fix (React #185)

## Summary

User-reported React error #185 ("Maximum update depth exceeded") on the json-form docs page after the user navigated to the Rich Fields tab and interacted with any input. Single-file fix in `parts/field-richtext.tsx`: stabilize the `value` and `onChange` props passed to `<ArticleBodyEditor>` so its internal reference-equality sync effect doesn't ping-pong with RHF's per-render fresh references.

## Symptom

```
Uncaught (in promise) Error: Minified React error #185
  at … (React internals: max-update-depth check)
  at … (Plate / RHF subject cascade: callback → next → next → em → onChange)
```

Reproduce: load `/components/json-form`, switch to Rich Fields tab, click any input.

## Root cause

Classic controlled-mode echo loop, same shape as flow-canvas-01 v0.2.3 (round-trip echo guard). The loop traces through three components:

1. **`ArticleBodyEditor`** (article-body-01 procomp) has a controlled-value sync effect that compares `controlledValue !== lastSyncedValueRef.current` to decide whether to call `editor.tf.setValue(controlledValue)`. **Reference equality, not content equality.**
2. **RHF Controller** in FieldWrapper emits a NEW `controller.value` reference on every state change, even when the underlying content is the same.
3. **`FieldRichtext`** (this file) was passing that fresh-per-render reference straight through to `<ArticleBodyEditor value={safe} ...>`, plus a fresh `(next) => onChange(next)` arrow on every render.

Every render that RHF triggered → ArticleBodyEditor's effect ran → `setValue(controlledValue)` → Plate emitted onChange → handleChange → outer onChange → RHF setValue → next render with fresh ref → loop. React capped it at 50 updates and threw #185.

The bug was latent in v0.1.1+ (when richtext landed), but the v0.2.0 substrate work changed the per-field re-render shape just enough that the loop tightened past React's update-depth cap. v0.1.x's pre-v0.2.0 codepath happened to defuse it by accident.

## Fix

`parts/field-richtext.tsx` adds two defenses at the consumer (json-form) layer:

1. **Content-stable `value` ref** — new internal hook `useStableRichtextValue(value)` returns the same ArticleBodyValue reference across renders as long as `JSON.stringify(value)` hashes the same. Identity flips only when content actually changes (Plate emits a semantic change).
2. **Stable `onChange` ref** — `useCallback((next) => onChange(next), [onChange])` instead of an inline arrow. ArticleBodyEditor's `handleChange = useCallback(..., [onChange])` then keeps a stable identity, so Plate's onChange prop doesn't churn.

These are defenses 1 and 2 of the project's three-defenses controlled-mode pattern; the third (suppressing notify during continuous Plate flows) is best fixed in article-body-01 itself.

## Where the real fix should live

`article-body-01`'s sync effect should use a content hash (or a deep-equal) instead of reference equality. That's a separate procomp ship cycle and a more invasive change. Until then, this consumer-side guard at the FieldRichtext layer contains the issue and is strictly additive — zero public-API change, zero behavior change for non-richtext fields.

Tracking item for future article-body-01 v0.3 ship: replace `controlledValue !== lastSyncedValueRef.current` ref check with a content-stable comparator in [article-body-01.tsx:52-57](../../src/registry/components/data/article-body-01/article-body-01.tsx#L52-L57).

## Icon forwardRef fix (`52e5f33`)

**Pre-existing latent bug surfaced by the C13 prod build.** HANDOFF flagged "pnpm build not run since C7. Re-run before C13 close." — that warning paid off.

`parts/icon.tsx` used `typeof icon === "function"` to detect component-type icons. lucide-react v0.475+ ships icons as `React.forwardRef` objects (`typeof === "object"` with `$$typeof` + `render` + `displayName`). The component object fell through to the ReactNode branch and crashed static prerender with:

```
Error: Objects are not valid as a React child (found: object with keys {$$typeof, render, displayName}).
```

Fix: add `isComponentObject` guard alongside the function check:

```ts
const isComponentObject =
  typeof icon === "object" && icon !== null && "$$typeof" in icon;
if (typeof icon === "function" || isComponentObject) { … }
```

Latent since C8 (when NavPrimaryAction / NavUser landed) — dev mode happened to render forwardRef icons via React 19's runtime fallback. Static export's pre-render path is stricter.

**Project-wide caveat:** this Icon helper is local to sidebar-nav-01. Other procomps with similar lucide-icon-as-component patterns should be audited. Out of scope for this ship; tracked as informal follow-up.

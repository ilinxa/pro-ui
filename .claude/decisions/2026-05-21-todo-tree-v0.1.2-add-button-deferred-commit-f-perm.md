# todo-tree v0.1.2 — `+ New` button + deferred-commit flow + F-perm closure

## Summary

Same-day follow-on after the v0.1.1 GATE 3 spotcheck. Closes the two v0.1.1 follow-ups that were targeted at v0.1.2 (F-perm) and v0.1.x cosmetic (F-grip-clip), and ships a UX/API batch surfaced mid-session by the user's iteration:

1. **F-perm closure** — keyboard `Space` (toggleActive) and `Delete`/`Backspace` (remove) now consult the declarative `permissions` matrix + `item.locked` + `readOnly`, fire `onPermissionDenied({ action, itemId, reason })` with one of `denied-by-rule` / `denied-by-lock` / `denied-by-readOnly`. The single-action `evalPermission` helper in `lib/permissions.ts` already existed; the patch wires it into `useTreeKeyboard` via a new `permissions` + `onPermissionDenied` + `readOnly` arg threaded through `TodoTreeList`.
2. **F-grip-clip closure** — grip moved from `absolute -left-4` (which clipped outside the row's bounding box inside any `overflow:hidden` / `overflow:auto` ancestor) into the row's flex flow as a leading 16px column. Drag is now reachable inside the demo containers.
3. **`editable` prop** (default `false`) — single gate for the toolbar `+ New` button + bulk action bar's Delete button. Reasoning: without an editing affordance, a freshly-added "Untitled" row is unrenameable and Delete points at rows the user can't interact with. `<TodoTreeWithEditor>` passes `editable={true}` automatically.
4. **`+ New` toolbar button** + **`createItem?: () => TodoItem` prop** — opt-in factory. Built-in fallback emits `{ id: crypto.randomUUID(), name: "Untitled", status: statusOptions[0].value, active: true, setAt: new Date().toISOString() }` when `statusOptions` is non-empty.
5. **`onCreateRequest?: (item: TodoItem) => void` prop** — deferred-commit hook. When provided, the toolbar fires it instead of mutating the tree. The consumer commits via `handle.addItem` when ready. Used by `<TodoTreeWithEditor>` to open the edit dialog on a pending item and commit on Submit only.
6. **`<TodoTreeWithEditor>` pending-item flow** — dialog opens immediately on `+ New` (pending mode); user fills the form; Submit calls `handle.addItem` + `focusItem`, Cancel/Escape/overlay discards. In edit-existing mode the old live-edit semantics are preserved (no Submit footer; each onChange propagates).
7. **Dialog UX hardening** — `max-h-[calc(100dvh-2rem)]` + inner `overflow-y-auto` so tall cards scroll inside the dialog; `max-w-252` (1.5× the previous `max-w-2xl`); `aria-describedby={undefined}` opt-out to silence Radix's "Missing Description" warning; blur `document.activeElement` before flipping `open=true` to prevent the Radix aria-hidden + focused-descendant a11y warning.

## Public-API impact

Three new public props on `TodoTreeProps`:
- `editable?: boolean` (default `false`) — gates `+ New` + bulk Delete
- `createItem?: () => TodoItem` — explicit factory; falls back to built-in if `statusOptions[0]` is available
- `onCreateRequest?: (item: TodoItem) => void` — deferred-commit hook

One new internal prop on `TodoTreeListProps` for permissions threading: `permissions` + `onPermissionDenied` + `readOnly`.

No breaking changes. All new props are optional with safe defaults. Existing consumers see no behavioural change unless they opt in (set `editable={true}`) — including the 6 bare-tree demo sections, which lose `+ New` + bulk Delete because they don't pass `editable`. The "With editor" demo (section 2) gains them automatically via the wrapper.

## Decision points logged

- **Imperative handle does NOT gate on permissions.** `handle.removeItem` / `handle.toggleActive` / `handle.removeItems` / `handle.toggleActiveBulk` mutate regardless of `permissions` / `item.locked` / `readOnly`. The handle is the raw mutation surface for advanced consumers (controlled-mode orchestrators, server-side reconcilers); UI affordances (keyboard, toolbar, DnD) ARE the gates. Documented in guide § Permissions. JSDoc note still owed (see F-handle-doc follow-up below).
- **Top-level `+ New` is NOT gated by `permissions.default.addChildren`.** The matrix is per-node; there's no implicit root node. Gate via `editable` + `readOnly` instead. F-create-perm follow-up for the guide doc closure.

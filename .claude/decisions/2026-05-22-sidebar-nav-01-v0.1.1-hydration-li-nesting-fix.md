# sidebar-nav-01 v0.1.1 — hydration error fix (user-reported, same-day patch)

## What broke

User opened `/components/sidebar-nav-01` (the sectioned demo) immediately after the v0.1.0 ship and got a console error:

```
In HTML, <li> cannot be a descendant of <li>.
This will cause a hydration error.
```

Stack trace pointed at `SidebarNavRow` inside `SidebarNavList[entries.map() > entry.items.map()]`. The serialized snippet showed:

```
<SidebarNavSection>
  <li class="list-none">
    <div>
    <div id="…" role="group" aria-label="Workspace">
      <span>
        <SidebarNavRow>
          <li class="list-none">
```

## Root cause

Two **independent structural bugs** that compound only when SidebarNavSection's *sidebar-collapsed* branch is rendered (i.e., the icon-only sidebar mode with an expanded section group):

### Bug 1 — `<span key>` wrappers in `parts/sidebar-nav-list.tsx`

`SidebarNavList` returns `<ul role="list">`. Inside its `entries.map()` it wrapped every returned row in a `<span key={…}>`:

```tsx
// top-level NavItem
return <span key={entry.id}>{renderRow(entry, null, i)}</span>;

// section body items
{entry.items.map((child, idx) => (
  <span key={child.id}>{renderRow(child, entry.id, idx)}</span>
))}

// section render-prop path
return (
  <span key={entry.id}>
    {renderSection({ … })}
  </span>
);
```

Two consequences:

1. `<ul><span><li>` — `<span>` is not a valid direct child of `<ul>` (only `<li>` is). React's hydration validator flagged this independently.
2. Section bodies stacked another wrapper layer: `<li>` (section root) → body → `<span>` → `<li>` (row). Combined with bug 2 below, the inner `<li>` ended up as a descendant of the outer `<li>` via a non-list container, tripping the documented "cannot be a descendant" check.

### Bug 2 — `<div role="group">` in `parts/sidebar-nav-section.tsx` (collapsed-sidebar branch)

The sidebar-collapsed branch (icon-only mode) of `SidebarNavSection` wrapped its non-collapsed section body in a `<div>`:

```tsx
{!isCollapsed && (
  <div id={bodyId} role="group" aria-label={section.title}>
    {children}
  </div>
)}
```

The children are `<li>` rows (produced by `SidebarNavRow`). `<li>` is invalid as a direct child of `<div>`. Combined with bug 1 (the `<span>` wrappers), the final structure was:

```html
<li>                                ← section root
  <div>…icon header…</div>
  <div role="group">                ← INVALID list container
    <span>                          ← INVALID <ul>-direct-child wrapper
      <li>…row…</li>                ← nested <li> warning fires here
    </span>
  </div>
</li>
```

The expanded-sidebar branch already used `<ul role="group">` correctly. Only the collapsed-sidebar branch was wrong — easy to miss because the docs-site demo defaults to expanded mode.

## Fix

### `parts/sidebar-nav-list.tsx`

1. Dropped every `<span key>` wrapper.
2. Moved the key onto the returned `<SidebarNavRow>` directly inside `renderRow`.
3. Top-level NavItem map call returns `renderRow(entry, null, i)` directly.
4. `renderSection` slot-path return uses `<Fragment key={entry.id}>{…}</Fragment>` so the key lives on a node that doesn't render to DOM — the consumer's `<li>` becomes a direct child of `<ul>`.

```tsx
const renderRow = (item, sectionId, indexInSection) => {
  const defaultRender = (
    <SidebarNavRow key={item.id} item={item} … />
  );
  if (!renderItem) return defaultRender;
  return (
    <li key={item.id} className="list-none">
      {renderItem({ …, defaultRender })}
    </li>
  );
};
```

### `parts/sidebar-nav-section.tsx`

Collapsed-sidebar branch body switched from `<div>` to `<ul>`:

```tsx
{!isCollapsed && (
  <ul
    id={bodyId}
    role="group"
    aria-label={section.title}
    className="flex flex-col gap-1"
  >
    {children}
  </ul>
)}
```

The resulting structure is `<li><ul><li>…</li></ul></li>` — valid HTML for a nested list group; the inner `<li>` is a descendant of the outer `<li>` only through a `<ul>`, which the HTML spec and React's hydration check accept.

## Generalizable lesson

When threading content into a `<ul>`, **never** wrap entries in a non-list element to carry a key. Use one of:

- The keyed element directly (`<li key={…}>` or `<Component key={…}>` where `Component` renders an `<li>`).
- `<Fragment key={…}>` for consumer-rendered output where the parent expects a specific child shape.

The same rule applies to `<dl>` / `<dd>` + `<dt>`, `<select>` + `<option>`, `<table>` + `<tr>`, etc. — any element with a strict required-child-type contract.

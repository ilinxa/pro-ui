## Why this matters

User feedback explicitly identified json-form as substrate-grade: "this component is one of the most essential components of our library — it must be production-grade A+ effectively developed component." The B+ → A+ gap was real:

1. **Per-keystroke perf** — every `FieldWrapper` subscribed to the full values bag via `useWatch({ control })` to populate `allValues` for renderers. 20-field form = 20 re-renders per keystroke regardless of which field changed. The 50-conditional dev-warn was treating the symptom, not the cause.
2. **No headless layer** — consumers building custom layouts had to drop down to RHF primitives. No `useJsonFormFieldValue` / `useJsonFormFieldsValue` for narrow-deps reads.
3. **No devtools** — debugging schemas / values / conditionals / errors meant `console.log`-spelunking.
4. **No typed-renderer authoring** — consumer-registered renderers got `value: unknown` + `field.config: FieldConfig | undefined` with no narrowing.
5. **Compile pipeline coupled schema + strings** — locale switches re-walked the entire field list.

v0.1.6 fixed (1) partially via narrow-deps in the two hooks `useConditional` + `useComputed`. v0.1.7 added (2)+(3)+(4)+(5) as strictly-additive public surface.

The cap-lifter (full default-registry watch drop) is intentionally deferred to **v0.2.0** under the locked staging plan: v0.1.7 rides ≥7 days post-deploy OR until one external consumer installs cleanly, then v0.2.0 ships the behavioral change. This hedges against unanticipated downstream impact while keeping the v0.1.7 ship strictly additive.

## Key technical decisions

### Compile split (A3, v0.1.7 C1)

`compileSchema(schema, strings)` is now internally split into:
- `compileStructural(schema): CompiledStructural` — schema-keyed, cheap (field-list extraction + schema reference)
- `injectStrings(structural, strings): ZodObject` — strings-keyed, does the heavy lifting (Zod chain construction + outer refinements)
- `compileSchema(schema, strings)` becomes the composition wrapper for any internal callsite that wants the one-shot

`useJsonForm` uses a two-`useMemo` pattern: strings-only changes (locale switch, message overrides) skip the schema re-walk. Real perf savings are microseconds for typical schemas — but the cleanly-separated stages make the cache reasoning obvious. All three helpers are **internal-only** (not exported from `index.ts`).

### `dependsOn` typed metadata (A1, v0.1.7 C2)

`FieldDefinition.dependsOn?: ReadonlyArray<string>` declares which field names a custom renderer's `allValues` access depends on. **v0.1.7 ships typed-only:** the runtime watch-gating that honors this flag lands in v0.2.0. v0.1.7 fires a `validateSchemaDev` warn on dangling references (paths not matching any `schema.fields[].name`) so consumers can author forward-compatible schemas today.

Three semantic states per Q-P1 lock (in the v0.2.0 plan):
- `undefined` → full-bag subscription (current v0.1.x behavior)
- `[]` → no subscription; renderer receives a `rhf.getValues()` snapshot under `allValues`
- `['a', 'b']` → narrow subscription; `allValues` rebuilt from watched names via `setByPath`

The companion `BUILTIN_RENDERER_TYPES_SKIPPING_ALL_VALUES` set (25 types — all default-registry slots, including `computed` whose values-reading happens inside `useComputed`'s own watch independently of FieldWrapper's) is defined-but-unused in v0.1.7 to keep v0.2.0's diff small.

### `<JsonFormDevtools>` — lazy-loaded body chunk (C3, v0.1.7 C5)

Split into two files for `React.lazy()` to actually deliver chunk-level isolation:
- `parts/json-form-devtools.tsx` — loader stub (~60 LOC). Holds the `JsonFormDevtools` component, prod no-op gate (`process.env.NODE_ENV` + `force` override), `Suspense fallback={null}` wrapping, and the lazy-import boundary. Ships in the main bundle.
- `parts/json-form-devtools-body.tsx` — panel body (~250 LOC). Four tabs (Schema / Values / Conditionals / Errors), `Ctrl+Shift+J` shortcut (overridable), inline-vs-floating posture. Lazy chunk only fetches when the component renders to non-null.

**Important clarification:** the prod-NODE_ENV check is a **runtime no-op**, NOT a bundler tree-shake. The loader stub itself still ships unless consumer-side import is conditional. Usage docs spell out the `{process.env.NODE_ENV !== "production" && <JsonFormDevtools />}` pattern for true dead-code elimination.

### F-S1 cross-procomp `/types` rewrite (v0.1.5)

Symptoms surfaced during smoke harness path-b run: shadcn 4.6.0's path rewriter mangles `@/registry/components/data/article-body-01/types` → `@/components/json-form/types` (CURRENT slug, not the dep's slug) for cross-procomp imports. Component-file imports (`/article-body-01.tsx`) are preserved correctly.

Fix: re-export the needed symbols (`ARTICLE_BODY_EMPTY_VALUE` value + `ArticleBodyValue` type) from `article-body-01.tsx` itself; json-form's `field-richtext.tsx` imports all three from the component-file path the rewriter handles correctly. Cost: `article-body-01` patch-bump (additive re-export, no behavior change).

This is the THIRD confirmation of F-S1 across procomps (json-form v0.1.4 → v0.1.5, rcif v0.1.0, todo-tree v0.1.0). Pattern is now locked at the project level — pre-apply to any cross-procomp imports in future procomps.

### Controlled-mode three-defenses pattern (v0.1.5)

json-form's `ChangeBridge` (lines 174-225 of `json-form.tsx`) added defense #2 (structural-resync guard on round-trip echoes) via key-sorted `stableStringify`. Defense #1 (microtask-deferred consumer notify) is the existing 100ms debounce. Defense #3 (mid-flow suppression) doesn't apply — forms are discrete-field updates, not continuous flows.

# calendar-01 v0.2.0 — the editable calendar (Stages 1–4 + ship)

The opt-in **editing layer** over v0.1.0's read-only date grid. `editable` defaults off → the
read-only surface (with two intentional, opt-out-able exceptions, below). Built across four
stages (1–3 at the checkpoint `79573c2`; Stage 4 + ship this session), reviewed by **3 AI-assisted
adversarial passes**, and shipped to `master`.

## What shipped

- **Pointer editing:** month @dnd-kit drag-move + native day-resize; week/day native-pointer
  move/resize/draw + the all-day spanning band; a live resize preview (commit on release).
- **Keyboard editing:** focus an event → `←/→` move (day in month/agenda, `snap` in the grid),
  `Shift+←/→` (+`↑/↓` in the grid) resize, `Delete`, `Enter` open editor, `F2` rename; focus a
  day cell → `Enter` opens the quick-composer. Delegated off `data-occ-id`/`data-day-ms` so it
  never collides with the v1 view/period keys; focus restored when a transient closes.
- **Cross-surface copy / cut / paste** over a shared `ilinxa/task` clipboard envelope
  (`lib/clipboard.ts`, `text/plain`). Document-level handlers gated on the calendar owning focus.
  Tasks carry to gantt/kanban/tree via the OS clipboard; **paste-target decides all-day vs timed**
  (subsuming the all-day⇄timed conversion). Re-id on paste; one `onChange` per paste.
- **Edit surfaces completed** (Stage 1–3 declared the transient state but never rendered it):
  a modal **detail-editor overlay** + **inline-rename** popup (so Edit/Enter/F2 work without the
  inspector), a shared `EventEditorPanel` reused by the inspector, context-menu Copy/Cut/Rename.
- **F-04 height-responsive month overflow** (rAF-gated ResizeObserver; `maxEventsPerCell` overrides).

## Key decision — copy/paste IS the cross-surface mechanism (deviation from GATE-2 plan §9)

A deep review before Stage 4 found the assembly renders **one view at a time**, so the plan's
"cross-view drag (agenda → month)" is physically impossible on one screen. With the user, the
cross-surface mechanism was redefined to **copy/paste over the shared `TodoItem` envelope** — it
spans every task-family surface via the OS clipboard (drag can't), and converts all-day⇄timed by
the paste target. **Deferred/reserved:** external HTML5 drop tray (`onExternalDrop` stays declared
+ inert, JSDoc marked reserved), agenda-row dragging, bespoke conversion-drag. Full rationale in
[plan §14c](../../docs/procomps/calendar-01-procomp/calendar-01-procomp-plan-v0.2.0.md).

## Two intentional non-byte-identical-to-v1 read-only changes (both opt-out-able)

1. **Status-driven event color** (`colorBy` default `"status"`; `colorBy="urgency"` restores v0.1's
   deadline ramp). (Carried from the §14b checkpoint change.)
2. **Height-responsive month overflow** (`maxEventsPerCell` restores a fixed cap; unbounded ≈ 3).

## Lessons

- A single-view assembly invalidates "cross-view drag" — for cross-surface task transfer, a shared
  **clipboard envelope over the canonical type** beats drag (works across surfaces + apps; converts
  by target). Worth generalizing to the whole task family.
- "Declared transient state" ≠ "rendered surface": Stage 1–3 set `editingId`/`renamingId` but only
  the optional inspector rendered the editor and rename had no surface at all — the headline "Edit"
  was dead by default. Completing the surfaces was a real (review-caught) requirement, not polish.
- The 3-pass adversarial review (correctness/consistency/design-a11y) caught a real default-path bug
  that green gates + a single self-review missed.

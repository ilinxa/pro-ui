# json-form v0.2.5 — radio-group always-controlled

## Summary

User-reported React warning:

```
radio-group.tsx:13  RadioGroup is changing from uncontrolled to controlled.
Components should not switch from controlled to uncontrolled (or vice versa).
```

Root cause in `parts/field-radio-group.tsx:24`:

```tsx
<RadioGroup value={value == null ? undefined : String(value)} ... />
```

For a radio-group field with no `defaultValue` (e.g. `kind` on the Conditional demo tab, or any consumer schema with `type: "radio-group"` + `validators: { required: true }`), the initial RHF value is `undefined`. The renderer passed `undefined` to Radix's `RadioGroupPrimitive.Root` → Radix saw "no `value` prop" → treated the group as uncontrolled. On first click, RHF stored the picked value; next render, the renderer passed a defined string → Radix flipped to controlled → React logged the warning. The shadcn `radio-group.tsx` line 13 is just the `{...props}` spread on the primitive — not the bug site.

## Fix

One-character class change (mechanically aligned with `FieldSelect`'s already-correct pattern at `parts/field-select.tsx:49`):

```diff
- value={value == null ? undefined : String(value)}
+ value={value == null ? "" : String(value)}
```

Empty string is the Radix-canonical "no item selected" marker (no `RadioGroupItem` has `value=""`, so the group renders with no selected radio). The renderer is now always-controlled. No public-API change; no schema migration.

## A1 — the consequential fix

**What the report flagged:** `changePriority` ([use-calendar-edit.ts](../../src/registry/components/data/calendar-01/hooks/use-calendar-edit.ts)) fires `onChange` only;
`changeStatus` fires both `onStatusChanged` + `onFieldEdited(status)`;
`applyEditedSubtree` fires only `onStatusChanged`. Asymmetric — a granular-event
consumer can't pick one channel.

**What I found going deeper:** the asymmetry is wider. `EventEditorPanel` wires the
embedded `<TodoRichCard editable>` with `onChange` only ([calendar-edit-overlays.tsx:82](../../src/registry/components/data/calendar-01/parts/calendar-edit-overlays.tsx)),
so the card's own granular events (it *does* emit `fieldEdited` per field +
`statusChanged`, [todo-rich-card edit-popup.tsx](../../src/registry/components/data/todo-rich-card/parts/edit-popup.tsx)) are dropped. So **every** modal/inspector edit
except status collapsed to `onChange`-only.

**Fix:** make `applyEditedSubtree` the chokepoint — it now diffs `prev` ↔ `nextItem`
across the eight `TodoEditableField`s and emits `onFieldEdited` per changed field
(keeping the existing `onStatusChanged`). The embedded card's granular events stay
unwired, so **no double-fire**. This keeps everything flowing through the single
dispatcher rather than bypassing it.

**`priority` stays `onChange`-only — deliberately.** `priority` is not a member of
`TodoEditableField` ([todo-rich-card/types.ts:134](../../src/registry/components/data/todo-rich-card/types.ts)), so `TodoFieldEditedEvent.key`
cannot carry it. Emitting `onFieldEdited({key:"priority"})` would require widening
the **shared** `TodoEditableField` union — a cross-procomp change touching card /
kanban / gantt / tree. The report itself offered "document it" as an acceptable
alternative; I took that (documented at the `onFieldEdited` prop JSDoc, the guide's
new "Granular `onFieldEdited` coverage" paragraph, and inline comments in
`changePriority` + `applyEditedSubtree`). The type extension is recorded as a
deferred option, not done here.

Also additive: `TodoEditableField` is now re-exported from the calendar barrel, so
a consumer wiring `onFieldEdited` gets the key type without a second import.

## C2 — why NOT to apply the report's fix

The report wants the `coveredDays` exclusive-midnight decrement gated on `allDay`,
arguing a timed deadline at 00:00 "genuinely covers that day." The code is exactly
as described, but the conclusion is wrong:

- A timed event can only end at exactly midnight if its span < 24h (else `classify`
  makes it all-day), so it **starts on the immediately-preceding day** and occupies
  `[start, midnight)` — **zero extent** on the next day.
- The time-grid already excludes that day: `DayColumn` filters `o.endMs > dayStartMs`
  ([calendar-time-grid.tsx:412](../../src/registry/components/data/calendar-01/parts/calendar-time-grid.tsx)), so the next day's column (`dayStartMs === endMs`) drops it.
- Applying the `allDay`-gate would make **Month** show the event on a day the
  **week/day** grid does not → a new Month↔time-grid inconsistency.

The half-open `[start, end)` model is uniform across the whole component; the
report's premise that "exclusive-midnight is only for all-day" is itself the
questionable assumption. Left as-is.

## Lessons

- **A consumer report's per-finding fix can be locally accurate yet globally wrong.**
  C2's code description was correct; its fix would regress a different view's
  consistency. Validate the *fix*, not just the *observation*. (Echoes the
  2026-06-18 R10 "report wrong vs the other backend" lesson, different axis:
  here it's wrong vs another *view*, not another *backend*.)
- **"Not a `TodoEditableField`" is a real API boundary, not an oversight.** The clean
  within-component fix for priority is documentation; the "emit the event" fix is a
  shared-type change with cross-procomp blast radius. Name the boundary, don't paper
  over it.
- **A dead duplicate branch can hide a live bug.** P1's "identical milestone branch"
  was unreachable *because* milestones carry `allDay=true` and hit the all-day branch
  first — so milestones were silently mislabeled "All day," losing their time. The
  dead code was the symptom; the ordering was the bug.

# F-03 closed — default-registry whitelist drift lint

## Summary

Closes the last ⚠️ High follow-up from the json-form v0.2.0 GATE 3 full-checklist review. New build-step lint at [`scripts/validate-default-registry-whitelist.mjs`](../../scripts/validate-default-registry-whitelist.mjs) audits every type in `BUILTIN_RENDERER_TYPES_SKIPPING_ALL_VALUES` (currently 25 entries) and verifies its renderer source does NOT read `allValues`. Wired into `pnpm registry:build` (and therefore `pnpm vercel-build`), so a future built-in renderer that regresses by accessing `args.allValues` without being pulled out of the whitelist fails producer-side CI before consumers ever see a stale-snapshot bug.

No public-API change, no consumer migration. Internal tooling + a one-paragraph JSDoc refresh on `BUILTIN_RENDERER_TYPES_SKIPPING_ALL_VALUES` to point at the new lint.

## Why it had to exist

The v0.2.0 watch drop in `field-wrapper.tsx` skips the FieldWrapper-level `useWatch({ control })` when the resolved renderer is a built-in default in the whitelist. `allValues` then comes from a `getValues()` snapshot taken at render time — fresh on the field's own value change (via Controller) but **not reactive to other fields**. That's correct ONLY while the built-in renderers don't read `allValues` reactively. If a future built-in (say, `field-select`) starts using `args.allValues` to compute its options and nobody removes `"select"` from the whitelist, the option list silently stops reacting to upstream field changes. The drift class is exactly the kind that "compiles + renders + has demo" can't catch — same shape as the F-cross-07 meta-dep drift that the original `validate:meta-deps` lint was built for.

## How it works

Single-purpose script, ~190 lines. Hardcodes `src/registry/components/forms/json-form/lib/default-registry.ts` as the source of truth.

1. **Parse the whitelist** — regex out the string entries of `BUILTIN_RENDERER_TYPES_SKIPPING_ALL_VALUES: ReadonlySet<string> = new Set([...])`.
2. **Parse the registry object** — `defaultJsonFormRegistry: Record<string, FieldRenderer> = { ... }` yields `type → identifier` pairs (e.g., `"text" → FieldText`, `"code" → FieldCode`).
3. **Resolve identifier → source path**, with three shapes covered:
   - Named imports: `import { FieldText } from "../parts/field-text"` → direct.
   - `lazy(() => import("../parts/field-code"))` const declarations.
   - The createElement-wrapper indirection for lazy renderers: `const FieldCode: FieldRenderer = (args) => createElement(LazyFieldCode, args)` → chases one hop into the lazy const's path.
4. **Resolve to a file on disk** — `.tsx` / `.ts` / `/index.tsx` / `/index.ts` candidates.
5. **Scan for `\ballValues\b`** in the resolved file. On a hit, surface `type / identifier / file:line / evidence line / remediation`.

Three structural drift classes also surfaced as `high`:
- `whitelist-type-not-in-registry` — type listed but not present in `defaultJsonFormRegistry`.
- `unresolvable-identifier` — registry references an identifier the parser can't trace.
- `source-file-missing` — relative path doesn't resolve to a file.

Exit codes match `validate-meta-deps`: 0 clean / 1 findings / 2 script error.

## Wiring

```diff
  "scripts": {
    "validate:meta-deps": "node scripts/validate-meta-deps.mjs",
+   "validate:default-registry-whitelist": "node scripts/validate-default-registry-whitelist.mjs",
    ...
-   "registry:build": "node scripts/validate-meta-deps.mjs && shadcn build"
+   "registry:build": "node scripts/validate-meta-deps.mjs && node scripts/validate-default-registry-whitelist.mjs && shadcn build"
  }
```

`pnpm vercel-build` → `pnpm registry:build && pnpm build:source-map && next build`, so the lint runs on every Vercel deploy.

# json-form v0.2.4 — bridge RHF FormProvider inside JsonFormProvider

## Summary

User-reported runtime crash: opening the Json Form demo page and clicking the **Custom registry** tab killed the entire page with `Uncaught TypeError: Cannot destructure property 'control' of useFormContext(...) as it is null`. The Next.js error boundary surfaced as **"This page couldn't load."**

Root cause was NOT in the Custom Registry tab itself. It was in the `HeadlessExample` block embedded inside the Imperative tab's `<details>` (`src/registry/components/forms/json-form/demo.tsx:364-387`), which got mounted by a `<Tabs>` re-render path during the click. `HeadlessExample` faithfully follows the **documented** headless pattern:

```tsx
const { form, zodSchema, handle } = useJsonForm(contactFormSchema);
return (
  <JsonFormProvider value={{ ...handle, rhf: form, ... }}>
    <HeadlessForm />   {/* renders <JsonFormField name="topic" /> */}
  </JsonFormProvider>
);
```

…which omits an explicit `<FormProvider {...form}>` wrap. Before v0.2.4 that omission was load-bearing: `<JsonFormField>` → `<FieldWrapper>` calls `useFormContext()` (RHF's, not ours) to read `control / register / unregister / trigger / getFieldState / getValues`, and v0.2.0's per-field subscription gate also calls `useWatch({ control })`. Without RHF's `<FormProvider>` above, `useFormContext()` returns `null`, the destructure throws, every FieldWrapper in the headless tree crashes, and the error boundary tears down the whole page.

This is the v0.2.0 watch-drop substrate work biting at the documented API. v0.1.x got away with the broken-as-documented headless example because the destructure was narrower (`control, register, unregister` only) and FieldWrapper hadn't started reading RHF state during the initial render path the same way.

## Fix

One small change at the substrate. `<JsonFormProvider>` now wraps children in `<FormProvider {...value.rhf}>` internally:

```diff
+ import { FormProvider } from "react-hook-form";
  ...
  export function JsonFormProvider<TValues>({ value, hasSubmitted, children }) {
    return (
+     <FormProvider {...value.rhf}>
        <JsonFormContext.Provider value={value}>
          <JsonFormSubmittedContext.Provider value={hasSubmitted}>
            {children}
          </JsonFormSubmittedContext.Provider>
        </JsonFormContext.Provider>
+     </FormProvider>
    );
  }
```

Because the `JsonFormContextValue` contract requires `value.rhf: UseFormReturn` (the "escape hatch" the headless API was built around), the bridge is always satisfiable from props.

The outer `<FormProvider {...form}>` in `<JsonForm>` itself is now technically redundant when rendering the full `<JsonForm>` tree — but kept in place since nesting is a no-op (both providers see the same `form` instance, inner wins, identical values). The added `<FormProvider>` overhead is one extra Context.Provider call per render — same RHF context-value spread shape that `<JsonForm>` was already doing, no measurable cost.

## What this fixes

- **The reported crash.** Custom Registry tab (and any other tab in the demo that triggered the Imperative-tab tree to mount as a sibling render) now loads without the page-killing useFormContext-null destructure.
- **The documented headless pattern.** Consumers can now follow [`docs/procomps/json-form-procomp/json-form-procomp-guide.md`](../../docs/procomps/json-form-procomp/json-form-procomp-guide.md)'s "Fully-headless variant (standalone parts)" example verbatim — `<JsonFormProvider value={{ ...handle, rhf: form, ... }}>` is now self-contained.
- **Future demos / consumer recipes.** The headless pattern is the documented public surface; the substrate now matches what consumers will write.

## What this does NOT change

- The full `<JsonForm>` API: still wraps RHF + Json contexts identically. No behavior change for consumers of the high-level component.
- `useJsonFormContext` throwing semantics: still throws on null. The fix is upstream of where consumers call hooks.
- Any consumer who was already correctly wrapping with `<FormProvider>` AND `<JsonFormProvider>` keeps working — the inner-provider override is identical-shape (same `form` instance) so consumers see no difference.
- No version bump on `useFormContextOptional` or any other API surface. No deprecations.

## Why this is a patch (not minor)

Strictly additive. Zero public-API change. Zero consumer migration required. The headless pattern that previously crashed now works as documented; consumers who were already wrapping with both providers see no change. Patch-bump exemption per the readiness-review rule — GATE 3 skipped, v0.2.0 verdict carries forward.

## How it was diagnosed

Spent ~30 minutes pattern-matching the user's minified stack against the v0.2.0 watch-drop code path before falling back to a Chrome-DevTools-Protocol headless repro (`e:/tmp/jf-repro.mjs`). The repro:

1. Started `pnpm dev`.
2. Drove Chrome via CDP pipe transport — navigate `/components/json-form`, dispatch synthetic mouse events on the "Custom registry" tab, capture every `Runtime.consoleAPICalled` + `Runtime.exceptionThrown`.
3. Confirmed the crash reproduces 100% in dev mode.
4. Instrumented `FieldWrapper` with `console.error("useFormContext returned null!", { fieldName, fieldType, schemaFieldsCount, hasCtx, hasRhf, hasRhfControl })` — got back `{ fieldName: "topic", fieldType: "select", schemaFieldsCount: 4, hasCtx: true, hasRhf: true, hasRhfControl: true }`.
5. The `hasCtx: true / hasRhf: true / hasRhfControl: true / useFormContext-null` combo immediately identified the bug: JsonFormProvider has full context (with valid rhf.control inside), but RHF's FormProvider isn't above. Grepped for that pattern → `demo.tsx:364 HeadlessExample` is the carrier; further mounts of HeadlessExample fire because Tabs sibling re-render walked the Imperative tab tree during the Custom Registry mount handler.
6. Authored the JsonFormProvider bridge, removed instrumentation, re-ran the repro: Custom Registry panel renders cleanly (`activeTabs: ["Custom registry"]`, no page errors, body shows form fields).

## Lessons / memory candidates

- **Documented public patterns must be exercised end-to-end by the demos, not just type-checked.** `HeadlessExample` compiled and rendered DOM but crashed on first FieldWrapper inside it. A demo-side smoke test (open the headless `<details>` and interact) would have surfaced this at v0.2.0 GATE 3. The "browser-test controlled-mode interactions" memory generalizes — extend it to "browser-test the documented headless pattern at every minor bump."
- **Substrate-grade behavioral additions can retroactively break documented patterns.** v0.2.0 made FieldWrapper's RHF context more load-bearing without anyone noticing the headless example had no RHF FormProvider. The lint-friendly addition to the readiness-review template: "Walk every documented standalone-parts pattern in the guide; mount each in a smoke harness."

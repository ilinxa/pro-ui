# rich-card v0.4.2 — empty card can now add fields / blocks

## Summary

`rich-card@v0.4.2` lifts the `+ FIELD` / `+ BLOCK` add-affordance row out of the `hasBody`-gated body wrapper in [`parts/card.tsx`](../../src/registry/components/data/rich-card/parts/card.tsx). Empty cards (zero fields, zero predefined entries) can now be populated through the standard UI; pre-v0.4.2 they only exposed `+ CHILD`, which meant any card created via `dispatchers.cardAdd` was a dead-end for content (only nestable, never fillable).

**Trigger:** surfaced 2026-05-17 during the rich-card-in-flow editor audit (post flow-canvas-01 v0.2.4 ship). User reported "add block options disappear after use" (turned out to be by-design — predefined keys are at-most-one-per-card) and "root card has no add options" (turned out to be a misperception — root's `+ CHILD` is rendered after all children at the bottom and was obscured by an open popover). Code investigation surfaced THIS distinct latent bug as the genuine issue.

**Patch-bump exemption** per the [readiness-review rule](../rules/component-readiness-review.md). Internal-only fix, no public API touch, GATE 3 skipped. rich-card's status remains `beta`; v0.4.0 promotion review carries forward.

### Release notes

> **Bug fix:** Cards with no flat fields and no predefined entries now expose the standard `+ FIELD` / `+ BLOCK` add buttons when `editable={true}`. Previously the entire body block — including these buttons — was hidden when `hasBody === false`, so any card created via `+ CHILD` (which begins with empty fields and predefined) was unfillable through the UI.
>
> **Observable behavior change:** if a consumer relied on the absence of these buttons as a signal that the card was empty (none in the wild we know of), the buttons will now appear immediately after `+ CHILD` adds a new card. Standard behavior; should not surprise anyone.
>
> **No public API change.** All v0.4.x types, props, dispatchers, and validators unchanged.

## Root cause

[`parts/card.tsx:325-428`](../../src/registry/components/data/rich-card/parts/card.tsx#L325-L428) (pre-v0.4.2) wrapped the rendered fields, rendered predefined entries, AND the add-affordances row inside a single `{hasBody && !collapsed ? (...) : null}` block. The intent was layout grouping — keep the body content under one wrapper for consistent spacing. The side effect was that the add-affordances row, which logically should render whenever the card is editable, was gated on `hasBody = hasFields || hasPredefined`.

The dispatchers.cardAdd path in [`rich-card.tsx:499-507`](../../src/registry/components/data/rich-card/rich-card.tsx#L499-L507) creates new cards with empty `fields: []` and `predefined: []`. Result: a freshly-added card always has `hasBody === false` and shows no add buttons.

## What changed in v0.4.2

Single edit in [`parts/card.tsx`](../../src/registry/components/data/rich-card/parts/card.tsx):

- Removed the inner `{config.editable ? (... add affordances ...) : null}` from inside the `hasBody && !collapsed` wrapper.
- Added a new sibling block immediately after the body wrapper (and before the children list), gated on `config.editable && !collapsed`. It renders the same `FieldAddForm` / `FieldAddButton` / `PredefinedAddMenu` triad with the same `levelStyle.fieldsClassName` for visual consistency.
- Added a load-bearing comment marking the change rationale.

The `hasBody && !collapsed` wrapper is retained around the rendered fields + predefined entries — those correctly should not render when empty.

## Lessons

**Layout grouping ≠ behavioral grouping.** Nesting "add this content" UI inside "render existing content" UI is convenient for layout but creates a discoverability dead-end when the existing-content branch is empty. Add-affordances should be gated on permissions, not on whether anything exists to add to.

**The misperceptions surfaced this fix.** The user-reported bugs ("options disappear" and "root has no actions") both turned out to be non-bugs (by-design + misread of the screenshot), but the investigation that ruled them out surfaced the actual problem. Worth doing the investigation even when the report is fuzzy.

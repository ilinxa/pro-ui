### Rotating dimension

- **Chosen:** Dim 2 — Public API
- **Why:** content-card-news-01 is the renderItem consumer for grid-layout-news-01 and the canonical news-family card — its API shape (item + variant + slot + override surfaces) defines what the rest of the news-domain family composes around. Verifying API-design discipline (openness, defaults, positional-vs-object callback signatures, public-helper exports) is exactly Dim 2's remit.
- **Verified?** yes (with 2 small findings)
- **Notes:**
  - **Openness vs sealedness ✓** — `actions: ReactNode` slot for nested interactives above the link overlay; `linkComponent: ElementType` polymorphic root; `categoryStyles: Record<string, string>` open theming map; 3 className overrides (`titleClassName`, `imageClassName`, `className`) for granular styling.
  - **Defaults are sane** — `variant` defaults to `"medium"` (the most common case); `LinkComponent` defaults to `"a"`; `loading` defaults to `"lazy"` with `"eager"` for `featured` (correct LCP optimization for above-the-fold hero); `categoryStyle` falls back to `"bg-muted"` for unknown categories.
  - **Discriminated union for variants** ✓ — `ContentCardNewsVariant = "featured" | "large" | "medium" | "small" | "list"`. Switch statement at line 103 dispatches to per-variant parts.
  - **No `any` in public types** ✓ — `ContentCardItem` is precise (id/title/image required + 6 optional fields); `ContentCardNewsLabels` is fully typed.
  - **Stable-identity rules** — `React.memo` wraps the root (line 118); per `meta.features` "stable item refs prevent re-renders in long feeds". ✓
  - **Naming consistency** ✓ — `format*` for callbacks, `*ClassName` for classnames, `linkComponent` for polymorphic root, `actions` slot. Matches the rest of the library's conventions.
  - **Public helpers exported** ✓ — `defaultRelativeTime` + `defaultDateFormat` are re-exported via `index.ts` for consumer extension. One sibling helper missing — see F-02.
  - **Soft-fail item shape** ✓ — only `id`, `title`, `image` are required; `excerpt`, `category`, `author`, `date`, `readTime`, `views` are all optional. Each variant gracefully omits when absent (per `meta.features`). Future-proof for sparse datasets.
  - One callback-shape concern + one helper-export consistency nit — see findings.

## Findings (2)

### F-01 — `onClick(item, event)` uses positional callback args; versioning trap if v0.2 wants `index` / `source` context

- **Severity:** 🔹 Low
- **Dimension:** 2 — Public API (future-proofness)
- **Location:** `src/registry/components/data/content-card-news-01/types.ts:62` + `content-card-news-01.tsx:75-80`
- **Observed:** `onClick?: (item: ContentCardItem, event: MouseEvent) => void` — positional callback. Per Dim 2 review-guide bad signal: *"onChange callback receives positional args (onChange(id, value)) instead of an object (onChange({ id, value })) — positional callbacks are a versioning trap."* If v0.2 wants to pass extra context — `index` (which slot in a tower), `source` (which list), `variant` (the visual variant the user clicked) — the signature must grow with new positional args, breaking existing consumers. Same shape as grid-layout-news-01 F-02 (session 8) — recurring pattern across the library.
- **Suggested fix:** Object-shape the callback in v0.2 (breaking): `onClick?: (args: { item: ContentCardItem; event: MouseEvent; index?: number; variant: ContentCardNewsVariant }) => void`. v0.1.x stays as-is; queue in migration notes when v0.2 lands. Consider documenting "positional callbacks are a versioning trap" in the project's component-guide so future components default to object-shape from v0.1.

### F-02 — `toDate` is imported internally but not re-exported via `index.ts`

- **Severity:** 🔹 Low
- **Dimension:** 2 — Public API (helper-export consistency)
- **Location:** `src/registry/components/data/content-card-news-01/index.ts` + `content-card-news-01.tsx:6` (imports `toDate` from `./lib/format-default`)
- **Observed:** `index.ts` re-exports `defaultRelativeTime` (from `./hooks/use-relative-time`) and `defaultDateFormat` (from `./lib/format-default`) — both are Dim 2 "public helpers exported" wins. But `toDate` (the sibling coercion helper, also from `./lib/format-default`) is NOT re-exported. A consumer building a custom date pipeline (e.g., wrapping `defaultRelativeTime` with timezone normalization) would need `toDate` to coerce `item.date` (string | Date) to a `Date` consistent with how the component itself does it. Inconsistent: two helpers exported, one omitted.
- **Suggested fix:** Add `export { toDate, defaultDateFormat } from "./lib/format-default";` to `index.ts` (currently exports only `defaultDateFormat`). One-line change. Closes the helper-export consistency loop.

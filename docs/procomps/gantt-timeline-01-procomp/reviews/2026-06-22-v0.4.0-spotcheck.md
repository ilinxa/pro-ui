## Why this is a minor (not a patch)

The two changes are bug-driven, but the fix mechanism adds public surface and flips a default:

- **New public API** — `GanttContextValue.drawMode: boolean` + `setDrawMode(on)`. `GanttContextValue` is exported, so this is additive-but-public (a hand-assembled consumer can drive its own Draw control off context). The toolbar gains a visible **Draw** toggle (editable only).
- **Behavior change** — empty-row drag previously **always** draw-created a task; now it **pans** by default and only draws when Draw is on. A consumer relying on the old "drag empty → create" loses it unless they toggle Draw. Semver says signal that with a minor; burying it in a patch would be dishonest.

The v0.2.1 / v0.3.1 patch precedents added **zero** public API — that's why they were clean patches. This isn't.

### Smoke checklist — reasoned (live run = F-01)

The whole pipeline is now: `onPointerDown` defers (sets `pendingRef` + classifies under the press) → first move past `DRAG_THRESHOLD` (4px) calls `resolvePending`, which picks exactly one gesture → below threshold a release is a click.

1. **Click (no move) → select** — `pendingRef` never crosses 4px; `endPointer` nulls it; the shape's `onClick` calls `activate`. ✔
2. **Vertical drag → scroll** — `resolvePending`: `|dy| > |dx|` ⇒ lock `"v"`; mouse path adjusts `scrollTop`. Works over bars *and* empty space (classification is irrelevant to vertical). ✔
3. **Horizontal drag on `void` → pan** — fall-through ⇒ `beginPan` + lock `"h"` + velocity tracking (momentum preserved). ✔
4. **Bar body drag → move** (movable) — `classifyTarget` → `barBody` → `resolvePending` starts a `move` `EditDrag`; `commitEdit` → `rescheduleItem`. ✔
5. **Bar edge drag → resize** — `edgeAt` (pixel proximity, needs `width ≥ 3·EDGE_PX` so a narrow bar stays all-move) + `can("resize")` ⇒ `barEdge`. ✔
6. **Summary bracket drag → group-move** — `[data-summaryid]` hit + `canGroupMove` + `summarySpanFor` ⇒ `summary` ⇒ `groupmove` drag ⇒ `moveSubtree`. ✔ (v0.3 path intact.)
7. **Empty row drag, Draw OFF → pan** — `classifyTarget` returns `void` because `ctx.drawMode` is false (the empty-row branch is gated on it) ⇒ pan. *This is the conflict fix.* ✔
8. **Empty row drag, Draw ON → create** — `drawMode` true + `[data-rowid]` non-summary ⇒ `emptyRow` ⇒ `create` drag ⇒ `createItem(parentId, …, index)`; a sub-`MIN_MS` draw is treated as a click (no stray task). ✔
9. **Draw toggle visibility** — toolbar renders the Draw `Button` only when `editable`; `aria-pressed={drawMode}`; `variant` flips default/ghost. ✔
10. **Cursor** — `cursor-crosshair` when a `create` drag is live **or** (`editable && drawMode`); `cursor-grab/grabbing` otherwise (suppressed in draw mode + during any edit). ✔
11. **Right-click → menu, not gesture** — `e.button === 2` early-returns in `onPointerDown`; the Radix `ContextMenu` trigger handles it. ✔
12. **Right-click items now work (the fix)** — the menu Portal renders to `document.body`, so its DOM is **not** a descendant of the scroll surface, but its events still bubble through the React tree to the body/gutter handlers. `onPointerDown` + `onKeyDown` (body) and `onKeyDown` (gutter) now guard with `!e.currentTarget.contains(e.target as Node) → return`, so a menu-item click/key no longer starts a canvas gesture (which had been swallowing the item's own `pointerup`) or double-triggers row nav/delete. ✔
13. **Pinch (2 pointers)** — second pointerdown abandons any `pendingRef` / in-flight `editDrag` and switches to focal-point pinch zoom. ✔
14. **Read-only (`editable={false}`)** — `classifyTarget` returns `void` unconditionally ⇒ only pan/scroll; byte-identical to v1. ✔
15. **`disableGestures`** — `onPointerDown` early-returns; cursor affordances gated on `!disableGestures`. ✔

## Dimension 5 (rotating) — Robustness

*Picked because the headline risk is regression in the rewritten gesture arbitration + the correctness of the portal-target guard.*

- ✅ **No-regression on the existing gestures.** The refactor preserves every prior arm (move / resize / group-move / pan / pinch / vertical-scroll / keyboard) — they're now reached *after* a deferred decision rather than at press-time. The 4px threshold makes a click unambiguous (no accidental 1px pan/edit), and the vertical-vs-horizontal split means a scroll intent is never hijacked into an edit. Net: more robust than the v0.3.x immediate-classify path.
- ✅ **Portal-target guard is the right shape.** `currentTarget.contains(target)` distinguishes a *DOM* descendant (real in-canvas press) from a *React-tree* descendant (portaled menu event in `document.body`) — exactly the asymmetry that caused the bug. The guard is on the three handlers that the menu's events reach (`onPointerDown` + `onKeyDown` on the body, `onKeyDown` on the gutter).
- ✅ **`onPointerMove`/`endPointer` deliberately un-guarded** — harmless: for a portaled menu event `pendingRef`/`editDragRef` were never set (guarded at `pointerdown`), so move does nothing and `endPointer` finds no edit to commit and no pending to clear. No extra guard needed; documented here so a future reader doesn't "fix" it.
- ✅ **Pointer capture + cleanup** — `setPointerCapture` on down; `endPointer` deletes the pointer, drops pinch ref when `< 2`, and resets locks at `size === 0`. `onPointerCancel` shares `endPointer`. No leaked lock/momentum across gestures.
- ✅ **Finite-date / NaN safety preserved** — the v0.3.1 `geoOk` guards, `edgeAt` `Number.isFinite` checks, and the create-draw `MIN_MS` click-vs-draw threshold all carry through unchanged.
- 🔹 **drawMode is Root-local with no prop** (see F-02) — acceptable for the assembly; the context escape hatch covers hand-assembly.
- **Verdict:** Pass.

## Findings (max 5)

### F-01 — Skeleton uses `role="region"` without an accessible name (🔸 Medium)

- **Location:** [`stat-card.tsx:104-122`](../../../src/registry/components/data/stat-card/stat-card.tsx) — `StatCardSkeleton` component
- **Description:** The skeleton wrapper has `role="region"` + `aria-busy="true"` + an sr-only `<span>{labels.loadingLabel}</span>`. ARIA spec requires `region` landmarks to have an accessible name (via `aria-label` or `aria-labelledby`); inner sr-only text content does NOT satisfy this. Result: SR users navigating by landmarks may hear "region: " with no name OR an inconsistent fallback depending on the SR.
- **Suggested fix:** Drop `role="region"` from the skeleton wrapper. `aria-busy="true"` + the sr-only span is sufficient — the skeleton is transient state, not a navigable landmark. Loaded card uses `<dl>` (which has its own implicit semantics).
- **Owner:** author. **Bump target:** v0.1.1.

### F-02 — Skeleton icon-block class derivation uses fragile string-replace (🔸 Medium)

- **Location:** [`stat-card.tsx:121`](../../../src/registry/components/data/stat-card/stat-card.tsx) — `<Skeleton className={spec.iconBoxSize.replace("bg-muted", "")} />`
- **Description:** The skeleton's icon placeholder strips `"bg-muted"` from `spec.iconBoxSize` to defer to Skeleton's own pulse-animation bg. For the `default` and `compact` variants this works (their `iconBoxSize` contains `"bg-muted"`). For the `detailed` variant, `iconBoxSize` is `"h-11 w-11 rounded-lg bg-primary/10"` — the replace is a no-op and the Skeleton inherits `bg-primary/10`, which conflicts with Skeleton's own animation styling. Visual artifact: detailed variant's icon skeleton looks different (and possibly worse) than default/compact during loading.
- **Suggested fix:** Hardcode the per-variant skeleton sizes in a local const map without piggybacking on `iconBoxSize`. Simpler, no string surgery, behavior consistent across variants.
- **Owner:** author. **Bump target:** v0.1.1.

### F-03 — Linked card with `value === undefined` produces an unnamed link (🔹 Low)

- **Location:** [`stat-card.tsx:230-234`](../../../src/registry/components/data/stat-card/stat-card.tsx) — `computedAriaLabel` resolution
- **Description:** When the consumer supplies `href` but no `ariaLabel` AND `value === undefined`, the fallback `\`${label}: ${value}\`` doesn't apply (value isn't string|number). Result: the overlay-link gets `aria-label={undefined}` — an unnamed link. SR experience: "link" with no name. Rare in practice (linked card with empty value), but the component should fall back to just `\`${label}\`` in that case.
- **Suggested fix:** Add the label-only fallback chain: `ariaLabel ?? (typeof label === "string" ? label : undefined)` BEFORE giving up on the typed-fallback combination.
- **Owner:** author. **Bump target:** v0.1.1.

### F-04 — `text-chart-2` for the "good" tone is a workaround, not a token (🔹 Low — design system)

- **Location:** [`stat-card.tsx:33-37`](../../../src/registry/components/data/stat-card/stat-card.tsx) — `TONE_CLASSES`
- **Description:** pro-ui has no dedicated `--success` token; the component falls back to `text-chart-2` (calibrated green-leaning hue from the chart palette) for the "good" delta tone. This is internally documented (comment + procomp guide note), but consumers reading the source see a chart token used for a non-chart purpose, which is mildly surprising. Real design-system fix is adding a `--success` (and `--success-foreground`) token to globals.css.
- **Suggested fix:** Defer until v0.2 OR a separate "design system tokens" pass. For v0.1, the workaround is documented and works. Possible v0.2 PR: add `--success` token, switch stat-card + any future siblings (gauge-card, etc.) to use it, deprecate `text-chart-2` for tone purposes.
- **Owner:** unassigned (cross-cutting design-system). **Bump target:** v0.2 OR separate plan.

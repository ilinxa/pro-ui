## 1. Executive summary

Entity-picker is the searchable typed picker Tier 1 component for the graph-system, built on shadcn's `Command` (cmdk) for search and `Popover` for the dropdown. The implementation lands the locked decisions cleanly: function overloads for mode-aware value typing (Q-P1), `<div role="button">` trigger to allow nested chip remove buttons (Q-P8 validate-pass refinement), id-set selection equality with `onChange` dedup (Q-P4), case-insensitive substring default match with try/catch around the host predicate (Q-P2), controlled-or-uncontrolled open state (Q-P9), 4-method imperative handle (Q-P10).

The 18-file structure matches plan В§7.1 exactly. The plan is the most carefully validated of the Tier 1 plans (6 validate-pass refinements; the trigger element type was caught and corrected pre-implementation). Implementation follows the plan with one notable discipline shift: the trigger ref pattern uses `useState`-as-callback-ref (line 89) instead of the documented `useRef`, with an explicit comment justifying the choice ("React Compiler-aware lint rule that flags ref-accessor functions passed to user-rendered functions during render"). Reasonable practical refinement.

The findings cluster around the same ecosystem patterns surfaced in properties-form: phantom `radix-ui` npm dep + over-declared shadcn primitives in `meta.ts`; `process.env.NODE_ENV` use (entity-picker plan В§12.5 #5 EXPLICITLY locks this вЂ” the carrier is intentional per the plan); missing `guide.md`; usage import-path drift.

**Headline finding:** Entity-picker plan В§12.5 #5 explicitly locks `process.env.NODE_ENV` as the deliberate dev-warn-gate mechanism. This component carries the F-cross-08 violation **by design** вЂ” strongest single piece of evidence that the cross-cutting finding's underlying rule (description В§6 #10 "no `process.env`") needs amending, not the component patching. Resolution: relax the rule sweep-wide at session 7.

**Recommended next step:** v0.1.1 patch removing the phantom `radix-ui` + over-declared `button`/`tabs` from `meta.ts` (F-03), authoring `guide.md` (F-02). F-01 (=F-cross-08) waits for the sweep-wide resolution. F-04, F-05 are docs touchups.

#### F-05 вЂ” Plan В§6.3 documents `useRef`-callback-ref pattern; implementation uses `useState` instead

- **Severity:** рџ”ё Medium
- **Dimension:** 1 (plan-vs-code drift).
- **Location:** Plan В§6.3 + В§3.3 documents `triggerRef: (node: HTMLElement | null) => void` as a callback ref attached to a single ref site. [`entity-picker.tsx:89`](../../../../src/registry/components/forms/entity-picker/entity-picker.tsx#L89) uses `useState<HTMLElement | null>(null)` and passes `setTriggerNode` (the setter) as the callback. The setter IS a stable callback ref by React's guarantees; functionally equivalent.
- **Observed:** The implementation comment (lines 86-89) explicitly justifies the deviation: "Track the trigger element via state (not a ref) so the setter is React's own setState вЂ” passes the React Compiler-aware lint rule that flags ref-accessor functions passed to user-rendered functions during render." Reasonable plan refinement caught at implementation time, but the deviation isn't documented in description, plan, guide, STATUS.md, or usage.tsx.
- **Why it matters:** Same "deviations are loud, not silent" theme that recurred in workspace + properties-form. The implementation choice is correct; the doc trail isn't.
- **Suggested fix:** Add a one-line note to plan В§6.3: "Implementation uses `useState`-as-callback-ref (the setter doubles as the ref callback) to satisfy React Compiler's lint rule against ref-accessor functions in render. Functionally equivalent." Trivial; v0.1.1 docs cleanup.

### Rotating dimension

- **Chosen:** Dim 7 — Accessibility
- **Why:** category-cloud-01 is a forms-category interactive control (single-select with toggle behavior). A11y is load-bearing for forms components — keyboard navigation, screen-reader announcements, and focus management make or break the experience for assistive-tech users.
- **Verified?** yes (with 2 minor findings)
- **Notes:**
  - **Native `<button type="button">`** as the chip element ✓. Tab-navigable + Space/Enter activation work natively. No custom keyboard handlers needed.
  - **`aria-pressed={isActive}`** on each chip ✓. Toggle button pattern is correct semantically (vs `aria-selected` which would imply listbox; vs `aria-checked` which implies radio/checkbox).
  - **`role="group"` + `aria-label={groupLabel}`** on the chip container ✓. `groupLabel` falls back to `title` → `ariaLabel` → `"Categories"` (sane default).
  - **Heading element configurable** via `headingAs="h2"|"h3"|"h4"` (default `h3`) ✓. Lets consumers set the right semantic level for their page hierarchy.
  - **Focus-visible ring** on the button: `focus-visible:ring-2 ring-ring ring-offset-2` ✓. Keyboard focus is visible and matches the project's design-system focus token.
  - **No swallowed clicks**: `<Badge>` is a `<span>` inside the button — clicks bubble correctly.
  - **`value = null` no-selection state** behaves correctly — no chip is pressed; ARIA states all `aria-pressed="false"`. ✓
  - Two minor a11y polish items surfaced — see F-01 + F-02.

## Findings (3)

### F-01 — Inline count not separately labeled for screen readers

- **Severity:** 🔹 Low
- **Dimension:** 7 — Accessibility
- **Location:** `src/registry/components/forms/category-cloud-01/category-cloud-01.tsx:12` + `parts/category-chip.tsx:31-32`
- **Observed:** The default count formatter `(count) => \` (${count})\`` produces a string like " (12)" that is concatenated visually and semantically to the chip label. A screen reader will announce: *"Technology, open paren, 12, close paren, pressed"*. The count's role (number of items in the category) isn't conveyed; the parentheses/digits read as inline label text. Sighted users decode `(12)` as "12 items" instinctively; SR users get the raw string.
- **Suggested fix:** Wrap the count in a visually-equivalent + screen-reader-augmented span: `<span aria-label="${count} items" className="sr-only">${count} items</span>` plus a visually-rendered ` (${count})` for sighted users. Or add `aria-describedby` pointing to a per-chip `<span class="sr-only">12 items</span>`. The current implementation is functional but suboptimal for assistive tech.

### F-02 — Touch target size potentially under 44×44 px on dense clouds

- **Severity:** 🔹 Low
- **Dimension:** 7 — Accessibility (mobile / WCAG 2.5.5)
- **Location:** `src/registry/components/forms/category-cloud-01/parts/category-chip.tsx:21-34`
- **Observed:** Each chip is a `<button>` wrapping a shadcn `<Badge>` (default size). shadcn Badge defaults to `text-xs` + tight padding; in dense category clouds the touch area can fall below 44×44 px (WCAG AAA's "Target Size", also Mobile WCAG). Sidebar use on desktop is fine; mobile breakpoints risk small mis-tap rates. The button's outer `rounded-full` doesn't add padding; the click area equals the badge size.
- **Suggested fix:** Either (a) raise the chip's min-height/width on mobile via `min-h-9 px-3` (or similar Tailwind utility) to clear the 36 px Mobile WCAG threshold (preferred — least intrusive), or (b) document the constraint in the guide so consumers using this in mobile-first surfaces add their own padding via the `className` prop.

### F-03 — `value` prop accepting an unmatched item is silent

- **Severity:** 🔹 Low
- **Dimension:** 7 — Accessibility (and contract)
- **Location:** `src/registry/components/forms/category-cloud-01/category-cloud-01.tsx:54-55`
- **Observed:** When the consumer passes a controlled `value` that doesn't match any item's `value`, no chip is pressed and no warning fires. From the user's UX standpoint this looks identical to "nothing selected" — but the consumer's intent was a particular selection that silently dropped. Common scenarios: stale URL filter persisting after a category was renamed; a typo'd preset; race conditions where items load asynchronously after value is set. SR users see no indication; sighted users likewise see no indication.
- **Suggested fix:** Add a dev-only `console.warn` in development when `value !== null && !items.some(i => (typeof i === "string" ? i : i.value) === value)`. Production builds strip the warning per the project's dev-warn convention (per F-cross-08 closure). Bundlers will tree-shake the dead code.

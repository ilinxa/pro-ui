## Findings (1)

### F-01 — 3-state status (before/active/after) has NO visual color differentiation; bar + marker are always lime

- **Severity:** 🔸 Medium
- **Dimension:** 6 — Design system (semantic state encoding)
- **Location:** `src/registry/components/data/progress-timeline-01/progress-timeline-01.tsx:96-117` (Progress + marker, no status-conditional className)
- **Observed:** The component derives a 3-state machine (`before` / `active` / `after`) from `start`/`end`/`now` (`deriveTimelineState`), and the center caption text changes per state (`labels.beforeText` / `activeText` / `afterText`). But the **visual encoding is identical across all 3 states**:
  - shadcn `<Progress>` indicator: `bg-primary` (lime) — locked by shadcn primitive, no override
  - Marker dot: `bg-primary` (lime) — explicit className, applies regardless of state
  
  Per the project's signal-lime design mandate, lime IS the action / active color. Using it for `before` (event hasn't started — should signal "pending" / muted) and `after` (event ended — should signal "complete" / faded) is semantically off:
  - `before` lime suggests "action available" when actually "wait."
  - `after` lime suggests "active" when actually "complete."
  
  The existing color-AND-text pattern in event-card-01 is the project precedent: status differentiated by both color AND a status icon for color-blind safety (`meta.features` for event-card explicitly says *"Status differentiated by BOTH color AND icon (color-blind safe)"*). progress-timeline-01 is missing the color half.
- **Suggested fix:** Add status-conditional classNames to bar + marker:
  - `before`: bar `[&>*]:bg-muted` (or `bg-muted-foreground/30`); marker `bg-muted-foreground`
  - `active`: bar default (`bg-primary`); marker `bg-primary` (current behavior)
  - `after`: bar `[&>*]:bg-secondary` (or `bg-success` if added); marker `bg-secondary` 
  
  Implementation: derive a `statusColors` object from `effectiveStatus`, apply via `barClassName` + `markerClassName` (which are already public override props). v0.1.1 fix. Color choices should be locked with the design-system owner before shipping.

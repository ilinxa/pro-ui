### Rotating dimension

- **Chosen:** **Public API (slot/callback shapes; F-cross-12 lessons)**
- **Why:** This component introduces ~50 props, three modes (`view` / `edit` / `terminal`) × two variants (client / typed-narrowed RSC scaffold), 8 standalone-exported parts, and 7 different callback shapes. API hygiene is the highest-signal rotating pick — drift here would propagate to every downstream consumer (chat-panel, rich-card code-section, markdown-editor fenced blocks).
- **Verified?** yes, with the findings noted below.
- **Notes:**
  - **Object-shape callbacks throughout:** `onChange({ value })`, `onCopy({ value })`, `onSave({ value })`, `onDownload({ value, filename })`, `onLineClick({ line })`, `onExpandedChange({ expanded })`, `onWrapChange({ wrap })`, `filenameToLang({ filename })`. No positional shapes. ✓
  - **`onCopy` semantics correct:** fires AFTER clipboard write succeeds (`handleCopy` in `code-block.tsx`); is NOT a replacement for the clipboard write. Matches description Q9 + the description-stage re-validation pass refinement. ✓
  - **`renderHeader` slot args:** `{ filename, lang, copyButton, expandButton, wrapButton, downloadButton, trafficLights, actions }` — each conditional part `null` when its `show*` flag is `false`. Matches plan Q13 (refined). ✓
  - **Filename → lang priority chain:** explicit in `lib/lang-resolution.ts#resolveLang`: `lang` > `filenameToLang(filename)` > `FILENAME_OVERRIDES` (dockerfile/makefile special case) > `FILENAME_TO_LANG_MAP` ext lookup > `'plaintext'`. Matches plan. ✓
  - **Imperative handle:** `{ copy, focus, getValue, scrollToLine }` exposed via `ref`. `scrollToLine` is a no-op stub in v0.1.0 — logged as F-02 (low; documented limitation, not a contract violation, since CodeMirror's `scrollIntoView` integration is non-trivial).
  - **`CodeBlockServerProps` typed-narrowed type:** shipped in `types.ts` as scaffolding for the deferred v0.2.0 RSC variant. No runtime use in v0.1.0. ✓ (consumer-facing public API is forward-compatible.)
  - **Standalone parts exported:** 8 atoms (`<CodeBlockHeader>`, `<CodeBlockFilename>`, `<CodeBlockLangPill>`, `<CodeBlockCopyButton>`, `<CodeBlockExpandButton>`, `<CodeBlockWrapButton>`, `<CodeBlockDownloadButton>`, `<CodeBlockTrafficLights>`) all use the internal `useCodeBlock()` context (composable inside `renderHeader` slot without re-passing props). ✓
  - **`useCodeBlock` exported:** advanced consumers can read context inside their own custom header components. Documented in `index.ts` exports.

## Findings (max 5)

#### F-01 — Streaming algorithm is NOT yet append-only

- **Severity:** 🔸 Medium
- **Dimension:** Public API (specifically the v0.1.0 substrate behaviour vs the description's success criteria)
- **Location:** `src/registry/components/code/code-block/hooks/use-shiki-highlighter.ts:88-104`
- **Observed:** The plan's §2 (streaming append-only tokenization) describes a line-boundary diff that reuses cached HTML for stable lines and re-tokenizes only the new tail. The shipped v0.1.0 implementation does a full re-tokenize per render but batches to a single `requestAnimationFrame` (mitigates thrash). The `lib/streaming-cache.ts#diffForRetokenize` helper is locked-in and tested but not yet consumed for the optimisation. In practice this is fine for typical chat-block sizes (~30–100 lines) — Shiki tokenizes < 5 ms after warm-up. For very-long streamed blocks (1000+ lines) the cost is real.
- **Suggested fix:** v0.1.1 — wire `diffForRetokenize` into `useShikiHighlighter` and re-tokenize only `cache.retokenizeSlice`. Add a perf-regression demo to the docs site (1000-line streaming sample) to confirm. The cache helper is already in place; this is ~20 lines.

#### F-02 — `scrollToLine` imperative handle is a no-op stub

- **Severity:** 🔹 Low
- **Dimension:** Public API
- **Location:** `src/registry/components/code/code-block/code-block.tsx:187-189` (commented as v0.2 stub)
- **Observed:** `handle.scrollToLine(line)` exists on the public ref API but does nothing in v0.1.0. Documented inline as reserved for v0.2 CodeMirror integration. Consumers calling it will see no error and no behavior — quietly inert.
- **Suggested fix:** v0.1.1 — at minimum, `console.warn` in dev when called. v0.2.0 — implement via `view.dispatch({ effects: EditorView.scrollIntoView(...) })` for edit mode + native `scrollIntoView` for view/terminal.

#### F-03 — Dedicated RSC variant (`/server`) deferred to v0.2.0

- **Severity:** 🔹 Low
- **Dimension:** Procomp planning docs (description scope drift)
- **Location:** Description + plan both originally committed to a `<CodeBlock.Server>` variant; deferred mid-implementation
- **Observed:** Building a truly zero-client-Shiki RSC variant requires a separate client-island shell that lacks the Shiki import; non-trivial. v0.1.0 ships only the client `<CodeBlock>` (which SSRs fine in Next.js via the standard `'use client'` hydration boundary). The `CodeBlockServerProps` type is shipped as forward-compatible scaffolding. Description was updated (in-scope §SSR posture) to reflect the deferral; matches the P2 substrate-shift pattern.
- **Suggested fix:** v0.2.0 — implement `code-block.server.tsx` + `parts/code-block-server-shell.tsx` (client island accepting `pretokenizedHtml: string` prop). Re-export from `@ilinxa/code-block/server`. Bundle target: <10 KB gz client chunk (Shiki stays server-only).

#### F-04 — Guide.md not authored

- **Severity:** 🔹 Low
- **Dimension:** Procomp planning docs
- **Location:** Missing: `docs/procomps/code-block-procomp/code-block-procomp-guide.md`
- **Observed:** Workflow step 9 ("Author procomp guide.md alongside the implementation") is incomplete. The `usage.tsx` covers consumer-facing recipes and `meta.ts.context` covers the substrate story, so the immediate consumer experience is not blocked — but the guide is the place for the longer-form gotchas (Shiki bundle posture, `@shikijs/codemirror` deferral note, filename→lang precedence, P2 vs v0.2 visual diff, streaming explicit-prop contract).
- **Suggested fix:** v0.1.1 — author `code-block-procomp-guide.md` following the file-tree-procomp-guide.md shape. Cover the substrate decisions (Shiki + custom CM theme), bundle posture (~300 KB gz initial client chunk), v0.2 upgrade paths (streaming append-only + Shiki→CM bridge + RSC variant), and the canonical "drop-into-chat-panel / drop-into-markdown / drop-into-rich-card" recipes.

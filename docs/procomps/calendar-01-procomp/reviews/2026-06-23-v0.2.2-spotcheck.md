## What changed (and why)

The fork team's report listed 4 correctness, 1 API, 2 design, 5 polish items. Each was re-validated first-hand against the source before acting. Outcome:

| ID | Report sev | Validation | Action in v0.2.2 |
|---|---|---|---|
| **A1** `changePriority`/modal emit no granular event | MED | **Valid — and understated.** The modal/inspector detail editor echoed via `onChange` only (the embedded `<TodoRichCard>`'s own granular events were dropped by `EventEditorPanel`). | **Fixed.** `applyEditedSubtree` now diffs prev↔next and emits `onFieldEdited` for each changed `TodoEditableField` (name/description/status/active/setAt/startAt/expireAt/duration). `priority` documented as `onChange`-only (it is not a `TodoEditableField`; the cross-procomp type extension was deliberately **not** taken). `TodoEditableField` re-exported from the calendar barrel. |
| **C3** `effectiveEnd` drops `dateOnly` for duration-derived end | LOW | Valid but **inert today** (`end.dateOnly` is only read at `classify.ts:76`, where `start.dateOnly` already dominates). | **Fixed** as a robustness tidy — propagates `effectiveStart(item).dateOnly`. No behavior change. |
| **C4** `occurrencesInRange` dead + inclusive-end | LOW | Valid — confirmed uncalled and **not even in the public barrel**. | **Deleted.** |
| **P1** dead `timeLabel` milestone branch | nit | Valid — **and hid a real UX bug**: milestones carry `allDay=true`, so they hit the "All day" branch first → mislabeled, losing their time. | **Fixed** — milestone check moved above `allDay`; shows `◇ <time>`. |
| **P2** `periodLabel` `sameYear` no-op | nit | Valid — both ternary arms identical. | **Fixed** — same-year shows the year once (on the end); cross-year shows it on both. |
| **P3** skeleton `35` ≠ `42` | nit | Valid — `monthGrid` returns up to 6 weeks → load jump. | **Fixed** — reserve 42. |
| **P4** `initials()` empty on whitespace name | nit | Valid. | **Fixed** — trim + filter + `"?"` fallback. |

**Deliberately NOT changed:**

- **C2** (`coveredDays` exclusive-midnight not gated on `allDay`) — **report code-accurate but the defect framing is disputed.** A timed event ending exactly at midnight has zero extent on the next day and is already excluded by the time-grid (`o.endMs > dayStartMs`); the half-open `[start, end)` model is uniform across the component. The proposed fix would make Month show the event on a day the week/day grid does **not** → a new inconsistency. Left as-is. (Documented here so it isn't re-raised.)
- **C1** (DST intra-day geometry), **D1** (all-day band cap), **D2** (cross-midnight timed continuation cue), **P5** (mini-nav focus) — carried to follow-ups (see below). C1 touches the time-grid math and warrants its own verification; D1/D2/P5 are design calls.

### Rotating dimension

- **Chosen:** Public API.
- **Why:** the one substantive change (A1) is an event-surface change. Even though additive, emitting *more* events on an existing callback is observable to consumers, so API correctness + minimality is the right axis.
- **Verified?** yes.
- **Notes:**
  - No double-fire: `EventEditorPanel` still passes only `onChange` to the embedded `<TodoRichCard>`, so the card's own granular events remain unwired; the new emits come solely from `applyEditedSubtree`'s diff. Status fires once (`onStatusChanged` kept; `onFieldEdited(status)` added — matching `changeStatus`'s existing both-fire shape, now symmetric across paths).
  - `priority` exclusion is honest and documented at the type (`CalendarProps.onFieldEdited` JSDoc), the guide, and in `changePriority`/`applyEditedSubtree` comments — the cross-procomp `TodoEditableField` extension was the explicitly-rejected alternative.
  - Additive `TodoEditableField` re-export gives consumers wiring `onFieldEdited` the key type without a second import (same pattern as the other re-exported `Todo*` types).

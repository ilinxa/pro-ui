## Findings (3)

#### F-01 — Optimize `enforceUniqueKeys` allocation under stable items

- **Severity:** 🔹 Low
- **Dimension:** Public API / Performance
- **Location:** `lib/enforce-unique-keys.ts:8-29`
- **Observed:** The dedup pass always allocates a new `Set` + `Array` + `duplicateKeys[]` even when the input has no duplicates. Wrapped in `useMemo([items])` so it runs once per items-reference change, but a fast-path "no dupes found, return input as-is" branch would avoid the allocation entirely in the common case.
- **Suggested fix:** Pre-scan `items` for duplicates; only allocate the deduped array when at least one duplicate is found. Defer to v0.1.x if real perf signal surfaces (>100 items + frequent items-array re-creation).

#### F-02 — Path-b consumer-tsc smoke deferred to post-deploy

- **Severity:** 🔸 Medium
- **Dimension:** Verification
- **Location:** Project verification protocol (not a code location)
- **Observed:** Path-b smoke not run pre-ship — the harness at `e:/tmp/ilinxa-smoke-consumer/` runs against the live Vercel artifact, which requires push first. This follows the established pattern from the last 4 ships (rich-sidebar v0.1.0, todo-tree v0.1.1, todo-rich-card v0.1.1, json-form v0.2.x).
- **Suggested fix:** Run smoke against locally-served `public/r/account-switcher-01.json` immediately after push lands on Vercel. Same-day patch loop reserved for v0.1.1 if F-cross-13 surfaces on Popover.onOpenChange. Defensive guard already pre-emptive — risk is low but not zero.

#### F-03 — Source comments on Q4 "right-then-flip" semantics could be clearer

- **Severity:** 🔹 Low
- **Dimension:** Public API (consumer comprehension)
- **Location:** `types.ts:38-44` JSDoc on `CollapsedPopoverSide`
- **Observed:** The JSDoc says `"right" (collision-aware auto-flip applied by Radix Popover)` — accurate but doesn't mention that the auto-flip happens INSIDE Radix and the consumer can't disable it. A consumer who wants strictly-right-no-flip will be surprised when Radix flips to left near the viewport edge.
- **Suggested fix:** Append to JSDoc: "Note: Radix's collision detection always applies; library does not expose a `disableCollisionDetection` prop in v0.1. If you need strict positioning, wrap the trigger in a parent with explicit positioning constraints."

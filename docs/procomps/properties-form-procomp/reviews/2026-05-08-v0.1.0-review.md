## 1. Executive summary

Properties-form is the schema-driven controlled read/edit form Tier 1 component for the graph-system, built without an external form library (per Q10 lock — "build from scratch, ~200-400 LoC"). The implementation lands the locked decisions cleanly: 3-state per-field permissions with layered resolution (host predicate → declarative → editable default), counter-based dirty tracking, sync per-field + form-level validation with first-error focus, async `onSubmit` with submit-id-token suppression of stale dispatches, schema-instability dev-only warning, React 19 ref-as-prop pattern preserving the generic. The 25-file structure matches plan §8.1's listing exactly (modulo the count claim — see F-05).

The findings cluster around the ecosystem: 8 `process.env.NODE_ENV` occurrences across 4 files (F-cross-08 carrier — now 3/9 Tier 1); the same phantom `radix-ui` npm dep in `meta.ts` that markdown-editor carries (F-03; now 2 of 3 forms components); over-declared shadcn primitives (`tabs` + `badge` are demo-only but listed in `meta.shadcn`); and the missing `guide.md` (F-cross-01 carrier; second of three Tier 1 holdouts).

**Headline finding:** F-03 — `meta.ts` over-declares deps in two shapes — phantom `radix-ui: ^1.4.3` npm entry that no source file imports + `tabs` and `badge` shadcn entries that only the docs-site `demo.tsx` uses. Both bloat consumer installs unnecessarily and diverge from `registry.json`'s `registryDependencies` (which is correctly minimal at 6 primitives).

**Recommended next step:** v0.1.1 patch removing the phantom `radix-ui` + over-declared `tabs`/`badge` from `meta.ts` (F-03), authoring `guide.md` (F-02), and resolving F-cross-08 sweep-wide at session 7's mid-checkpoint (F-01 + the entity-picker carrier paired here). F-04, F-05, F-06, F-07 are docs/meta touchups.

### 3.2 High (⚠️)

#### F-01 — F-cross-08 carrier — heavy `process.env.NODE_ENV` use (8 occurrences across 4 files)

- **Severity:** ⚠️ High
- **Dimension:** 7 (Portability — sealed-folder rules) + 14 (Cross-cutting findings).
- **Location:** 8 occurrences across:
  - [`properties-form.tsx:89`](../../../../src/registry/components/forms/properties-form/properties-form.tsx#L89), `:212`, `:275`, `:298` (4 occurrences)
  - [`lib/validate.ts:14`](../../../../src/registry/components/forms/properties-form/lib/validate.ts#L14), `:62` (2)
  - [`lib/resolve-permission.ts:19`](../../../../src/registry/components/forms/properties-form/lib/resolve-permission.ts#L19) (1)
  - [`parts/field-row.tsx:88`](../../../../src/registry/components/forms/properties-form/parts/field-row.tsx#L88), `:154` (2)
- **Observed:** All gates use the same pattern (`process.env.NODE_ENV !== "production"`) to throttle dev-warns/errors to non-production builds. Plan implicitly relies on this (multiple §10 edge cases say "Dev-only `console.error`") but doesn't lock the mechanism the way entity-picker plan §12.5 #5 explicitly does. Description §6 #10 / plan §8 portability rule "no `process.env`" is violated — F-cross-08 now confirmed at **3 of 9 Tier 1** (flow-canvas-01: 4 occurrences; markdown-editor: 1; properties-form: 8).
- **Why it matters:** Properties-form is the densest carrier of the violation among reviewed components. The entity-picker plan §12.5 #5 explicitly locks the `process.env.NODE_ENV` gate as the intended pattern ("Bundlers strip the dead code"), affirming the F-cross-08 recommendation to relax description §6 #10 sweep-wide. Properties-form's heavy use here is the strongest single-component evidence that the rule needs amending.
- **Suggested fix:** Resolve F-cross-08 sweep-wide at session 7. Recommendation per the tracker: relax description §6 #10 to allow `NODE_ENV` dev-warn gates explicitly + document the exception. Don't patch properties-form in isolation. The 8 sites here all follow the same idiom and would either survive (relax-criterion path) or migrate to a shared `IS_DEV` constant (architecturally-pure path) cleanly.

#### F-03 — `meta.ts` over-declares deps (phantom `radix-ui` + demo-only `tabs`/`badge`)

- **Severity:** ⚠️ High
- **Dimension:** 10 (meta + manifest sync) + 7 (Portability — declared deps must reflect actual imports).
- **Location:** [`meta.ts:33`](../../../../src/registry/components/forms/properties-form/meta.ts#L33) + `:36`.
- **Observed:** Two shapes of over-declaration:
  - **(a) Phantom `radix-ui: ^1.4.3` npm entry.** No source file imports `radix-ui` directly — all Radix usage flows through shadcn primitives (`@/components/ui/{input,select,switch,textarea,tooltip,button}`). Same shape as markdown-editor F-03 from session 5. Phantom-dep flag now hits **2 of 3 forms components** (markdown-editor + properties-form; entity-picker paired-review of this session is the 3rd; see entity-picker F-03).
  - **(b) `tabs` + `badge` shadcn entries used only by `demo.tsx`.** Source grep confirms `@/components/ui/tabs` and `@/components/ui/badge` appear ONLY in [`demo.tsx`](../../../../src/registry/components/forms/properties-form/demo.tsx) (which is excluded from the registry artifact). Shipped parts use `button`, `input`, `select`, `switch`, `textarea`, `tooltip` (6 primitives). The `registry.json` `registryDependencies` correctly lists only those 6 — but `meta.ts` declares 8.
- **Why it matters:** The discrepancy between `meta.ts` (8 shadcn primitives + phantom npm) and `registry.json.registryDependencies` (correct 6 primitives) means the docs-site rendering of "what to install with this component" disagrees with what `pnpm dlx shadcn add @ilinxa/properties-form` actually pulls. Consumer installs work correctly (registry is the source of truth for installs); but the docs-site dep panel surfaces phantom + demo-only deps to readers, making the install footprint look heavier than it is.
- **Suggested fix:** Three edits in `meta.ts`:
  1. Remove `"radix-ui": "^1.4.3"` from `dependencies.npm`.
  2. Drop `"tabs"` from `dependencies.shadcn` (demo-only; shipped parts don't use it).
  3. Drop `"badge"` from `dependencies.shadcn` (same reason).
  Result: `dependencies.shadcn = ["button", "input", "select", "switch", "textarea", "tooltip"]` matching registry.json exactly; `dependencies.npm = { "lucide-react": "^1.11.0" }` minimal. Verify via fresh smoke install (once F-cross-09 is resolved): consumer's install diff matches what registry.json projected.

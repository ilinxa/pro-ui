## 1. Executive summary

Markdown-editor is the heaviest Tier 1 pro-component by bundle weight (~180KB ceiling), built on a CodeMirror 6 substrate with a custom `marked`-based preview, runtime-updatable wikilink decoration via CM6 `StateField` + `StateEffect`, and a generic-typed candidate API. The implementation is **structurally faithful to the plan to a remarkable degree** — 28 files match plan §8.1 exactly; the four sync effects (value, readOnly, candidates, extensions) in [`use-codemirror.ts:178-212`](../../../../src/registry/components/forms/markdown-editor/hooks/use-codemirror.ts#L178-L212) are wired exactly as plan §5.2 specified; the `SyncAnnotation` echo guard at line 134-144 lands the Q-P9 contract; the per-instance `new Marked()` in [`parse-markdown.ts:38`](../../../../src/registry/components/forms/markdown-editor/lib/parse-markdown.ts#L38) honors validate-pass refinement #1; `useDeferredValue` schedules the preview parse per Q-P7. The 8-tab demo (Basic / View modes / Wikilinks / Custom toolbar / Read-only / onSave / No toolbar / Imperative handle) covers all 12 success criteria.

The findings target the ecosystem more than the code. The component **carries F-cross-08** with a single `process.env.NODE_ENV` gate in `use-imperative-handle.ts:12` (now 2 of 5 Tier 1 carriers), **declares a phantom `radix-ui` npm dep** in `meta.ts` that no source file imports, and **ships without `guide.md`** (F-cross-01 carrier — third of the three original missing-guide Tier 1 components, the others being `properties-form` + `entity-picker` scheduled for session 6). These three are bounded and don't reflect on the editor's substrate work.

**Headline finding:** F-03 — `meta.npm` declares `radix-ui: "^1.4.3"` that no markdown-editor source file imports; the actual Radix usage flows through shadcn primitives (`@/components/ui/{badge,button,tabs,tooltip}`) which are correctly declared in `dependencies.shadcn`. The phantom entry causes consumers running `pnpm dlx shadcn add @ilinxa/markdown-editor` to install an unused 1.4.3 meta-package.

**Recommended next step:** v0.1.1 patch removing the phantom `radix-ui` entry from `meta.npm` (F-03), wrapping `process.env.NODE_ENV` in a project-level `IS_DEV` constant or relaxing the criterion sweep-wide via F-cross-08 (F-01), and authoring `guide.md` (F-02). F-04, F-05, F-06 are docs touchups; F-07 is trivia.

## 2. Strengths

- **Plan §8.1 file count exact** — 28 files at the source paths the plan called for (4 hooks, 4 lib utilities, 6 lib/extensions, 6 parts, plus root + types + meta + index + demo + dummy-data + usage + default-toolbar). Largest pro-component plan in the registry; landed line-for-line.
- **Q-P9 echo-guard implemented as planned.** [`hooks/use-codemirror.ts:133-144`](../../../../src/registry/components/forms/markdown-editor/hooks/use-codemirror.ts#L133-L144) wires the `EditorView.updateListener` to skip `onChange` when the dispatched transaction carries `SyncAnnotation`. The companion sync effect at line 178-190 dispatches with `annotations: SyncAnnotation.of(true)` exclusively when the value-prop differs from both the last-synced ref AND the live CM6 doc. `lastSyncedValueRef = useRef(value)` initialization (line 68) matches validate-pass refinement #10. No infinite loop possible from `setValue`-inside-`onChange` patterns.
- **Q-P5 runtime-updatable candidates via `StateField`.** [`lib/extensions/wikilink.ts:14-25`](../../../../src/registry/components/forms/markdown-editor/lib/extensions/wikilink.ts#L14-L25) defines `setCandidatesEffect` + `candidatesField`; the `wikilinkPlugin` (line 34-72) reads candidates from the field per `update()` cycle and rebuilds decorations via `MatchDecorator.createDeco` when candidates change OR `MatchDecorator.updateDeco` when only the doc changed. Phase A end gate's "candidates change at runtime → broken-link styling flips without remount" contract holds.
- **Q-P1 per-instance `new Marked()`.** [`lib/parse-markdown.ts:38-39`](../../../../src/registry/components/forms/markdown-editor/lib/parse-markdown.ts#L38-L39) creates a fresh `customMarked` and applies the wikilink extension to it — no global `marked.use(...)` call that would pollute every other consumer of `marked` in the same bundle. Validate-pass refinement #1 honored.
- **Q-P7 `useDeferredValue` for preview parse.** [`markdown-editor.tsx:111-121`](../../../../src/registry/components/forms/markdown-editor/markdown-editor.tsx#L111-L121) wraps `value` with `useDeferredValue(value)` and feeds the deferred value into a `useMemo`-keyed `parseMarkdown` call. No internal debounce; React 19 schedules the parse at low priority. Matches validate-pass refinement #9.
- **Generic typing preserved through ref-as-prop.** [`markdown-editor.tsx:40-42`](../../../../src/registry/components/forms/markdown-editor/markdown-editor.tsx#L40-L42) signature `MarkdownEditor<TCandidate extends WikilinkCandidate = WikilinkCandidate>(props: MarkdownEditorProps<TCandidate>)`. Q-P10 ref-as-prop pattern lands cleanly via [`useMarkdownEditorHandle`](../../../../src/registry/components/forms/markdown-editor/hooks/use-imperative-handle.ts) called inside the function body.
- **Demo covers all 12 success criteria.** [`demo.tsx`](../../../../src/registry/components/forms/markdown-editor/demo.tsx) ships 8 sub-demos (Basic / View modes / Wikilinks / Custom toolbar / Read-only / onSave / No toolbar / Imperative handle) — matches plan §B end-gate "8 sub-demos covering description success #12".
- **Container-query split-view.** [`markdown-editor.tsx:189-192`](../../../../src/registry/components/forms/markdown-editor/markdown-editor.tsx#L189-L192) uses `@container` + `@[480px]:flex-row` Tailwind v4 native container queries — a markdown-editor in a 320px sidebar stacks vertically regardless of viewport width. Matches description §8.5 #2.
- **Symmetric wikilink grammar via single-source-of-truth.** [`lib/wikilink-grammar.ts`](../../../../src/registry/components/forms/markdown-editor/lib/wikilink-grammar.ts) hosts `WIKILINK_PATTERN` source string + `makeWikilinkRegex()` factory; consumed by `lib/extensions/wikilink.ts` (CM6 `MatchDecorator`) AND `lib/parse-markdown.ts` (`marked` extension) AND `lib/extensions/wikilink-autocomplete.ts`. Q-P2 lock honored. Validate-pass refinement #7's factory pattern (fresh `g`-flag instance per call) avoids stateful-regex bugs.
- **Mount-time prop snapshot pattern.** [`use-codemirror.ts:73-85`](../../../../src/registry/components/forms/markdown-editor/hooks/use-codemirror.ts#L73-L85) defensively snapshots `value`, `readOnly`, `wikilinkCandidates`, `userExtensions`, `ariaLabel` into refs that the mount effect reads — avoids the closure-captures-initial-render-stale-props footgun. STATUS-row plan deviation 3 ("mount-time prop snapshot") is correctly applied.
- **`meta.related` populated** — `["properties-form", "detail-panel", "filter-stack", "entity-picker"]` cross-links to the four sibling Tier 1 components. Better hygiene than workspace's empty `related`.

### 3.2 High (⚠️)

#### F-01 — F-cross-08 carrier — `process.env.NODE_ENV` violates portability rule

- **Severity:** ⚠️ High
- **Dimension:** 7 (Portability — sealed-folder rules) + 14 (Cross-cutting findings).
- **Location:** [`src/registry/components/forms/markdown-editor/hooks/use-imperative-handle.ts:12`](../../../../src/registry/components/forms/markdown-editor/hooks/use-imperative-handle.ts#L12) — `const DEV = process.env.NODE_ENV !== "production";`
- **Observed:** Single `process.env.NODE_ENV` gate at module scope (used to throttle the `getView() called before mount` console.warn to dev-only). Description §6 #10 / plan §8 portability rule "no `process.env`" is violated — this brings F-cross-08 to **2 of 7 Tier 1 reviewed** (flow-canvas-01 had 4 occurrences in session 3; the other 5 reviewed Tier 1 components — kanban + rich-card + article-body-01 + data-table + workspace — are clean).
- **Why it matters:** The criterion's intent is portability for the eventual NPM extraction. `process.env.NODE_ENV` is universally compiled out by Webpack / Rollup / Turbopack / esbuild / Vite, so the runtime cost is zero in production bundles. But the rule is currently absolute and the violation is real per the rule. F-cross-08's recommendation is to relax the criterion sweep-wide rather than fix per-component — this finding affirms that recommendation.
- **Suggested fix:** Pick the F-cross-08 resolution sweep-wide. Two options: (a) Relax description §6 #10 to allow `process.env.NODE_ENV` dev-warn gates explicitly; document the exception. (b) Replace with a project-level `IS_DEV` constant — e.g., `const IS_DEV = typeof process !== "undefined" && process.env?.NODE_ENV !== "production";` exported from a shared util — and forbid raw `process.env` references. Recommendation per F-cross-08 is option (a): NODE_ENV gates are standard React-ecosystem practice and bundlers compile them out. Until resolution, log this carrier and move on; don't patch markdown-editor in isolation.

#### F-02 — F-cross-01 carrier — `guide.md` missing

- **Severity:** ⚠️ High
- **Dimension:** 1 (Procomp planning docs).
- **Location:** [`docs/procomps/markdown-editor-procomp/`](../../../../docs/procomps/markdown-editor-procomp/) contains only `markdown-editor-procomp-description.md` + `markdown-editor-procomp-plan.md`. No `markdown-editor-procomp-guide.md`.
- **Observed:** The component-guide §2 procomp gate requires three documents per component: description (what & why), plan (how it was built), and guide (how to use it). markdown-editor ships description + plan but no guide. Per F-cross-01 in the sweep tracker, markdown-editor is one of three Tier 1 carriers (alongside `properties-form` and `entity-picker`, both scheduled for session 6).
- **Why it matters:** Markdown-editor is the heaviest pro-component (28 files; ~180KB; 7 type exports; 1 component + 1 helper + 1 constant export; CodeMirror 6 substrate with a non-trivial extension stack). Consumers integrating it need consumer-facing prose docs beyond `usage.tsx` — the workspace `guide.md` is a strong template (gotchas, composition patterns, persistence cookbook). `usage.tsx` covers ~70% of what a guide would: when/when-not, quick start, wikilinks, force-graph integration recipe, custom toolbar, reference-stability, imperative handle, notes. The remaining gap: known limitations / deferred-to-v0.2 inventory (currently scattered through description §3), force-graph v0.5 + detail-panel composition deeper-dive (the recipe is in `usage.tsx` but not the why), `getView()` substrate-leak risk (in description §8.5 #6 but not user-facing), bundle weight discipline (in plan §11.2).
- **Suggested fix:** Author `markdown-editor-procomp-guide.md` alongside the v0.1.1 patch. Use the workspace guide as a structural template. Estimated 2 hours of focused work. Alternative: promote `usage.tsx` content to the guide and re-derive `usage.tsx` from the guide — avoids double-maintenance.

#### F-03 — Phantom `radix-ui` npm dep in `meta.ts`

- **Severity:** ⚠️ High
- **Dimension:** 10 (meta + manifest sync) + 7 (Portability — declared deps must reflect actual imports).
- **Location:** [`meta.ts:53`](../../../../src/registry/components/forms/markdown-editor/meta.ts#L53) — `"radix-ui": "^1.4.3"` in `dependencies.npm`.
- **Observed:** Grep across the entire markdown-editor folder finds **zero `radix-ui` imports** — all Radix usage flows through shadcn primitives (`@/components/ui/badge`, `@/components/ui/button`, `@/components/ui/tabs`, `@/components/ui/tooltip`), which are already correctly declared in `dependencies.shadcn`. The producer's `package.json` line 59 has `"radix-ui": "^1.4.3"` (the meta-package), but that's a producer-pkg artifact unrelated to markdown-editor's actual imports. `meta.ts` `dependencies.npm` should reflect what the **component** imports, not what the producer happens to have installed.
- **Why it matters:** When a consumer runs `pnpm dlx shadcn add @ilinxa/markdown-editor`, the registry artifact's npm-deps list drives the install plan. A phantom `radix-ui: ^1.4.3` entry pulls in an unused meta-package on the consumer's side, increasing install diff and possibly pulling in unintended Radix-version conflicts with the consumer's existing primitives. Mirrors the F-cross-07 family ("dep declaration drift") but in a different shape — phantom rather than wrong-version.
- **Suggested fix:** Remove `"radix-ui": "^1.4.3"` from `meta.ts` `dependencies.npm`. The shadcn primitives in `dependencies.shadcn` will resolve their own Radix deps via shadcn CLI's primitive-install flow. Verify with a fresh smoke install from a tmp consumer (once F-cross-09 — see §7 — is resolved): the consumer's `node_modules` should contain only `@radix-ui/react-*` packages installed by the shadcn primitives, not the umbrella `radix-ui`. Trivial fix; v0.1.1.

#### F-05 — Plan §3.3 vs source: `getView()` return type

- **Severity:** 🔸 Medium
- **Dimension:** 1 (plan-vs-code drift).
- **Location:** Plan §3.3 specifies `getView(): EditorView` (non-null); [`types.ts:74`](../../../../src/registry/components/forms/markdown-editor/types.ts#L74) declares `getView(): EditorView | null`; plan §10 documents the null edge case ("`getView()` called before mount → returns `null`; dev-only `console.warn` once per session"); [`hooks/use-imperative-handle.ts:41-49`](../../../../src/registry/components/forms/markdown-editor/hooks/use-imperative-handle.ts#L41-L49) implements the documented edge-case behavior correctly.
- **Observed:** Source is more correct than plan §3.3 — the function genuinely can return null pre-mount or post-unmount. Plan §3.3 has the type spec wrong; plan §10 has the behavior spec right. Internal contradiction in the plan doc.
- **Why it matters:** Type spec is the consumer's first read on the API surface; current spec lies about nullability. Future reviewers comparing plan to code surface this as a drift even though the runtime behavior is correct.
- **Suggested fix:** Update plan §3.3 to `getView(): EditorView | null` with a sentence linking to §10's edge-case section. v0.1.1 docs cleanup.

#### F-06 — Toolbar separator empty-label sentinel undocumented

- **Severity:** 🔸 Medium
- **Dimension:** 1 (Plan/Type-surface clarity) + 6 (Composition pattern).
- **Location:** [`default-toolbar.tsx:50`](../../../../src/registry/components/forms/markdown-editor/default-toolbar.tsx#L50) — `{ id: "sep-1", label: "", run: () => {} }` and [`demo.tsx:106`](../../../../src/registry/components/forms/markdown-editor/demo.tsx#L106) (`{ id: "sep-2", label: "", run: () => {} }`).
- **Observed:** Visual separators in the toolbar are encoded as `ToolbarItem` entries with empty `label: ""` and no-op `run`. The `<Toolbar>` component must detect this sentinel and render a separator instead of a button. This is an undocumented convention — `ToolbarItem`'s shape in plan §3.1 + types.ts says `label: string` (required, no special handling); the empty-string sentinel is a hidden contract.
- **Why it matters:** Hosts spreading `defaultMarkdownToolbar` and adding their own items must guess that empty-label sentinels are visual-only. Plan §13.5 (refinements) #7 mentions a separator at index 5, but the consumer-facing API doesn't surface the convention. Type-safety is also weakened — `label: string` permits empty strings, but they have a non-obvious side effect.
- **Suggested fix:** Pick one. (a) Add a discriminated `type: "separator"` to `ToolbarItem` (breaking type change — defer to v0.2). (b) Document the empty-label sentinel in plan §3.1 + `usage.tsx` "Custom toolbar" section + `default-toolbar.tsx` JSDoc; keep the current API. Recommendation: (b) for v0.1, queue (a) for v0.2's render-fn-toolbar additive expansion.

## Findings (2)

### F-01 — `onClick(project, mouseEvent)` positional callback — third occurrence of this pattern across the library

- **Severity:** 🔹 Low
- **Dimension:** 2 — Public API (versioning / future-proofness)
- **Location:** `src/registry/components/data/project-card-01/types.ts` (`onClick?: (project, event) => void`) + `project-card-01.tsx:42-58` (passed through to variant parts)
- **Observed:** `onClick?: (project: ProjectCardItem, mouseEvent: MouseEvent) => void` — positional callback args. Same shape as:
  - `content-card-news-01` F-01 (session 10) — `onClick(item, event)`
  - `grid-layout-news-01` F-02 (session 8) — `renderItem(item, slot)` (also positional, render-prop variant)
  
  **Three occurrences.** Per Dim 2 review-guide bad signal: *"onChange callback receives positional args (onChange(id, value)) instead of an object — positional callbacks are a versioning trap."* If v0.2 wants to add `index` (which slot in a tower), `source` (which list), or `variant` (the visual variant the user clicked), the signature must grow with new positional args, breaking existing consumers.
  
  This is starting to look like a **library-wide pattern worth surfacing as a F-cross-NN at sweep close** — currently 3/16 Tier 2 components have this shape; a systematic audit across the remaining cards in sessions 12 + revisited Tier 1 components could quantify the scope per the audit-systematic-scope memory.
- **Suggested fix:** v0.1.x: per-component, leave as-is (changing now is breaking). At sweep close (s13 rollup): consider flagging as a F-cross-NN with a library-wide migration plan to object-shape callbacks in v0.2. Each component's v0.2: `onClick?: (args: { project: ProjectCardItem; event: MouseEvent; ... }) => void`.

### F-02 — `href` precedence chain has 4 sources; consumer passing multiple may be unclear about which wins

- **Severity:** 🔹 Low
- **Dimension:** 2 — Public API (clarity)
- **Location:** `src/registry/components/data/project-card-01/project-card-01.tsx:36-37`
- **Observed:** `const resolvedHref = (getHref ? getHref(project) : undefined) ?? href ?? project.href ?? "#";` — 4-level precedence chain:
  1. `getHref(project)` callback prop
  2. `href` prop
  3. `project.href` data field
  4. `"#"` fallback
  
  `meta.features` documents this as "href precedence chain — getHref(project) > href > project.href > '#'". Implementation matches docs. But content-card-news-01 (a sibling card) uses simpler `href ?? "#"` (no `getHref`, no `project.href` fallback). Different cards = different href patterns = potential consumer confusion when composing both in the same surface (e.g., grid-layout-news-01 hosting both via `renderItem`). Documented but worth flagging as a Dim 14 cross-coherence note (sibling cards drift in href-resolution complexity).
- **Suggested fix:** v0.1.x docs-only: add a sentence to the project-card-01 guide cross-referencing content-card-news-01's simpler pattern, so consumers composing both know what's expected. v0.2: consider consolidating href-resolution across all card primitives (4-level chain or 1-level `href` prop only — pick one, apply consistently). At sweep close (s13), evaluate whether this is a Dim 14 finding worth promoting to F-cross-NN.

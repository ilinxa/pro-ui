## 1. Executive summary

Workspace is the most ambitious component reviewed so far ‚Äî a 26-file flat-list-rendered binary-tree layout primitive with corner-drag splits, edge-drag resize, controlled/uncontrolled state, per-leaf depth caps, and a renderer-registry pattern that lands as a third datapoint after kanban + flow-canvas-01. The implementation hits the headline contracts cleanly (state preservation on split via stable area-ids as React keys, hard cap honored per-leaf, `useReducer` with pure tree ops, `role="application"` landmark) and is paired with gold-standard planning docs (12 locked Qs, 6 locked Q-Ps, complete guide).

The implementation is generally faithful to the plan, but two contract drifts surfaced. The **adaptive-flatten branch silently discards content** at the tablet breakpoint when the existing tree exceeds the cap (Q12 contracts "deeper subtrees flatten to card stacks within their parent region"; implementation collapses each over-depth subtree to its first leaf and renders only that ‚Äî every other leaf disappears from the screen until the viewport widens). This is a real UX defect and the headline finding. Separately, the **plan-locked `Alt+Enter` keyboard menu shortcut isn't wired** ‚Äî the implementation provides keyboard-equivalent functionality through an always-visible header chevron-button menu, but neither the keymap shortcut nor the deviation are documented (the plan was explicit at ¬ß15: "deviations are loud, not silent").

**Headline finding:** F-01 ‚Äî adaptive flatten violates Q12 "deeper subtrees flatten to card stacks within their parent region" ‚Äî `lib/tree.ts`'s `collapseToFirstLeaf` discards every leaf except the first when a viewport shift makes the tree exceed the new cap.

**Recommended next step:** v0.1.1 patch fixing F-01 (replace `collapseToFirstLeaf` with a stack-region representation rendered by `<Area>` as a `<CardStack>`) and F-02 (either wire the `Alt+Enter` shortcut as planned OR amend plan ¬ß5.3 + usage.tsx to lock the chevron-button-as-keyboard-menu pattern). All other findings are non-blocking.

### 3.2 High (‚öÝÔ∏è)

#### F-01 ‚Äî Adaptive flatten silently discards content past cap

- **Severity:** ‚öÝÔ∏è High
- **Dimension:** 6 (Composition pattern) + 13 (Edge cases) ‚Äî the responsive-collapse contract from description Q12 + plan ¬ß6.2.
- **Location:** [`src/registry/components/layout/workspace/lib/tree.ts:123-142`](../../../../src/registry/components/layout/workspace/lib/tree.ts#L123-L142) (`flattenSubtreesPastDepth` + `collapseToFirstLeaf`).
- **Observed:** When the breakpoint shifts and the existing tree exceeds the new cap (e.g., desktop 4-leaf depth-3 tree resized to tablet with `maxSplitDepth.tablet=2`), `flattenSubtreesPastDepth` walks down and at each over-depth subtree calls `collapseToFirstLeaf(subtree)`, which descends `tree.a` recursively until it hits a leaf and returns that single leaf. Every other leaf in the over-depth subtree is **not rendered**. `workspace.tsx` then renders the resulting tree as a regular split layout (via `computeLayout`); the "missing" leaves are simply absent from the visible UI.
- **Why it matters:** Description ¬ß2 in-scope and Q12 explicitly contract: "deeper subtrees flatten to card stacks within their parent region. The tree is preserved internally and resizing back up restores the full layout." Plan ¬ß6.2 reiterates: "tablet: ... deeper subtrees become CardStack regions (no internal splits / no corner handles)". The implementation preserves the input tree (so widening back to desktop restores all leaves ‚Äî half-correct on the contract), but at tablet **content is invisible** during the in-cap window. A user with a Counter, Notes, Clock, and Data Table loses 3 of 4 widgets when the viewport hits tablet width with a deep layout. Silent content loss, not graceful flattening.
- **Suggested fix:** Replace the over-depth-collapse path with a "stack region" representation. Concretely: introduce a `{ kind: "stack"; id: string; leaves: AreaTreeLeaf[] }` discriminator (or a sentinel leaf carrying the leaf list); update `computeLayout` and `<Area>` rendering to detect a stack node and render it via `<CardStack>` filling its allocated rect; preserve the original tree alongside as today (resizing back to desktop already restores correctly because the ref tree was untouched). Phase A end gate from plan ¬ß8.2 doesn't cover this case ‚Äî add a fixture: "open a 4-leaf depth-3 tree at desktop ‚Üí resize to tablet with cap 2 ‚Üí all 4 leaves visible (in stacked regions); resize back to desktop ‚Üí 4-leaf split tree restored". Until fix lands, document the gap as a v0.1 known limitation (currently absent from `guide.md`'s "Known limitations" section).

#### F-02 ‚Äî Plan-locked keyboard contract diverged silently

- **Severity:** ‚öÝÔ∏è High
- **Dimension:** 1 (Procomp planning docs ‚Äî plan-vs-code drift) + 11 (A11y).
- **Location:** Plan ¬ß5.3 vs implementation in [`parts/area-header.tsx:52-122`](../../../../src/registry/components/layout/workspace/parts/area-header.tsx#L52-L122) and [`usage.tsx:73-83`](../../../../src/registry/components/layout/workspace/usage.tsx#L73-L83).
- **Observed:** Plan ¬ß5.3 locked `Alt+Enter` on a focused area to open an "Area actions" menu (split / merge with neighbor / change component / close). The implementation does not wire `Alt+Enter` anywhere ‚Äî instead it always renders a `ChevronsUpDownIcon` button in each area's header (top-right) that, when clicked or keyboard-activated, opens the same menu. `usage.tsx` documents the chevron-menu pattern but not the divergence ("Header chevron menu ‚Äî split / merge / pick a component (works without a mouse)"). The `Alt+Shift+Arrow` resize IS wired (workspace.tsx:277-321) and works as planned.
- **Why it matters:** Plan ¬ß15 explicitly warned: "If at any point during implementation a plan decision turns out wrong, stop, document the issue, ask before deviating. The plan is the contract; deviations are loud, not silent." The chevron-menu pattern provides functional keyboard parity (Tab cycles to the chevron, Enter opens), so a11y is intact, but the contract drift is undocumented in description, plan, guide, STATUS.md, or usage.tsx. A consumer reading the plan and looking for `Alt+Enter` won't find it; a future reviewer comparing plan to code surfaces the same gap.
- **Suggested fix:** Pick one. (a) Wire `Alt+Enter` in [`hooks/use-keyboard-actions.ts`](../../../../src/registry/components/layout/workspace/hooks/use-keyboard-actions.ts) as a keymap that opens the chevron's `DropdownMenu` programmatically (Radix supports this via controlled-open state). The chevron stays as the visual affordance; `Alt+Enter` becomes a power-user shortcut. (b) Amend plan ¬ß5.3 to lock the chevron-menu pattern (drop `Alt+Enter`); update STATUS.md with a one-line entry explaining the trade-off (visible chevron > invisible shortcut). Either way, propagate to `guide.md`'s "Keyboard alternatives" section.

- **Severity:** üî∏ Medium

- **Severity:** üîπ Low

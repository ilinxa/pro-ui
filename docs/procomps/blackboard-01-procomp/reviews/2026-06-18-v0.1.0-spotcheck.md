### Rotating dimension

- **Chosen:** Design system
- **Why:** the component requested three scoped exceptions to a deliberately strict mandate (handwriting font, dark-in-light board, chalk-red unread number) — the highest-risk surface for this artifact.
- **Verified?** yes
- **Notes:** Handwriting font is scoped to **note text + the unread number only**; all chrome (composer, buttons, pickers, headings) inherits Onest via `font-sans`. Board owns local `--bb-*` surface tokens and is dark in both themes by design (chalkboard identity), not a theme bug. Chalk-red unread `oklch(0.66 0.19 22)` — chroma 0.19 ≤ 0.20 (passes the "no neon" rule). Ink palette tones all chroma ≤ 0.20. **Signal-lime remains the interactive accent** (focus-visible rings use `var(--ring)` / `--bb`-neutral white rings; red is unread-only). Motion: single new-note reveal is light; reduced-motion honored via the `animate-pulse`-only loading indicator. Deviations match what GATE 1 flagged + the user explicitly requested.

## Findings (max 5)

#### F-01 — Consumer-install smoke not yet run

- **Severity:** 🔸 Medium
- **Dimension:** Verification
- **Location:** n/a (post-deploy)
- **Observed:** v0.1.0 ships before the F-cross-11 path-b consumer smoke (`shadcn add @ilinxa/blackboard-01` + consumer-side `tsc`). Established "ship → smoke → patch → re-smoke" cadence. Risk surface here is small: only 2 shadcn primitives (`button`, `textarea` — low Base-UI divergence), no cross-procomp deps, standard `@fontsource`/`date-fns`/`lucide-react`. Defensive measures already in place: no `PopoverAnchor`, no `toggle-group` (pickers are plain `Button`/`button`), ref-as-prop, side-effect font import order fixed for meta-deps.
- **Suggested fix:** After push + Vercel deploy, run the consumer smoke; expect possible `lucide-react@^1` icon-name or Base-UI `textarea`/`button` prop nits. Patch → v0.1.1.

#### F-02 — Visual / interaction walkthrough is user-owned

- **Severity:** 🔹 Low
- **Dimension:** Verification
- **Location:** n/a
- **Observed:** This environment has no X server, so a pixel/interaction walkthrough (post + optimistic reconcile, `@`-picker keyboard nav, scroll-anchored lazy prepend, pin lift, unread clear, custom-image legibility, narrow-container collapse) couldn't be driven. SSR render + all automated gates verified.
- **Suggested fix:** User runs `pnpm dev` → `/components/blackboard-01` and walks the three demo tabs. File any runtime finding as v0.1.x.

#### F-03 — Reconciled-extra prune deferred

- **Severity:** 🔹 Low
- **Dimension:** Component internals
- **Location:** `hooks/use-blackboard-state.ts` (the `merged` memo)
- **Observed:** Optimistic/appended notes absorbed into the controlled `notes` prop linger in the local `extras` buffer (display stays correct — `merged` dedups them out). The prune-via-effect was removed to satisfy `react-hooks/set-state-in-effect`; `appendNote` caps the buffer at 300 so growth is bounded.
- **Suggested fix:** Opportunistic prune of absorbed extras inside the existing `setExtras` callers (no new effect). → v0.1.1.

#### F-04 — Live in-textarea mention highlight deferred to v0.2

- **Severity:** 🔹 Low
- **Dimension:** Public API / scope
- **Location:** `parts/note-composer.tsx`
- **Observed:** v0.1 inserts `@name` as plain text and styles mention tokens on *rendered* notes (`MentionText`); there is no highlight overlay inside the composer textarea while typing. This was a deliberate v0.1 scope call (the mirror-overlay is cosmetic-only and the riskiest piece). The `@`-picker, storage (`note.mentions`), `onMention`, and `@you` emphasis all ship.
- **Suggested fix:** Add the highlighted-textarea mirror overlay in v0.2 (plan §11 documents the approach).

### F-02 — Demo never exercises `onFilteredChange`

- **Severity:** 🔸 Medium
- **Dimension:** 1 — Procomp planning docs
- **Location:** `src/registry/components/forms/filter-stack/demo.tsx` (5 sub-demos)
- **Observed:** All 5 sub-demos (Basic / ModeToggle / SoloButtons / CustomRange / Rich) compute their filtered list via a host-side `useMemo(() => NODE_FIXTURES.filter(...))` that re-runs `categories.every(c => c.predicate(...))` themselves, ignoring the component's built-in `onFilteredChange` convenience callback. Description §6.1 canonical use-case explicitly demonstrates the callback path (`onFilteredChange={(filtered) => setVisibleNodeIds(new Set(filtered.map(n => n.id)))}`); plan Q-P5 locks the change-detection semantics; the implementation in `hooks/use-filtered-items.ts` is non-trivial. None of this is demonstrated to users browsing `/components/filter-stack`.
- **Suggested fix:** Modify `RichDemo` (or add a 6th sub-demo) to consume `onFilteredChange={setFilteredNodes}` and read from the resulting state, dropping the host-side `useMemo`. Demonstrates the callback's pull-from-the-component idiom and validates the Q-P5 referential-stability contract end-to-end.

### F-03 — `useFilteredItems` memoizes even when no `onFilteredChange` callback supplied

- **Severity:** 🔹 Low
- **Dimension:** 3 — Component code
- **Location:** `src/registry/components/forms/filter-stack/hooks/use-filtered-items.ts:13-16`
- **Observed:** `useMemo(() => applyFilters(items, categories, values), [items, categories, values])` runs unconditionally regardless of whether `onFilteredChange` is supplied. The return value is also discarded at the call site (`filter-stack.tsx:87` invokes the hook for side-effects only). Plan §9 explicitly accepts this ("the hook still computes filtered because filter-stack itself doesn't render filtered items — pure convenience"), but for hosts with N=10k+ `items` arrays who don't subscribe to `onFilteredChange`, every render walks the full predicate pipeline for nothing. The early-bail in `apply-filters.ts` softens but doesn't eliminate the cost (active-predicate construction still happens; `.filter` is skipped only when ALL categories are empty).
- **Suggested fix:** Early-return inside `useFilteredItems` when `onFilteredChange === undefined`. Trade-off: makes the hook's return value unreliable for hosts that consume both, but no in-repo consumer reads the return today; the hook is purely side-effect-driven. If hook composition is the bigger concern, alternative: gate the `useMemo` body on the callback and document the behavior.

### F-04 — `clear()` and `clearAll()` handle `__mode` keys inconsistently

- **Severity:** 🔹 Low
- **Dimension:** 3 — Component code
- **Location:** `src/registry/components/forms/filter-stack/filter-stack.tsx:96-126`
- **Observed:** Two divergent behaviors for the same UX intent (user wants to wipe filter state):
  - `clear(categoryId)` — for a checkbox-list with `modeToggle`, **deletes** `next[`${categoryId}__mode`]` (line 110-112).
  - `clearAll()` — explicitly **preserves** every key ending in `__mode` (line 122-124).
  Plan §9 row "clear(id) called via ref" doesn't specify mode-key behavior; plan §12.5 #3 covers category-removal-orphan handling but not clearing. From the user's mental model, mode is UI preference (do I want union or intersection when I'm filtering?) while value is filter selection — clearing the value shouldn't reset the preference. `clearAll()` already takes that posture; `clear()` doesn't.
- **Suggested fix:** Align `clear(id)` to preserve `${id}__mode`. Drop the conditional `delete next[`${categoryId}__mode`]` branch. Document the invariant in plan §9 + 12.5: "clear and clearAll preserve all mode keys; mode is UI preference, not filter state."

### F-05 — `validate-schema.ts` has a 4th dev-check not enumerated in plan §6

- **Severity:** 🔹 Low
- **Dimension:** 1 — Procomp planning docs
- **Location:** `src/registry/components/forms/filter-stack/lib/validate-schema.ts:32-40`
- **Observed:** Plan §6 enumerates 3 dev-only checks: (1) reserved suffix, (2) duplicate id, (3) checkbox-list with empty/missing options. Implementation adds a 4th: warns when `modeToggle === false` AND `defaultMode !== undefined` (line 32-40) — this matches plan §9's edge-case row ("checkbox-list with `modeToggle: false` AND `defaultMode` set | dev-only `console.warn`") but isn't reflected in the validation enumeration in §6. Plan-vs-code drift: implementation is correct; plan §6's check list is the stale half.
- **Suggested fix:** Update plan §6 to enumerate the 4th check, OR add it to §12.5 plan-stage refinements with a sentence pointing to §9 as the source. Both halves of the plan should agree on what `validate-schema.ts` does. Implementation needs no change.

## Findings (3)

### F-01 — `useMagazineFilter` JSDoc claims auto-reset on dataset change; implementation does not auto-reset

- **Severity:** 🔸 Medium
- **Dimension:** 2 — Public API (contract)
- **Location:** `src/registry/components/layout/grid-layout-news-01/hooks/use-magazine-filter.ts:13-14` (JSDoc) vs `:17-82` (implementation)
- **Observed:** The hook's JSDoc states *"Reset semantics: any change to `items`, `filterPredicate`, or `sortComparator` reference resets the page back to 1."* The implementation does NOT do this — `page` is `useState(1)` and only resets when the consumer calls the exposed `reset()` callback. There is no `useEffect` keyed on `items` / `filterPredicate` / `sortComparator` references that calls `setPage(1)`. The slice clamping (`regularItems.slice(0, page * pageSize)`) means the display doesn't crash when items shrink, but `page` retains its prior value; if items grow back later, the consumer is mysteriously on page 5 with items they never paged into. Consumers reading the JSDoc will form an incorrect mental model and skip wiring their own reset.
- **Suggested fix:** Either (a) implement the documented behavior — add a `useEffect` keyed on `[items, filterPredicate, sortComparator]` that calls `setPage(1)` (preferred — matches the doc, removes consumer footgun), or (b) update the JSDoc to say *"Consumer is responsible for calling `reset()` on dataset / predicate / sort changes; the hook does not auto-reset"*. Option (a) is the principle-of-least-surprise fix.

### F-02 — Positional render-fn args (`renderItem(item, slot)`) are a versioning trap

- **Severity:** 🔹 Low
- **Dimension:** 2 — Public API (future-proofness)
- **Location:** `src/registry/components/layout/grid-layout-news-01/types.ts:33` + `src/registry/components/layout/grid-layout-news-01/grid-layout-news-01.tsx:74-76`
- **Observed:** `renderItem: (item: T, slot: GridLayoutItemSlot) => ReactNode` and `renderFeatured: (item: T) => ReactNode` use positional callback args. Per the review-guide's Dim 2 bad-signals: *"`onChange` callback receives positional args (onChange(id, value)) instead of an object (onChange({ id, value })) — positional callbacks are a versioning trap."* If v0.2 wants to pass `index`, `total`, or `key` to renderItem (common needs in news layouts — e.g., "the 4th item gets a different treatment"), the signature must grow with a new positional arg, or break the contract.
- **Suggested fix:** Object-shape the renderer args in v0.2: `renderItem: (props: { item: T; slot: GridLayoutItemSlot; index: number; total: number }) => ReactNode`. v0.1 → v0.2 will be a breaking change; queue in `meta.related` migration notes when v0.2 lands. No v0.1 fix; this finding is a deliberate v0.2 plan input.

### F-03 — `GridLayoutItemSlot` union locked at 2 values; `"small"` / `"wide"` plausible v0.2 additions

- **Severity:** 🔹 Low
- **Dimension:** 2 — Public API (future-proofness)
- **Location:** `src/registry/components/layout/grid-layout-news-01/types.ts:8`
- **Observed:** `type GridLayoutItemSlot = "large" | "medium"` — supports lead + tower-of-medium. Common news layouts also have a "thumbnail / small" tail (e.g., a 4-column "more from this section" footer) and sometimes a "wide" item that spans the main column at body width. Adding new variants is non-breaking *for the type*, but consumers' renderItem switch statements that today rely on exhaustive `switch (slot) { case "large": ... case "medium": ... }` will get a TypeScript narrowing warning. The plan locked at 2 slots for v0.1; this is a v0.2 candidate worth surfacing in guide migration notes.
- **Suggested fix:** No v0.1 fix. Add a "v0.2 candidates" line to the guide noting that the slot union may widen and that consumers should use `default:` in their `switch` statements (or `slot satisfies "large" | "medium"`) to fail loudly when new slots arrive.

## Findings (max 5)

#### F-01 — `history.execute()` not wrapping individual mutations

- **Severity:** 🔸 Medium
- **Dimension:** Public API (handle semantics)
- **Location:** [src/registry/components/media/media-editor-01/media-editor-01.tsx](../../../../src/registry/components/media/media-editor-01/media-editor-01.tsx) (entire mutation surface)
- **Observed:** R4 wired `useHistory` at the root and replaced the undo/redo handle stubs with `history.undo()` / `history.redo()`. Keyboard binding (Ctrl/Cmd+Z) is functional. However, no individual mutation (addTextOverlay, removeTextOverlay, addSticker, removeSticker, addDrawingStroke, setFilter, setAdjustments, setCrop) is wrapped in `history.execute(...)`, so the undo/redo stacks stay empty in practice. `history.canUndo` returns false even after edits.
- **Suggested fix:** Wrap each mutation in `history.execute({ do, undo, label })`. v0.1.5 story-composer-01 had this pattern (e.g., addTextOverlay at story-composer-01.tsx:260). Restore equivalent wrapping inside MediaEditor01. Tag v0.1.1.

#### F-02 — `setMode` workaround in story-composer-01 wrapper via `loadState`

- **Severity:** 🔸 Medium
- **Dimension:** Public API (handle completeness)
- **Location:** [src/registry/components/media/story-composer-01/story-composer-01.tsx:200-205](../../../../src/registry/components/media/story-composer-01/story-composer-01.tsx#L200)
- **Observed:** `MediaEditor01Handle` does NOT expose a public `setMode` method (description §7 enumerates 22 methods explicitly without it). The story-composer wrapper's `toComposerCtx.setMode` workaround round-trips through `loadState({...getState(), mode: m})` to set the mode externally. Works at runtime but is internally inconsistent — every other state field has a setter, mode does not.
- **Suggested fix:** Add `setMode(mode: ComposerMode | null) => void` to `MediaEditor01Handle` in v0.3.0 (handle expansion = minor bump). Story-composer wrapper drops the loadState workaround. Update description §7 + plan to enumerate 23 methods.

#### F-03 — `StoryComposer01Labels` cast in MediaEditor01 root is type-unsafe

- **Severity:** 🔸 Medium
- **Dimension:** Public API (label surface)
- **Location:** [src/registry/components/media/media-editor-01/media-editor-01.tsx:201](../../../../src/registry/components/media/media-editor-01/media-editor-01.tsx#L201) (`as Partial<StoryComposer01Labels>` cast)

The `StoryComposer01Labels` shim is `@internal` per types.ts:308 and explicitly scheduled for removal.

#### F-06 — `initial-source-loader` over-rejects on edit-only path (`enabledModes: []` + `initialSource`)

- **Severity:** 🚫 Blocker (demo regression — News-hero + Edit-only happy-path)
- **Dimension:** Public API (initialSource semantics)
- **Location:** [lib/initial-source-loader.ts:84](../../../../src/registry/components/media/media-editor-01/lib/initial-source-loader.ts#L84), [lib/initial-source-loader.ts:129](../../../../src/registry/components/media/media-editor-01/lib/initial-source-loader.ts#L129)
- **Observed:** The validator unconditionally returns `kind: "mode-not-enabled"` when `!enabledModes.includes(mode)`. Both gates fire for the documented edit-only configuration — `enabledModes: []` + `initialSource: { kind: "url", url, mode: "photo" }` — because an empty `enabledModes` array includes nothing. The News-hero demo (`docs/components/media-editor-01` News-hero tab) and the Edit-only "URL · photo (happy path)" preset both surface the error and never transition to `stage: "edit"`. **4 of 5 demo tabs render the SourceError card instead of the editor canvas.**
- **Root cause:** The validator conflates "consumer disabled the mode the source needs" (real error) with "consumer chose edit-only and disabled ALL capture modes by design" (valid configuration). The semantic of `enabledModes: []` is documented in description §4 as "edit-only path; capture surface absent; initialSource accepted for any mode."
- **Suggested fix:** Skip the `mode-not-enabled` gate when `enabledModes.length === 0` (both call sites). Concretely, replace `if (!enabledModes.includes(mode))` with `if (enabledModes.length > 0 && !enabledModes.includes(mode))`. Single-line change at each gate. Keeps the error behavior intact for the legitimate "consumer enabled photo only, source declares video" case (Edit-only "URL · mode mismatch" preset). Tag **v0.1.1**.
- **Test:** Edit-only tab → "URL · photo (happy path)" preset should land directly in `stage: "edit"` with the image rendered. News-hero tab should render the hero image in the 16:9 canvas with the editorial toolbar.

#### F-07 — Capture-surface wrapper collapses `<EditorCamera>` to 0-width

- **Severity:** 🚫 Blocker (demo regression — Defaults + Dark)
- **Dimension:** Component internals (layout)
- **Location:** [media-editor-01.tsx:786-800](../../../../src/registry/components/media/media-editor-01/media-editor-01.tsx#L786-L800)
- **Observed:** The canvas wrapper `<div className="flex flex-col items-center justify-center gap-2 overflow-hidden ... h-full max-h-[400px] w-full max-w-[600px]">` uses `items-center` (cross-axis center) on its flex-column. `<EditorCamera>`'s wrapper is `relative flex-1 flex flex-col overflow-hidden` — no explicit width. Combined with the parent's `items-center`, the EditorCamera collapses to its content-width (which is 0 since all children — `<video>`, bottom control row, file input — are absolutely positioned). Absolute-positioned `<video class="absolute inset-0 w-full h-full bg-black">` and the gallery/shutter/switch control row anchor to a 0-width box and get clipped by the parent's `overflow-hidden`. **Result: Defaults and Dark tabs show a dashed empty card with no video preview, no shutter, no gallery icon, no switch-camera icon.**
- **Suggested fix:** Two minimal options — (a) change parent to `items-stretch` (default) so EditorCamera fills width, or (b) add `w-full h-full` to `<EditorCamera>`'s wrapper at [parts/editor-camera.tsx:175](../../../../src/registry/components/media/media-editor-01/parts/editor-camera.tsx#L175). Option (a) is cleaner — `items-center` only matters for the fallback `<p>` text block (lines 904-918), which can wear its own `text-center` styling. Recommend (a). Tag **v0.1.1**.
- **Test:** Defaults + Dark tabs should show a black `<video>` viewport, large white shutter button at bottom-center, gallery icon at bottom-left, switch-camera icon at bottom-right (when ≥2 devices). With camera permission granted, the live preview replaces the black fallback.

#### F-08 — `<DialogContent>` missing `<DialogDescription>` / `aria-describedby` (Chat demo)

- **Severity:** ⚠️ High (a11y violation + dev-console noise)
- **Dimension:** Accessibility / Robustness (F-cross-13 Radix sub-trap)
- **Location:** [media-editor-01.tsx:1053](../../../../src/registry/components/media/media-editor-01/media-editor-01.tsx#L1053) (DialogContent for `presentation: "dialog"` path)
- **Observed:** Console emits `Warning: Missing Description or aria-describedby={undefined} for {DialogContent}` whenever Chat demo opens. Radix Dialog's spec requires either a `<DialogDescription>` child OR an explicit `aria-describedby={undefined}` opt-out to silence the rule. The component has `<DialogTitle>` but neither of the description affordances.
- **Suggested fix:** Two options — (a) add a `<DialogDescription>` with `composerDescription` label string from `mergedLabels` (semantically correct — describes the editor's purpose to screen readers), or (b) `aria-describedby={undefined}` opt-out on the DialogContent (silences without adding semantic content). Recommend (a) — `composerDescription` already exists at [media-editor-01.tsx:78](../../../../src/registry/components/media/media-editor-01/media-editor-01.tsx#L78) as a label and matches the v0.1.5 surface. Tag **v0.1.1**.
- **Test:** Chat demo open → no console warning, screen reader announces the description after the title.

#### F-13 — Crop pipeline never wired between orchestrator and canvas — overlay never renders

- **Severity:** 🚫 Blocker (feature completely non-functional)
- **Dimension:** Component internals (tool lifecycle + pipeline wiring)
- **Location:** [media-editor-01.tsx:862-882](../../../../src/registry/components/media/media-editor-01/media-editor-01.tsx#L862-L882) (EditorCanvas mount) + [parts/editor-canvas.tsx:453](../../../../src/registry/components/media/media-editor-01/parts/editor-canvas.tsx#L453) (gate) + [hooks/use-media-editor-state.ts:266-268](../../../../src/registry/components/media/media-editor-01/hooks/use-media-editor-state.ts#L266-L268) (setActiveTool)
- **Observed:** Two cascading gaps. (1) Clicking `crop` in the toolbar fires `editor.setActiveTool("crop")`, which sets `activeTool` but does NOT initialize `state.crop`. EditorCanvas gates the overlay on `cropActive && cropRect && stageSize.width > 0` — with `cropRect = null`, no overlay. (2) Even if cropRect were initialized, the orchestrator mounts `<EditorCanvas>` **without** the `onCropChange` or `cropAspectRatio` props — so the canvas has no callback to update state and no aspect lock to respect. The crop UI was wired into the toolbar but never plumbed into the canvas.
- **Suggested fix:** Two-part — (a) pass `onCropChange={editor.setCrop}` and `cropAspectRatio={aspectAsRatio(editor.state.cropAspect)}` at the EditorCanvas mount site; (b) inside EditorCanvas, when `cropActive && !cropRect && stageSize.width > 0`, auto-fire `onCropChange` with a sensible default: centered 80%-of-stage rectangle, snapped to `cropAspectRatio` if set. Both parts required — (a) opens the pipeline, (b) seeds it. Tag **v0.1.1**.
- **Test:** News-hero tab → click `crop` → a draggable crop rectangle locked to 16:9 should appear over the image, centered, sized to ~80% of the canvas. Dragging corners resizes (aspect-locked). Dragging center repositions (clamped to stage bounds).

#### F-11 — `takePhoto` captures full native frame; preview's `object-cover` crop is not applied

- **Severity:** 🚫 Blocker (capture/preview mismatch — what user sees ≠ what user gets)
- **Dimension:** Component internals (capture pipeline)
- **Location:** [hooks/use-media-capture.ts:370-388](../../../../src/registry/components/media/media-editor-01/hooks/use-media-capture.ts#L370-L388)
- **Observed:** The `<video>` element uses `object-cover` (editor-camera.tsx:183) so its preview fills the dialog viewport by center-cropping to the dialog aspect (e.g. 9:16). But `takePhoto` writes the full native `v.videoWidth × v.videoHeight` (e.g. 1280×720 landscape from OBS) into the canvas and returns that. Result: the captured blob has a different aspect from the framing the user saw — landscape camera in 9:16 dialog produces a landscape photo letterboxed inside a 9:16 edit canvas, with parts of the captured image that were never visible during framing.
- **Suggested fix:** Add an optional `aspectRatio?: number | null` opt to `takePhoto`. When provided, after `drawImage` at native size, compute the center-crop region matching that aspect (`videoAspect > targetAspect` → crop horizontal; else → crop vertical), draw to a second canvas at the cropped dimensions, and `toBlob` from that. EditorCamera receives a `captureAspectRatio` prop computed from the orchestrator's `aspect` prop (`9:16` → `9/16`, `free` → `null`). Backwards-compatible (opt is optional; existing consumers see no change). Tag **v0.1.1**.
- **Test:** Open Chat dialog (9:16) with OBS Virtual Camera (16:9 native) → tap shutter → captured photo should be 9:16 portrait with the same content the framing showed, not a 16:9 landscape with letterbox.

#### F-12 — Bottom control row cramped at narrow dialog widths (DEFERRED to v0.1.2)

- **Severity:** 🔹 Low (UX polish — controls remain functional at all widths)
- **Dimension:** Component internals (responsive layout)
- **Location:** [parts/editor-camera.tsx:217-220](../../../../src/registry/components/media/media-editor-01/parts/editor-camera.tsx#L217-L220)
- **Observed:** The bottom control row uses `px-6 flex justify-between` with fixed-size shutter (`size-[68px]`) + gallery + switch (`size-11` each). Total content ≈ 156px + 48px padding = 204px minimum. On very narrow desktop viewports (< 280px wide dialog) the controls cramp but stay tappable.
- **Initial attempted fix REVERTED:** v0.1.1 first attempt added `md:min-w-[320px]` (portrait) / `md:min-h-[280px]` (landscape) to floor the dialog dimensions. This **broke F-10's aspect-ratio invariant** at small viewports — when computed-width from height fell below 320px, the min-w override forced a non-9:16 box (e.g. 320×510 ≈ 5:8). Aspect-ratio integrity takes precedence over button slack.
- **Correct fix (deferred):** Scale the shutter button down via container-query when the dialog is < 320px wide (e.g. `@container (max-width: 320px) { .shutter { size: 56px } }`). Or trim the gallery/switch icons to `size-9` (36px) at narrow widths. Either approach preserves aspect-ratio. Tag **v0.1.2**.
- **Test (at v0.1.2):** Resize viewport down to ~250px dialog width → controls scale down proportionally, aspect-ratio remains 9:16.

#### F-10 — Dialog presentation uses fixed-px sizes instead of viewport-relative aspect-ratio

- **Severity:** 🚫 Blocker (demo regression — dialog clips toolbar on short viewports + stays small on tall ones)
- **Dimension:** Component internals (sizing) / Robustness
- **Location:** [lib/dialog-size-for-aspect.ts:17-23](../../../../src/registry/components/media/media-editor-01/lib/dialog-size-for-aspect.ts#L17-L23) + [media-editor-01.tsx:1070](../../../../src/registry/components/media/media-editor-01/media-editor-01.tsx#L1070)
- **Observed:** `dialogSizeForAspect("9:16")` returns a hardcoded `{ width: 400, height: 711 }`. The dialog renders at exactly 400×711px on every desktop viewport, only clamped by `md:max-h-[90dvh] md:max-w-[90vw]`. Two failure modes: (a) **short viewports** (< 790px tall) clip the bottom toolbar because the cap doesn't kick in until 90dvh < 711; (b) **tall viewports** waste vertical space — the dialog sits at 711px when the screen could afford 900+. The same issue applies to every aspect (`1:1` 600×600, `16:9` 800×450, `4:5` 500×625, `free` 800×600).
- **Root cause:** The lookup table approach predates viewport-relative + `aspect-ratio` CSS support. Should derive the dialog as a viewport-relative box constrained by aspect-ratio rather than a fixed-px lookup. The driving dimension should be the one most likely to constrain (height for portrait aspects, width for landscape).
- **Suggested fix:** Refactor `dialogSizeForAspect` → `dialogDimsForAspect` returning `{ aspectRatio: string; orientation: "portrait" | "landscape" }`. In media-editor-01.tsx, switch the dialog className from fixed CSS vars to viewport-relative + aspect-ratio: portrait → `md:h-[85dvh] md:w-auto md:aspect-[9/16] md:max-w-[90vw]`; landscape → `md:w-[85vw] md:h-auto md:aspect-[16/9] md:max-h-[85dvh]`. Aspect-ratio CSS computes the unspecified dimension; max-* clamps the secondary axis. Tag **v0.1.1**.
- **Test:** Chat dialog on tall viewport (1080p+) → dialog scales to ~85dvh tall with 9:16 width derived. Chat dialog on short viewport (720p) → dialog shrinks but maintains 9:16, toolbar stays visible.

#### F-09 — Dialog presentation off-center on desktop (shadcn base positioning conflict)

- **Severity:** 🚫 Blocker (demo regression — Chat dialog renders at top-left, half off-screen)
- **Dimension:** Component internals (positioning) / Robustness
- **Location:** [media-editor-01.tsx:1057-1066](../../../../src/registry/components/media/media-editor-01/media-editor-01.tsx#L1057-L1066) vs [src/components/ui/dialog.tsx:64](../../../../src/components/ui/dialog.tsx#L64)
- **Observed:** shadcn's `DialogContent` base applies `fixed top-1/2 left-1/2 ... -translate-x-1/2 -translate-y-1/2` to center on the viewport. The media-editor-01 override adds `fixed inset-0` for mobile fullscreen but never cancels the base's `top-1/2 left-1/2` nor the `-translate-x-*` / `-translate-y-*`. On desktop, the resolved styles set the element to top:0/left:0 (inset wins on those properties in the emitted cascade), and the surviving `-translate-x-1/2 -translate-y-1/2` then shifts the box by half its own size up + left → pushes it off-screen to the top-left. Mobile path happens to look acceptable because `w-screen h-[100dvh]` makes the translate shift align with viewport center coincidentally, but the math is still semantically wrong — any change to the mobile size would break it too.
- **Root cause:** Pre-existing v0.1.5 bug in story-composer-01's dialog presentation, inherited verbatim by the C7 wrapping in MediaEditor01. Masked until now because the user had never opened the Chat dialog tab post-deploy. The shadcn base's positioning was never explicitly defeated.
- **Suggested fix:** Explicitly reset the centering at mobile and re-apply at md+. Concretely, replace the current className block with:
  ```
  // Reset shadcn base centering for mobile (full-screen edge-to-edge):
  "!fixed !inset-0 !top-0 !left-0 !translate-x-0 !translate-y-0",
  "!max-w-none gap-0 overflow-hidden p-0",
  "h-[100dvh] w-screen !rounded-none",
  // Desktop: restore centering with derived size from aspect:
  "md:!top-1/2 md:!left-1/2 md:!-translate-x-1/2 md:!-translate-y-1/2",
  "md:!inset-auto",
  "md:h-[var(--media-editor-dialog-h)] md:w-[var(--media-editor-dialog-w)] md:!rounded-2xl",
  "md:max-h-[90dvh] md:max-w-[90vw]",
  ```
  The `md:!inset-auto` is critical — it cancels the mobile `!inset-0` at the md breakpoint so top/left:50% can position the element correctly. Tag **v0.1.1**.
- **Test:** Chat demo → open dialog → on desktop, dialog centers on viewport with derived 9:16 size (~360×640px); on mobile-emulator (<md), dialog fills viewport edge-to-edge.

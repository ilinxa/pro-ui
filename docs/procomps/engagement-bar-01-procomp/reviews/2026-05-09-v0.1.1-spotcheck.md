## Findings (2)

### F-01 — Sub-folder named `utils/` instead of `lib/`; drift from sealed-folder convention

- **Severity:** 🔸 Medium
- **Dimension:** 3 — Component code (and Dim 14 — Cross-component coherence)
- **Location:** `src/registry/components/data/engagement-bar-01/utils/format-count.ts` (only file in the folder) + `engagement-bar-01.tsx:19` (import: `from "./utils/format-count"`)
- **Observed:** Per Dim 3 review-guide's two reference patterns: *"sealed: `<slug>.tsx`, `parts/`, `hooks/`, `lib/` (optional)..."* — the canonical sub-folder names are `parts/`, `hooks/`, `lib/`. engagement-bar-01 alone uses `utils/` as the helper-modules folder. Other components reviewed across sessions 8-10 with similar utility-style code use `lib/`:
  - `post-card-01/lib/defaults.tsx`
  - `comment-thread-01/lib/format-time.ts`
  - `filter-stack/lib/apply-filters.ts` (+ `default-is-empty.ts` + `validate-schema.ts`)
  - `filter-bar-01/lib/format-default.ts`
  
  engagement-bar-01's `utils/format-count.ts` would be `lib/format-count.ts` under the convention. Per Dim 14 review-guide bad signal: *"One renderer-registry component using `kind` as the discriminator while others use `__type` and `rendererId`"* — same shape: one component using `utils/` while others use `lib/`. The .ts file itself is purposeful (one responsibility, descriptive name); the folder name is the drift.
- **Suggested fix:** Rename `src/registry/components/data/engagement-bar-01/utils/` → `lib/`. Update the import path in `engagement-bar-01.tsx:19` from `./utils/format-count` → `./lib/format-count`. Update `registry.json` path + target. Run `pnpm registry:build` + smoke-test. Mechanical change, ~5 line touches. Locks the convention before more components copy the `utils/` shape.

### F-02 — `.css` files use `type: "registry:file"` (not `registry:component`); CLAUDE.md locked-target-convention statement is too absolute

- **Severity:** 🔹 Low
- **Dimension:** 1 — Procomp planning docs / CLAUDE.md docs accuracy
- **Location:** `registry.json` (engagement-bar-01 base item) + `.claude/CLAUDE.md` (Registry conventions section: *"Locked target convention: every file `type: 'registry:component'`, `target: 'components/<slug>/<sub-path>'`."*)
- **Observed:** The `engagement-heart-burst.css` sibling file ships with `type: "registry:file"` rather than `registry:component`. This is a deliberate first-of-kind precedent — `.css` isn't a TypeScript/JSX component, so `registry:file` is the correct shadcn type. `meta.context` flags it explicitly: *"Sibling .css file via shadcn registry:file (first-of-kind precedent in pro-ui)."* But CLAUDE.md's "Registry conventions" section states the locked rule absolutely (*"every file `type: 'registry:component'`"*) without the exception — a future contributor reading CLAUDE.md would fail to anticipate the `.css` carve-out and might mistakenly use `registry:component` for a CSS asset.
- **Suggested fix:** Update CLAUDE.md's "Registry conventions" section to add the .css exception: *"Locked target convention: every TypeScript/TSX file uses `type: 'registry:component'` + `target: 'components/<slug>/<sub-path>'`. Sibling assets (CSS, JSON config, fonts) use `type: 'registry:file'` with the same target shape."* One sentence; closes the convention loop. The implementation is correct; the docs need to catch up.

### Rotating dimension

- **Chosen:** Robustness
- **Why:** the component's correctness rests on three stateful surfaces that tsc can't fully prove — object-URL lifecycle (create/revoke), `@dnd-kit` reorder, and a serially-remounted delegated editor with flatten-on-apply. This is where drift would hide.
- **Verified?** yes (with follow-ups)
- **Notes:** object URLs are tracked in an owned-set and revoked on remove / edit-apply / reset / unmount; remote/consumer URLs are never revoked. Selection is *derived* (no setState-in-effect) so it stays valid under controlled removals. Editor mounts only while editing (one instance, keyed by item id) — never trips media-editor-01's multi-instance guard. One residual race noted in F-05.

## Findings (max 5)

#### F-01 — Tooltip primitive split broke consumer-tsc (F-cross-13) — **RESOLVED in-review**

- **Severity:** ⚠️ High (resolved before ship)
- **Dimension:** Verification / Registry distribution
- **Location:** `parts/main-preview.tsx` (was: `TooltipTrigger asChild`)
- **Observed:** producer uses `radix-ui` tooltip (`asChild`); a `base-nova` consumer installs the **Base UI** tooltip (`render` prop) → `TS2322` on the only tooltip in the component (the video "coming in v0.2" hint). Producer-tsc passed; the local-registry smoke caught it.
- **Fix (applied):** replaced the Tooltip with a native `title` + `aria-label` on the disabled button; dropped the `tooltip` shadcn dep entirely (meta + registry.json). Re-smoke clean. **Lesson reinforced:** prefer native affordances over the Tooltip primitive for trivial hints in distributed components, and always run the local-registry smoke before push.

#### F-02 — Edit-panel runtime (R2 spike) verified only by tsc + static preview

- **Severity:** 🔹 Low
- **Dimension:** Robustness
- **Location:** `parts/edit-panel.tsx`
- **Observed:** the edit-only inline mount (`enabledModes={[]}` + `initialSource`) compiles and the seeded demo renders the read-only preview, but the actual Edit → canvas → Done flatten flow was not exercised headlessly.
- **Suggested fix:** confirm on manual test / post-deploy that the panel lands directly on the edit canvas (no capture chrome) and that Done writes the flattened export back. Owner: author. Target: confirm in v0.1.x.

#### F-03 — Video thumbnails use a live `<video>` element, not a poster frame

- **Severity:** 🔹 Low
- **Dimension:** Robustness / Performance
- **Location:** `parts/rail-thumb.tsx`, `parts/main-preview.tsx`
- **Observed:** each video item renders a `<video preload="metadata">` in the rail; many videos = many media elements.
- **Suggested fix:** extract a poster frame (canvas grab at t≈0.1s) for the rail thumb. Target: v0.2.

#### F-04 — Shared aspect is soft (object-fit), not a hard pixel crop on add

- **Severity:** 🔹 Low
- **Dimension:** Robustness
- **Location:** `parts/main-preview.tsx`, `lib/aspect.ts`
- **Observed:** Instagram hard-crops all items to one aspect; v0.1 only frames with `object-cover` + defaults the edit crop tool to the shared aspect. Un-edited items keep their native pixels.
- **Suggested fix:** optional hard-crop-on-add pass. Target: v0.2.

#### F-05 — `existingCount` read from an effect-updated ref → sub-frame double-add could momentarily exceed `maxItems`

- **Severity:** 🔹 Low
- **Dimension:** Robustness
- **Location:** `media-carousel-editor-01.tsx` (`addFiles` reads `itemsRef.current`, updated in an effect)
- **Observed:** two adds dispatched within the same frame (before the ref-sync effect runs) could both compute the same `existingCount`, allowing a transient overflow past `maxItems`.
- **Suggested fix:** enforce the cap inside `addItems` (truncate to `maxItems`) as a backstop. Target: v0.1.1.

## v0.1.1 — review-driven hardening (2026-06-05, same day)

After the user asked for a deeper multi-perspective review, a **3-agent adversarial
review** (code-correctness / UX+a11y / architecture) was run and surfaced 24 findings
the v0.1.0 spot-check missed. v0.1.1 fixes the blockers + highs + most mediums. This
is a patch bump (no new GATE 3 required); recorded here for the trail.

**Fixed in v0.1.1:**

- ⚠️ **Object-URL leak on controlled value-swap** — orphan-revoke effect in
  `use-carousel-state.ts` revokes owned URLs no longer present in `items` (catches
  drops via `loadValue`/controlled swap, not just `removeItem`).
- ⚠️ **Stale-ref intake race + cap** — `itemsRef` is now written *synchronously*
  inside every mutator; the `maxItems` cap moved into `addItems` (race-safe) and out
  of `filesToItems`; over-cap items' URLs are revoked.
- 🚫 **No loading state during ingest** — `ingesting` state → dropzone `busy` spinner
  ("Adding media…") + add-more tile spinner.
- 🚫 **No in-UI error/max surface** — `role="alert"` region lists rejected files +
  the (previously dead) `maxReached` label.
- ⚠️ **No screen-reader announcements** — polite `aria-live` region for add/remove/
  reorder/max + `DndContext accessibility={{ announcements }}`.
- ⚠️ **Export busy invisible / silent failure** — Done shows spinner + "Saving…" +
  `aria-busy`; `catch` surfaces an inline `role="alert"` error.
- ⚠️ **Re-edit double-overlay** — `initialSource` is passed ONLY when there's no prior
  `editorState` (mirrors media-substrate); else `loadState` supplies image + overlays,
  avoiding compositing the flattened export under re-applied overlays.
- ⚠️ **Disabled video-Edit a11y dead-end** — replaced the disabled (unfocusable)
  button with a static in-reading-order caption.
- ⚠️ **Sub-WCAG hit targets** — handle + remove bumped to 24px (`size-6`).
- 🔸 video poster (`playsInline` + seek-to-0.1s), add-more drag-over, empty-main
  fallback, rail read-only cue, `nextId` collision prefix, dead `"max-items"` kind
  removed, `useCarouselState` exported.
- 🚫 **F-01 (NEW, smoke-caught):** the rewriter resolves a barrel import to the `.tsx`,
  which didn't tail-re-export the carousel's types. Added a **tail type band** to
  `media-carousel-editor-01.tsx`; cross-procomp consumers import from the `.tsx` path.
  Local-registry re-smoke → 0 consumer-tsc errors.

**Verdict carries forward: Pass with follow-ups.** Remaining: multi-blob
upload-at-publish (v0.3, with the content-composer post backend) + minor polish.

## Findings (1)

### F-01 — Time column fixed at `w-20` (80px); overflows for verbose time formats like "9:00 AM - 5:00 PM"

- **Severity:** 🔹 Low
- **Dimension:** 6 — Design system (design-token constraint)
- **Location:** `src/registry/components/data/schedule-list-01/parts/schedule-row.tsx:54` (`<div className="w-20 shrink-0">{timeContent}</div>`)
- **Observed:** The time column is fixed at `w-20` (80px). With:
  - `text-lg font-bold` (≈18px characters, bold)
  - default `timeSeparator: " - "`
  - 4-digit time strings ("09:00")
  
  ...the column fits "09:00" comfortably and "09:00 - 18:00" (13 chars) marginally. But for verbose formats like "9:00 AM - 5:00 PM" (17 chars) or localized formats (Turkish "09:00–18:00 arası"), the time content **wraps or overflows**. Consumers using `renderTime` slot can override with their own width; consumers using the default have a hidden constraint.
- **Suggested fix:** Either (a) switch the column to `w-auto min-w-20` so it grows for verbose formats while having a sensible minimum (preferred — least restrictive); OR (b) document the constraint in the guide ("time column is sized for HH:MM and HH:MM–HH:MM patterns; use `renderTime` for verbose / localized formats"). Option (a) is the v0.1.1 layout fix.

## 1. Executive summary

`rich-card` v0.4.0 is the library's most ambitious component — 56 files implementing a JSON-driven structural-content tree with viewer + inline editor + drag-drop + permissions + multi-select bulk + custom predefined-keys + native search + meta + sync validation hooks (3-layer pipeline) + per-commit undo/redo. The v0.4 phase additions (validators + undo) match the plan exactly: 3 new files (`lib/validators.ts`, `hooks/use-undo.ts`, `parts/undo-toolbar.tsx`), all wired through the imperative handle and the dispatcher pipeline. The component passed smoke install via the harness in 12.3s.

**Headline finding:** the `virtualize` prop in `RichCardProps` and the "Opt-in virtualization for trees >500 nodes" feature listed in `meta.ts` are **not wired up**. The hook `useTreeVirtualizer` (in `hooks/use-virtualizer.ts`) is exported but **never imported** anywhere in the rich-card folder. The `virtualize` prop is accepted but never read by the root component. Consumers passing `virtualize="auto"` for performance receive no benefit. This is a published-API misrepresentation. The lint warning about TanStack Virtual incompatibility — which has been carried as "pre-existing" through prior reviews — is in fact a symptom of this dead code.

Seven findings total — two High, four Medium, one Low. No Blockers. Beta status remains appropriate; some claims need correction before stable.

**Recommended next step:** decide on the virtualization gap — either (a) wire `useTreeVirtualizer` into the render path (the plan documents how), OR (b) remove the prop + feature claim and defer virtualization to a true v0.5/v0.6. v0.4.1 patch should address F-01 (virtualization), F-02 (`@dnd-kit` version drift), F-03 (unused `separator` dep), and F-05 (demo text says "v0.3 demo"). F-04 (cross-cutting usage import-path drift — same pattern as kanban F-04, now affirmed across two components) is a sweep-wide fix opportunity.

## 2. Strengths

Locking these in for future components:

- **Phased planning paid off.** Four iterative plan documents (`-plan.md`, `-plan-v0.2.md`, `-plan-v0.3.md`, `-plan-v0.4.md`) tracked the component's evolution from viewer → inline editor → structural management → safety net. Each phase is independently shippable and the v0.4 plan inherits cleanly. This is the gold-standard procomp planning template — worth promoting in `component-guide.md`.
- **Event-queue ref pattern avoids React 19 reducer-during-dispatch issues.** [`rich-card.tsx:302-318`](../../../src/registry/components/data/rich-card/rich-card.tsx) — actions are dispatched immediately, but granular event firing waits for the next render's `state.version` change. This sidesteps the "calling onChange inside a reducer" anti-pattern. Smart pattern for any reducer-driven component with event callbacks.
- **3-layer validation pipeline is clean.** Built-in validators (v0.2) → per-action host validators (v0.4) → master host validator (v0.4). The first-failure-wins rejection chain in [`rich-card.tsx:261-291`](../../../src/registry/components/data/rich-card/rich-card.tsx) (`gateValidation`) matches plan §3.1 verbatim. Worth replicating for any commit-gating need.
- **Conditional `DndContextWrapper` mount.** [`rich-card.tsx:1064-1106`](../../../src/registry/components/data/rich-card/rich-card.tsx) — DnD sensors only mount when DnD is enabled. Same short-circuit pattern as kanban's readOnly mode. Saves runtime cost for read-only viewers.
- **Comprehensive imperative handle.** 16 methods (`getValue`, `getTree`, `isDirty`, `markClean`, `getSelectedId`, `getSelectedIds`, `setSelection`, `focusCard`, `addCardAt`, `removeCard`, `replaceRoot`, `getEffectivePermissions`, `findNext`, `findPrevious`, `scrollToMatch`, `clearSearch`, `undo`, `redo`, `canUndo`, `canRedo`, `clearHistory`) — actually 21 counting search nav methods. All match the plan and provide a complete imperative surface for hosts that need programmatic control.
- **`useUndo` hook hygiene.** [`hooks/use-undo.ts:42-66`](../../../src/registry/components/data/rich-card/hooks/use-undo.ts) — keyboard handler explicitly skips when focus is in `INPUT`/`TEXTAREA`/`isContentEditable`, preserving native textbox undo. Plus default `Cmd+Z`/`Cmd+Shift+Z`/`Cmd+Y` (Windows convention). Disable via `disableUndoShortcuts: true`.
- **Public API surface size acknowledged.** Plan §4 explicitly notes "by v0.4 the API surface mirrors what mature editor libraries have (Tiptap, Slate). Documented; not a 'wrong API' signal at this scale." 53 props is large but each is justified. The discriminated unions (e.g. `defaultCollapsed: 'all' | 'none' | (level) => boolean`) keep complexity contained.
- **File count matches plan exactly.** Plan §5.3 said v0.4 total = 56 files (53 v0.3 + 3 new). Actual: 56 files. Discipline maintained.
- **Plan v0.4 explicitly lists deferred items.** The "What's NOT in v0.4" list in plan §1 — async validators, will-change/did-change split, structural-diff dirty tracking, markdown adapter — is the right way to ship a focused phase without scope creep.
- **Smoke install passes.** `pnpm dlx shadcn@latest add @ilinxa/rich-card --yes` against production Vercel registry succeeded in 12.3s. All 2 declared shadcn primitives + 4 npm deps resolved.

### 3.2 High (⚠️)

#### F-01 — Virtualization is published in the API but not wired to the render path

- **Severity:** ⚠️ High
- **Dimension:** 2 (Public API) + 4 (Demo + usage)
- **Location:**
  - `RichCardProps.virtualize?: boolean | "auto"` declared in [`types.ts:425`](../../../src/registry/components/data/rich-card/types.ts)
  - `useTreeVirtualizer`, `resolveVirtualize`, `countNodes` defined in [`hooks/use-virtualizer.ts`](../../../src/registry/components/data/rich-card/hooks/use-virtualizer.ts)
  - Feature listed in [`meta.ts:23`](../../../src/registry/components/data/rich-card/meta.ts): *"Opt-in virtualization for trees >500 nodes (auto-threshold); search works regardless"*
  - Demo header text in [`demo.tsx:104-105`](../../../src/registry/components/data/rich-card/demo.tsx): *"Use the search bar to find content in collapsed subtrees, virtualized off-screen cards…"*
- **Observed:** The `virtualize` prop is in the public types, the meta features list, the description (v0.3 in-scope), and the demo's narrative. **But:** grepping the rich-card folder for `useTreeVirtualizer`, `resolveVirtualize`, or `countNodes` returns matches only inside `use-virtualizer.ts` itself — no other file imports these helpers. The root component's render in [`rich-card.tsx:1039-1075`](../../../src/registry/components/data/rich-card/rich-card.tsx) renders `<Card tree={state.tree} ...>` directly with no virtualization branch. The `virtualize` prop is destructured at [`rich-card.tsx:188-197`](../../../src/registry/components/data/rich-card/rich-card.tsx)? Actually it isn't — `virtualize` is in the props type but is not destructured or used at the prop-extraction site, meaning consumers get TypeScript-acceptance with zero runtime effect.
- **Why it matters:** This is a published-API misrepresentation. A consumer who reads the meta features ("Opt-in virtualization for trees >500 nodes") and passes `virtualize="auto"` for a 1000-node tree expects 60fps; they get the same all-render behavior as without it. Worse, the lint warning carried as "pre-existing rich-card warning" in prior reviews (`use-virtualizer.ts:22 — TanStack Virtual incompatible-library`) is in fact a symptom of this dead code: the warning surfaces because `useVirtualizer()` returns memoization-unfriendly functions, but the impact is moot because the hook is never used in render paths.
- **Suggested fix:** Two options — choose one and apply consistently:
  1. **Wire it through.** Import `useTreeVirtualizer` + `resolveVirtualize` + `countNodes` in `rich-card.tsx`. Compute `enabled = resolveVirtualize(virtualize, countNodes(tree))`. Render virtualized item slice when enabled; render full tree otherwise. Plan §6 of `rich-card-procomp-plan-v0.3.md` documents the intended approach. This is the originally-planned outcome.
  2. **Remove the false claim.** Delete the `virtualize` prop from public types, remove the feature line from `meta.ts`, mark the description as "v0.5 candidate (deferred)", and delete `hooks/use-virtualizer.ts` (which kills the lint warning too). Update plan-v0.3.md with a "(deferred)" annotation on the virtualization decision.
  
  Option 1 is the expected outcome from the original plan. Option 2 is the honest path if real-world consumer demand isn't there.

#### F-02 — `meta.ts` declares wrong dep version + non-standard semver notation

- **Severity:** ⚠️ High
- **Dimension:** 5 (Dependencies)
- **Location:** [`meta.ts:54-60`](../../../src/registry/components/data/rich-card/meta.ts)
- **Observed:** The `dependencies.npm` block declares:
  ```
  "@dnd-kit/core": "^6.x",
  "@dnd-kit/sortable": "^11.x",
  "@dnd-kit/utilities": "^3.x",
  "@tanstack/react-virtual": "^3.x",
  ```
  Three issues:
  1. `^N.x` is not standard semver. Standard ranges are `^6.0.0`, `^11.0.0`, etc. `^6.x` is non-canonical and may not resolve cleanly across all package managers.
  2. **`@dnd-kit/sortable: ^11.x` is wrong** — the producer's `package.json` has `@dnd-kit/sortable: ^10.0.0`. There is no v11 yet (v10 is current as of 2026-04). When a consumer installs rich-card via shadcn CLI, the meta deps may be written into their `package.json` — they'd get a request for v11 that doesn't exist, producing an install error or pulling a future-incompatible version.
  3. The kanban-board-01 component (which co-exists in the producer and shares dnd-kit deps) uses precise pins: `^6.3.1`, `^10.0.0`, `^3.2.2`. The library is internally inconsistent.
- **Why it matters:** Cross-component dep version drift is one of the most expensive integration problems for consumers. A consumer installing both kanban-board-01 and rich-card would receive conflicting peer requirements. The wrong major version (`^11`) is also a published falsehood — there is no `@dnd-kit/sortable@11`.
- **Suggested fix:** Replace with the precise versions actually in use (matching kanban-board-01):
  ```
  "@dnd-kit/core": "^6.3.1",
  "@dnd-kit/sortable": "^10.0.0",
  "@dnd-kit/utilities": "^3.2.2",
  "@tanstack/react-virtual": "^3.13.24",
  ```
  Then re-run `pnpm registry:build` and verify `public/r/rich-card.json` reflects the correction. Cross-component coherence: any future component using dnd-kit should follow the same pin convention.

### 3.3 Medium (🔸)

#### F-03 — `meta.ts` declares unused shadcn dep `separator`

- **Severity:** 🔸 Medium
- **Dimension:** 5 (Dependencies)
- **Location:** [`meta.ts:53`](../../../src/registry/components/data/rich-card/meta.ts) declares `dependencies.shadcn: ["popover", "separator"]`
- **Observed:** Grepping all 56 source files for imports from `@/components/ui/`: only `popover` is imported (in 4 files: bulk-toolbar, card-actions, meta-popover, predefined-add-menu). No file imports a `Separator` component or anything else from `@/components/ui/separator`. The `separator` declaration is unused.
- **Why it matters:** A consumer installing rich-card pulls in `separator` from the shadcn registry as a transitive dep, polluting their `components/ui/` folder with an unused primitive. Minor weight cost; bigger discipline issue (the meta should accurately describe dependencies).
- **Suggested fix:** Remove `"separator"` from `meta.ts.dependencies.shadcn`. Re-run `pnpm registry:build`. Confirm `registry.json` registryDependencies list is also updated (mirror per CLAUDE.md convention).

#### F-04 — `usage.tsx` uses producer-side import path `@/registry/components/data/rich-card`

- **Severity:** 🔸 Medium
- **Dimension:** 4 (Demo + usage)
- **Location:** [`usage.tsx:23`](../../../src/registry/components/data/rich-card/usage.tsx)
- **Observed:** The basic-example code block imports from `@/registry/components/data/rich-card`, which is the producer's docs-site path. After consumer install via `pnpm dlx shadcn add @ilinxa/rich-card`, the install lands at `components/rich-card/`, so the consumer's correct import is `@/components/rich-card` (or via the `@ilinxa/rich-card` namespace alias).
- **Why it matters:** Identical pattern to kanban-board-01's F-04. Now confirmed across two components (kanban + rich-card), this is **systemic library-wide drift** — a cross-cutting documentation issue worth a sweep-wide fix. A consumer copy-pasting any usage example into their app immediately hits an unresolved-module error.
- **Suggested fix:** Standardize on the consumer-side path. Replace line 23 with `import { RichCard } from "@/components/rich-card";`. Consider escalating to a cross-cutting finding (F-cross-NN in tracker) and sweeping all 37 components' `usage.tsx` files in a single pass — this is a Medium-finding-times-N where a single sweep is more efficient than per-component fixes.

#### F-05 — Demo header says "v0.3 demo" but component is v0.4

- **Severity:** 🔸 Medium
- **Dimension:** 11 (UI copy / text)
- **Location:** [`demo.tsx:101`](../../../src/registry/components/data/rich-card/demo.tsx)
- **Observed:** The demo's introductory paragraph reads *"v0.3 demo: structural management on top of v0.2's editor."*. The component is v0.4 (per `meta.ts.version`). The v0.4 features (validators + undo) are present in the demo's UI controls (validators toggle, RichCardUndoToolbar), but the descriptive header text wasn't updated when the version bumped.
- **Why it matters:** Stale version reference in marketing copy gives a poor first impression on the docs site. Easy to fix; impact is "doesn't read as fresh".
- **Suggested fix:** Update line 101's text to "v0.4 demo: validation hooks + undo/redo on top of v0.3's structural management. Toggle edit mode to enable inline editing, drag-drop, multi-select bulk, validators (priority must be 1–5; locked cards cannot be removed), and undo/redo via Cmd+Z."

#### F-06 — `use-virtualizer.ts` is dead code (correlated with F-01)

- **Severity:** 🔸 Medium
- **Dimension:** 3 (Component code)
- **Location:** [`hooks/use-virtualizer.ts`](../../../src/registry/components/data/rich-card/hooks/use-virtualizer.ts) — the entire 53-line file
- **Observed:** None of the three exports (`useTreeVirtualizer`, `resolveVirtualize`, `countNodes`) are imported anywhere outside the file itself. The file ships in the registry artifact (consumers receive it). Same root cause as F-01 — virtualization was planned, the helpers were authored, but the wiring to the render path was never completed.
- **Why it matters:** Dead code shipped to consumers. Project rule (review-guide §3): "no half-finished implementations". The lint warning carried as "pre-existing" through prior reviews — `use-virtualizer.ts:22` TanStack Virtual incompatible-library — is the symptom: the React Compiler skips memoization for this hook, but since nothing uses it, the impact is moot. Dead code masking a non-issue is worse than no code masking a real issue.
- **Suggested fix:** Couples with F-01. If the fix path for F-01 is "wire virtualization through" → this file becomes used and the lint warning becomes a real performance concern (TanStack Virtual + React Compiler). If the fix path is "remove the false claim" → delete this file and the lint warning disappears with it. Either way, this dead file shouldn't ship as-is.

### 3.4 Low (🔹)

#### F-07 — Demo's "dirty" indicator uses hardcoded amber palette colors

- **Severity:** 🔹 Low
- **Dimension:** 6 (Design system)

Or introduce a project-level `--warning` semantic token if one isn't already in `globals.css` and use it consistently across all docs-site "dirty" / "warning" / "stale" indicators.

## 5. Verdict

**Verdict:** **Pass with follow-ups**

**Justification:**

The component's core contract is delivered — viewer + inline editor + DnD + permissions + multi-select bulk + custom predefined-keys + native search + meta + 3-layer validation pipeline + per-commit undo/redo. The v0.4 phase plan was executed faithfully (3 new files, exact match). Smoke install passes, types compile, registry builds cleanly. The component is clearly the most ambitious in the library.

The Beta status remains appropriate — beta means "stabilizing" and this review surfaces real gaps that need closing before stable promotion. F-01 (virtualization gap) is the headline because it represents a published-API falsehood: the prop is documented, advertised in the meta features, mentioned in the demo, but does literally nothing at runtime. F-02 (dep version drift) is a real consumer-pain bug. Both warrant a v0.4.1 patch.

The four Medium findings are normal-cadence cleanup. The one Low finding (demo amber colors) is cosmetic, docs-only.

**Recommended path:** v0.4.1 patch addressing F-01 (decide wire-or-delete on virtualization), F-02 (correct dep pins), F-03 (drop unused separator), F-05 (demo text version), F-06 (correlated with F-01 outcome). F-04 (usage import-path) escalates to cross-cutting (now the same pattern across kanban + rich-card; sweeping all 37 components is more efficient). F-07 (demo amber) deferred to next polish pass.

**Stable-promotion gates** that should be met before bumping to `status: "stable"`:
- All Pass-with-follow-ups items closed
- Browser-validated end-to-end (currently deferred — pre-existing project gap)
- External consumer feedback (per description's "stable gated separately" criterion)
- 30-day no-break window after v0.5 markdown-adapter design is fixed (so the JSON model is locked)

## 7. Cross-component observations

Things noticed while reviewing rich-card that aren't rich-card-specific:

- **F-04 is now confirmed as a sweep-wide pattern.** Both kanban-board-01 and rich-card use the producer-side `@/registry/components/...` import in their `usage.tsx` basic example. With 35 more components left to review, this is highly likely systemic. Recommend promoting to a cross-cutting finding (F-cross-06) and tackling all 37 in a single sweep.
- **The phased planning convention is stronger than the procomp doc structure suggests.** Rich-card's 4 plan files (v0.1, v0.2, v0.3, v0.4) tracked the component's evolution cleanly. Worth promoting in `docs/component-guide.md` as the canonical pattern for components that ship in phases (vs. greenfield single-shot components).
- **Cross-component dep-version drift is a real risk.** kanban uses `@dnd-kit/sortable: ^10.0.0`; rich-card declares `^11.x`. A consumer installing both gets conflicting peer requirements. There should be a project-level convention (and ideally a lint check) that forces inter-component dep pins to match the producer's `package.json`.
- **The carried "pre-existing rich-card lint warning" was an unaudited debt.** It's been moved through STATUS.md and prior reviews as "pre-existing, no-op". This review identifies it as a real finding (F-06). The "pre-existing warnings" line in verification tables should be audited as part of every review, not assumed inert.

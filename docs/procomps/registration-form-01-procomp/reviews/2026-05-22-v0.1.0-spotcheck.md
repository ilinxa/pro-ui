### Rotating dimension

- **Chosen:** Dim 2 â€” Public API
- **Why:** Two load-bearing public-API contracts make this dimension the highest-signal one to audit for a v0.1.0 first ship: (a) the **discriminated submit payload** (`RegistrationSubmitPayload` with `stepCompleted: "single" | "step1" | "step2"`), which forces consumers into a `switch` on the discriminant and explicitly rejects optional-field narrowing as a misuse-prevention design call; and (b) the **status mutual-exclusion contract** (`status` + `onStatusChange` controlled escape hatch where passing `status` makes internal state read-only). Both are easy to mis-document or mis-implement. The strength-calculator BYO seam + the honeypot two-key duality (HTML `name="website"` vs RHF key `_honeypot`) are secondary public-API surfaces also covered.
- **Verified?** yes
- **Notes:**
  - `RegistrationSubmitPayload` is a discriminated union of three exhaustive variants (`single` / `step1` / `step2`); type-checked at `src/registry/components/forms/registration-form-01/types.ts:97-117`.
  - The submit handler in `hooks/use-registration-form.ts` correctly emits the right variant: `buildPayload()` at lines 232-253 builds the step-1 envelope from a subset of `RegistrationStep2Values`; step-2 / single hand the full values. `skip()` action calls `onSubmit({ stepCompleted: "step1", values: step1Values, ... })` explicitly.
  - The mutual-exclusion contract is implemented at `hooks/use-registration-form.ts:96-105` â€” `setStatus` is a no-op for internal `setState` when `controlledStatus !== undefined`; `onStatusChange` fires through a stable `useRef` regardless. The "what status SHOULD be" calc still runs in controlled mode so observers see every transition; consumers can reflect it back into their `status` prop or ignore it.
  - The strength-calculator BYO seam (`strengthCalculator?: (password: string) => 0 | 1 | 2 | 3 | 4`) is statically typed at `types.ts:53` and clamped defensively in the hook at `hooks/use-strength-meter.ts:55-61` so a buggy consumer-supplied calc can't crash the meter.
  - The honeypot two-key duality is documented inline in both `lib/honeypot.ts` (the constants + `isHoneypotTripped` helper) and the part itself at `parts/honeypot-field.tsx`, where the docstring spells out why attribute order matters (`{...register()}` BEFORE the `name` override; otherwise the HTML name is silently dropped from `"website"` back to `"_honeypot"`).

## Findings

### F-01 â€” Magic-link strategy keeps the password ZodTypeAny slot via `.optional()` even though the field renders nowhere

- **Severity:** đź”ą Low
- **Dimension:** Public API
- **Location:** `src/registry/components/forms/registration-form-01/lib/build-schema.ts:69-77`
- **Observed:** Under `passwordStrategy: "magic-link"`, `buildSchema` registers `passwordSchema = z.string().optional()` to keep the values-shape stable across both strategies. The runtime cost is nil, but a strict consumer reading the Zod schema for introspection (e.g., to drive a custom field-renderer) would see `password` as a possible-string field even though the renderer never renders it. The static type also keeps `password?: string` on `RegistrationStep1Values` for the same shape-stability reason â€” see the JSDoc comment at `types.ts:71-76`.
- **Suggested fix:** Acceptable as v0.1.0 design. If a real consumer hits the "magic-link shows password in introspected schema" trap, v0.2 can split into per-strategy schemas (`buildSchemaForPassword` / `buildSchemaForMagicLink`). Track as a v0.2 candidate.

### F-02 â€” `useFormStep.goNext` validates `password` even under `magic-link`

- **Severity:** đź”ą Low
- **Dimension:** Public API
- **Location:** `src/registry/components/forms/registration-form-01/hooks/use-form-step.ts:36-44`
- **Observed:** `step1Fields` is hardcoded to `["email", "password", "consentAccepted"]`. Under `magic-link`, `password` is `.optional()` in the schema so the trigger passes harmlessly; under `password`, it works as intended. The hardcoded list works in practice but is one edge case away from a bug â€” e.g., if v0.2 adds a `flow="two-step"` + `magic-link` combo where step-1 is just email + consent, the trigger list should adapt.
- **Suggested fix:** Acceptable for v0.1.0 (single-step + magic-link doesn't expose the case). When v0.2 considers two-step + magic-link, derive the step-1 trigger list from the strategy:
  ```ts
  const step1Fields = useMemo(
    () => passwordStrategy === "magic-link"
      ? ["email", "consentAccepted"]
      : ["email", "password", "consentAccepted"],
    [passwordStrategy],
  );
  ```

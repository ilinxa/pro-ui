#### F-02 — `dispatch({ kind: "patch-content" })` does NOT flip `edited` (cross-component consistency note)

- **Severity:** 🔹 Low
- **Dimension:** Public API
- **Location:** `src/registry/components/data/comment-thread-01/hooks/use-comment-state.ts:111-116` (the `patch-content` action handler; sibling of the `subscribe-delta { kind: "edited" }` path I DID change)
- **Observed:** Intentional asymmetry between two escape hatches that both patch comment content:
  - `subscribe-delta { kind: "edited" }` → flips `edited: true` (correct realtime semantics)
  - `dispatch({ kind: "patch-content" })` → does NOT flip `edited: true` (host might patch content for non-edit reasons, e.g., refetch with corrected payload, optimistic typo fix during composer-as-stage flows, content sanitization)
- This asymmetry is correct but not documented in the dispatch action's JSDoc.
- **Suggested fix:** Add a 2-line JSDoc note on `CommentLocalAction { kind: "patch-content" }` clarifying that this action only patches `content` — host that wants to mark the comment as edited should do so via the subscribe-delta path OR by replacing the comment via `dispatch({ kind: "add" })` with the new payload. Patch-bump-worthy if the backend team or another consumer requests it; otherwise informational.
- **Owner:** Author.
- **Bump target:** v0.2.1 of `comment-thread-01` (docs only; no behavior change).

#### F-03 — `permissions.canModerate` doesn't auto-cascade to existing matrix overrides

- **Severity:** 🔹 Low
- **Dimension:** Public API
- **Location:** `src/registry/components/data/post-card-01/lib/permissions.ts:38-71`
- **Observed:** A moderator (viewer with `canModerate: true`) sees the moderator section AND the viewer-side defaults (Mute / Block / Report). That's correct UX — a moderator IS still a viewer. But there's no opinionated default: hosts that want moderators to ALSO have owner-side actions (Edit / Delete / Pin) must explicitly set `permissions.canEdit: true` etc., or use the `canPerformAction` predicate. The library makes no assumption about "what a moderator can do beyond moderation" — that's a host-policy call.
- **Suggested fix:** Documented as-is in the v0.3.0 usage.tsx section ("Moderation is orthogonal to viewerMode — a moderator is usually a viewer (or sometimes an owner) with extra capability"). No code change. If backend / other hosts request a `viewerMode="moderator"` preset later, that's a v0.4 conversation — adds a third row to `PERMISSION_DEFAULTS_BY_MODE`. Tracked as informational, not blocking.
- **Owner:** Author.
- **Bump target:** v0.4 candidate (conditional on a host requesting it).

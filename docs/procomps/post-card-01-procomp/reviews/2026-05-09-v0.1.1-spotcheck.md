## Findings (2)

### F-01 — Cross-folder imports use `@/registry/...` (producer-side); shadcn rewrites to `@/components/<slug>/<slug>` (filename-direct, bypassing `index.ts`)

- **Severity:** 🔸 Medium
- **Dimension:** 14 — Cross-component coherence (and Dim 9 — Registry distribution)
- **Location:** `src/registry/components/data/post-card-01/post-card-01.tsx:13-20, 39` (cross-folder imports)
- **Observed:** Source imports use producer-side folder paths:
  ```ts
  import { defaultRelativeTime, toDate } from "@/registry/components/data/comment-thread-01";
  import type { EngagementBar01Handle, EngagementDelta } from "@/registry/components/data/engagement-bar-01";
  import { CommentThread01 } from "@/registry/components/data/comment-thread-01";
  ```
  Producer-side resolution goes through each sibling's `index.ts` barrel re-exports. After shadcn-add install in a consumer, the rewrite emits:
  ```ts
  import { defaultRelativeTime, toDate } from "@/components/comment-thread-01/comment-thread-01";  // file-direct, bypasses index.ts
  import type { EngagementBar01Handle, EngagementDelta } from "@/components/engagement-bar-01/engagement-bar-01";
  ```
  The rewrite mechanically appends the slug filename — assumes everything imported is exported from `<slug>.tsx` directly. Today this works because all the imported symbols ARE re-exported from `<slug>.tsx`. **But it's brittle:** if a future refactor moves `defaultRelativeTime` from `comment-thread-01.tsx` into `comment-thread-01/lib/format.ts` (re-exporting via index.ts), the producer-side `tsc` keeps passing (resolves through index.ts), but the consumer-side install ships broken — the file-direct import doesn't find the symbol. Smoke harness only tests `shadcn add` succeeds; it doesn't compile the installed code. So a broken cross-folder import would silently smoke-pass.
- **Suggested fix:** Either (a) document the constraint prominently in the post-card-01 plan + a cross-folder-imports section of `docs/component-guide.md` (every export consumed cross-folder MUST live directly in `<slug>.tsx`, NOT in lib/ subfolders re-exported via index.ts) — minimal change, lock the convention. OR (b) extend the smoke harness to run `pnpm tsc --noEmit` on the consumer after install — catches the brittleness via real type-checking. OR (c) restructure cross-folder imports to use the consumer-side path style (`@/components/<slug>`) so producer + consumer paths align — bigger change but most robust.

### F-02 — Slot density is high (8 slots) but defensible at the host-pattern scale

- **Severity:** 🔹 Low (informational — not a defect)
- **Dimension:** 14 — Cross-component coherence
- **Location:** `src/registry/components/data/post-card-01/post-card-01.tsx:82-89` (props destructure)
- **Observed:** post-card-01 exposes 8 slot escape hatches (`renderHeader`, `renderContent`, `renderMedia`, `renderEngagementBar`, `renderCommentSection`, `engagementActions`, `kebabActions`, `commentActions`). This is the highest slot count in any procomp reviewed so far. Per Dim 14 review-guide, slot density should match the consumer's plausible-override surface. For a 4-variant + 5-internal-dep + 2-channel-realtime social-post composite, 8 slots is defensible — every slot maps to a distinct override seam. The risk isn't that it's too many; it's that future v0.2 work (e.g., `renderRepostFooter`, `renderPollWidget`) could push toward "slot for everything" mush. Worth tracking.
- **Suggested fix:** No v0.1.1 fix. Add a "slot scaling" guidance line to the guide: **new slots in v0.2+ should justify themselves against existing seams** (does the consumer override use case truly need a NEW slot, or can `engagementActions(post, handlers, variant)` be parameterized further?). Defensive against drift.

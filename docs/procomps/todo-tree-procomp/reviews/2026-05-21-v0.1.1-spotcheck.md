### Rotating dimension

- **Chosen:** **Public API** (Dim 3)
- **Why:** todo-tree ships the largest single public surface in the project to date — 26 imperative handle methods + 8 slot props + 16 typed events + 21 reducer actions + a 368-line `types.ts` re-exporting 22+ types via [`index.ts`](../../../src/registry/components/data/todo-tree/index.ts). The plan §11 three-defenses controlled-mode pattern + the F-cross-13 same-day patch loop + the F-cross-12 callback-shape lessons make Public API the highest-leverage dimension to scrutinize before consumers depend on it.
- **Verified?** ✅ yes
- **Notes:**
  - All 16 events use object-args convention per post-F-cross-12 lock. `TodoTreeItemEvent.event` was widened in C8-fix (`bb4e28b`) to `MouseEvent | KeyboardEvent` so the Enter handler can fire `onItemClick` — verified via tsc; no existing single-event call sites broke (the union is structurally compatible with consumers reading only `args.item`).
  - The imperative handle's 26 methods follow consistent naming with `todo-rich-card`'s shape (`getValue` / `setValue` / `getItem` / `updateItem` / `addItem` / `removeItem` / `toggleActive` / `focusItem`) plus tree-specific extensions (`selectItem` / `selectItems` / `bulkToggleActive` / `bulkRemove` / `expandAll` / `collapseAll` / `setCollapsed` / `setSort` / `setFilter` / `setSearch` / `getVisible` / `getSelection`).
  - **`TodoTreeStateValue` from `useTodoTreeState`** is a superset of `TodoTreeHandle` — same 26 methods plus live state values (`items` / `selection` / `focusedItemId` / `collapsedIds` / `sort` / `filter` / `search`) and a `dispatch` escape hatch. Documented as a hot-spot in the guide ("identity changes per render, useMemo your selectors").
  - **Defensive permissions API surface** declared but **not threaded** end-to-end (F-perm below) — `TodoTreeProps.permissions` exists, the DnD predicates wire through via `canDropAsSibling` / `canDropIntoChildren` props on `useTreeDndInternal`, but `useTreeKeyboard` does not consume permissions. Documented in the spotcheck + the guide.
  - F-cross-13 defensive contravariance pattern applied at THREE primitive boundaries pre-emptively (Select.onValueChange × 2 in `parts/todo-tree-sort-dropdown.tsx` + `parts/todo-tree-filter-dropdown.tsx`) — but the Checkbox boundary at `parts/todo-tree-checkbox.tsx:37` was not pre-emptively coerced and tripped on path-b smoke. Patched in v0.1.1 with `typeof next === "boolean" ? next : true` (vs the syntactically narrower `=== "indeterminate"` check that fails type-narrowing under Base UI's `boolean`-only signature). Adds **Checkbox.onCheckedChange** to the F-cross-13 carrier list alongside Select.onValueChange + TooltipProvider.delayDuration.

## Findings

#### F-01 (renamed F-perm) — Permissions matrix declared but not threaded end-to-end through keyboard handlers

- **Severity:** 🔸 Medium
- **Dimension:** Public API + Robustness
- **Location:** [`types.ts:308`](../../../src/registry/components/data/todo-tree/types.ts) (`TodoTreeProps.permissions` declared) · [`todo-tree.tsx`](../../../src/registry/components/data/todo-tree/todo-tree.tsx) (host has 0 references to `permissions`) · [`hooks/use-tree-keyboard.ts`](../../../src/registry/components/data/todo-tree/hooks/use-tree-keyboard.ts) (Space → `toggleActive` + Delete → `removeItem` paths do not consult permission predicates)
- **Observed:** `TodoTreeProps.permissions` is declared on the public types but never read in the host. The DnD path consumes `canDropAsSibling` / `canDropIntoChildren` predicates via `useTreeDndInternal` props (wired at the dispatch site by per-action gating). The action-menu paths in `parts/todo-tree-bulk-action-bar.tsx` and `parts/todo-tree-row.tsx` gate at the affordance level (the button conditionally renders / is disabled). But the keyboard handler's Space → `toggleActive` and Delete/Backspace → `removeItem` paths bypass both `canToggleActive` and `canRemoveItem` predicates — a consumer who wires both a permissions matrix AND keyboard shortcuts cannot expect the predicates to gate the keyboard paths in v0.1.1.
- **Suggested fix:** **v0.1.2 patch** — thread `permissions` from `<TodoTree>` into `useTodoTreeState` (already passes options into `useSelection` / `useTreeEvents`; add a `permissions` slot) and into `useTreeKeyboard` (consume `canToggleActive(id)` before dispatch on Space; consume `canRemoveItem(id)` before dispatch on Delete; fire `onPermissionDenied` with `action: 'toggleActive' | 'remove'` and `reason: 'permission'` on denial). Same gate pattern should also wire the predicates through any future programmatic-handle paths (`handle.toggleActive` / `handle.removeItem` — currently they bypass too, which is a separate consideration: "imperative handle bypasses permissions by design" vs "imperative handle is authoritative and should bypass" — pick one in v0.1.2). Patch-bump, internal-only fix path, no public-API touch needed.

#### F-02 (renamed F-event) — Enter handler firing `onItemClick` — verify no UX regression on existing single-event consumers

- **Severity:** 🔹 Low
- **Dimension:** Public API
- **Location:** [`hooks/use-tree-keyboard.ts`](../../../src/registry/components/data/todo-tree/hooks/use-tree-keyboard.ts) (Enter handler) · [`types.ts`](../../../src/registry/components/data/todo-tree/types.ts) (`TodoTreeItemEvent.event` widened)
- **Observed:** C8-fix (`bb4e28b`) widened `TodoTreeItemEvent.event` from `MouseEvent` to `MouseEvent | KeyboardEvent` so the Enter handler in `useTreeKeyboard` can synthesize a `TodoTreeItemEvent` and fire `onItemClick`. tsc clean post-change; consumers reading only `args.item` are unaffected. **Suggested:** a render walkthrough on the demo's 7 sub-demos to confirm no UX edge case (e.g. consumer attaches handlers to `args.event.metaKey` and now KeyboardEvent's `metaKey` semantics differ from MouseEvent's — KeyboardEvent's `metaKey` is true if Meta was held during keypress, which is the same semantic; should be fine but worth a render pass).
- **Suggested fix:** Walk through the 7 sub-demos in dev mode on the next session start; if no regression, mark closed inline. If a consumer reads `args.event.button` (MouseEvent-only field), document the union split in the guide.

#### F-03 (renamed F-grip-clip) — Grip handle uses `absolute -left-4`, may clip against narrow containers

- **Severity:** 🔹 Low (cosmetic)
- **Dimension:** Component code internals
- **Location:** [`parts/todo-tree-grip.tsx`](../../../src/registry/components/data/todo-tree/parts/todo-tree-grip.tsx) (assumed; positioning via Tailwind)
- **Observed:** The grip is positioned `absolute -left-4` relative to the row, extending 16px to the left of the row content. In containers without left padding, the grip overflows the container's left edge and may be visually clipped or hover the wrong element. Consumer-side `pl-4` on the wrapper mitigates fully. Cosmetic — does not affect functionality.
- **Suggested fix:** **v0.1.x cosmetic** — move the grip into an in-flow column at the left of the row layout (alongside the chevron and status indicator) so it never overflows. Trade-off: adds ~16px to every row's width, which budges all field widths slightly. Worth doing because it makes the tree composable into tighter layouts (side panels, modal sub-views).

#### F-04 (process gate) — F-cross-11 path-b consumer-tsc smoke — CLOSED in v0.1.1

- **Severity:** ⚠️ High → ✅ **Closed**
- **Dimension:** Verification
- **Location:** Surfaced 1 F-cross-13 site at [`parts/todo-tree-checkbox.tsx:37`](../../../src/registry/components/data/todo-tree/parts/todo-tree-checkbox.tsx)
- **Observed (post-push of v0.1.0):** Smoke ran against published Vercel artifact for v0.1.0. Install passed. Consumer-side `pnpm tsc --noEmit` surfaced 1 todo-tree error — `Checkbox.onCheckedChange` callback's `=== "indeterminate"` check failed type-narrowing under Base UI's `boolean`-only signature (Radix passes `boolean | "indeterminate"`; the project's producer-side `src/components/ui/checkbox.tsx` is still Radix-based per the F-cross-13 path (b) hygiene gap).
- **Fix shipped (v0.1.1, commit `5aba0cd`):** Replaced `=== "indeterminate"` check with `typeof next === "boolean" ? next : true` (gates through both backends without an `unknown` cast). Coercion semantics unchanged: indeterminate (Radix-only state) commits as `true` because that's the only state the row transitions to when committing a partial check. Re-run of smoke against v0.1.1 artifact: ✅ 0 todo-tree errors. **F-04 closed.**
- **Cross-cutting note:** F-cross-13 carrier list grew by 1. Memory updated at [`project_shadcn_primitive_radix_baseui_divergence.md`](../../../../../.claude/projects/.../memory/project_shadcn_primitive_radix_baseui_divergence.md) — Checkbox.onCheckedChange added alongside Select.onValueChange + TooltipProvider.delayDuration. Future v0.1.0 first-ship plans for any procomp using `<Checkbox>` should pre-emptively use the typeof-guard pattern.

#### F-05 (renamed F-html5-edge) — External HTML5 drops always land as last child of target row (no edge-zone)

- **Severity:** 🔹 Low (doc)
- **Dimension:** Robustness + Doc
- **Location:** [`hooks/use-tree-dnd-html5.ts`](../../../src/registry/components/data/todo-tree/hooks/use-tree-dnd-html5.ts) + guide.md "Dual DnD" section
- **Observed:** Internal `@dnd-kit` drops respect the top 25% / middle 50% / bottom 25% edge-zone partition (with 8px cap on the top/bottom slivers). External HTML5 drops (from another procomp via the shared MIME) always land as **last child** of the row under the pointer — no edge-zone partitioning for the HTML5 transport. Documented in the C11 guide.md "Dual DnD" section + "Gotchas" but a consumer who expects symmetric edge-zone semantics across both transports will be surprised.
- **Suggested fix:** **v0.2 candidate** — extend the HTML5 hit-test to compute edge-zone from `dragover` event coordinates. Tradeoff: HTML5 drops are pointer-only (no keyboard equivalent), and the edge-zone math needs the row's bounding rect at fire-time. Manageable but not free. For v0.1.1, documentation is the close-out.

## Standing design choices (documented in handoff; not findings)

- **D-reducer-state** — `State.lastChangeReason` written by reducer wrapper; raw-dispatch consumers route correct reason to onChange.
- **D-handle-row-click** — `TodoTreeStateValue.handleRowClick` added (additive on locked type).
- **D-event-union** — `TodoTreeItemEvent.event` widened to `MouseEvent | KeyboardEvent` (see F-02).
- **D-permission-mapping** — Tree-only action codes `dropAsSibling` → `drag` rule key; `dropIntoChildren` → `addChildren` rule key (`lib/permissions.ts`).
- **D-echo-guard** — `treesMatchStructurally` only checks (`id`, `name`, `status`, `active`, `children`) per plan §11. External value-only mutations to other fields silently dropped as echoes — plan-locked tradeoff. Documented in guide.md "Controlled vs uncontrolled".
- **D-grip-absolute** — See F-03 above.

## Findings

#### F-01 — Imperative handle remains ungated (documented design)

- **Severity:** 🔹 Low
- **Dimension:** Public API
- **Location:** `hooks/use-todo-tree-state.ts`
- **Observed:** TT5 — `handle.removeItem` etc. still bypass the matrix by design; the guide note is now prominent (UI affordances are the gates).
- **Suggested fix:** Optional gated `try*` variants (`tryRemoveItem`) in a later minor.

#### F-02 — Inline rename still deferred

- **Severity:** 🔹 Low
- **Dimension:** Public API
- **Location:** guide "Open follow-ups"
- **Observed:** TT8 — click stays read-only; editing is delegated to consumer dialogs / `<TodoTreeWithEditor>`. Clearly marked deferred.
- **Suggested fix:** Ship inline-rename mode (Enter/Esc) in a later minor.

#### F-03 — `dropAsSibling` evaluated against the target

- **Severity:** 🔹 Low
- **Dimension:** Composition integrity
- **Location:** `todo-tree.tsx` `resolveDropPermission` + `hooks/use-tree-dnd-internal.ts`
- **Observed:** The existing hook's drop predicates are target-keyed, so `dropAsSibling` (→ `drag` rule) is evaluated on the drop target, not the dragged source. Source draggability is separately gated by the per-row `canDrag`, so a non-draggable source can't start a drag anyway — the combined behavior is correct, but the target-keyed mapping is a pre-existing design choice worth noting.
- **Suggested fix:** None required; revisit the predicate signature if a consumer needs source-keyed sibling rules.

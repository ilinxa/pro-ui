## What changed in v0.1.2

Same-day-after-v0.1.1 batch. Public-API additive, no breakage.

1. **F-perm closure** — `useTreeKeyboard` consumes `permissions` + `readOnly`; Space (`toggleActive`) and Delete/Backspace (`remove`) consult the matrix + `item.locked`, fire `onPermissionDenied({ action, itemId, reason })` on denial with one of `denied-by-rule` / `denied-by-lock` / `denied-by-readOnly`. `lib/permissions.ts` already existed (single-action `evalPermission`); no resolver rewrite needed.
2. **`+ New` toolbar button** + `createItem?: () => TodoItem` prop + built-in fallback factory using `statusOptions[0]`. Hidden behind `canAdd = editable && !readOnly && factory != null`.
3. **`editable?: boolean` prop** (default `false`). Gates the `+ New` button AND the bulk action bar's Delete button (Activate / Deactivate / Edit keep their existing gates).
4. **`onCreateRequest?: (item: TodoItem) => void` prop** — deferred-commit hook. When provided, the toolbar fires it instead of mutating; consumer commits via `handle.addItem` on Submit.
5. **`<TodoTreeWithEditor>` deferred-commit flow** — pending-item state, dialog opens immediately on `+ New`, Submit commits + focuses the new row, Cancel/Escape/overlay discards.
6. **Dialog UX** — `max-h-[calc(100dvh-2rem)]` + inner overflow-y-auto + `max-w-252` (1.5× the previous `max-w-2xl`); `aria-describedby={undefined}` opt-out; blur-active-element guard before `setEditTarget` to prevent the Radix focus-blur aria-hidden race.
7. **F-grip-clip closure** — grip moved from `absolute -left-4` into the row's flex flow (leading 16px column). Drag is now reachable inside `overflow:hidden` / `overflow:auto` containers.

### Rotating dimension

- **Chosen:** Public API
- **Why:** v0.1.2 is dominated by three new public props (`editable`, `createItem`, `onCreateRequest`), a behavioural change to the toolbar (`+ New` button), a behavioural change to the bulk action bar (Delete gated on `editable`), a behavioural change to the keyboard handlers (permissions consulted), AND a new wrapper-level deferred-commit semantic. The single highest-risk dim is whether these compose correctly with the existing public surface (controlled-mode, slot props, imperative handle, permission predicates).
- **Verified?** yes (with two follow-ups below — see F-handle-doc + F-create-perm).
- **Notes:**
  - `editable` defaults to `false` — bare-tree demos (sections 1, 3, 4, 5, 6, 7) lose `+ New` + bulk Delete. Demo section 2 (With editor) gets them via wrapper default. This is intentional but worth a one-liner in the guide pointing to the change for any consumer reading old demos.
  - `onCreateRequest` is opt-in. When omitted, the toolbar uses the original immediate-add path (`state.addItem` + focus). Backwards-compatible for any consumer already using `+ New` from a hypothetical earlier patch (none such consumer exists yet — v0.1.2 IS the first ship of `+ New`).
  - `<TodoTreeWithEditor>` passes `editable` BEFORE `{...rest}` so consumer can override via spread. Verified by reading the JSX order.
  - Imperative handle methods (`addItem` / `removeItem` / `toggleActive`) do NOT gate on permissions. Documented in guide § Permissions — explicit decision per the v0.1.1 spotcheck's "Decide imperative-handle gating posture in same patch" follow-up.
  - F-grip-clip closure has a layout-shape side effect: rows reserve a 16px gutter for the grip; in `statusIndicator="strip"` mode the strip now sits 16px in from the row's outer edge rather than flush. Noted in last session's exchange but not yet pulled into the guide.

## Findings (max 5)

#### F-01 (F-handle-doc) — Imperative handle gating posture documented in guide only; not surfaced in TodoTreeHandle's JSDoc

- **Severity:** 🔹 Low
- **Dimension:** Public API
- **Location:** [`src/registry/components/data/todo-tree/types.ts`](../../../../src/registry/components/data/todo-tree/types.ts) (`TodoTreeHandle` interface, lines ~186–224)
- **Observed:** The guide § Permissions now states "Imperative handle methods do NOT gate." That's the chosen posture per F-perm follow-up. But the interface itself has no JSDoc note — a consumer reading the type definition (e.g., in their IDE) won't see this until they read the guide.
- **Suggested fix:** Add a 2-line JSDoc note on `TodoTreeHandle` (or on each of `removeItem` / `toggleActive` / `removeItems` / `toggleActiveBulk`) saying "Bypasses permission checks; consumers driving the tree imperatively enforce their own policy."
- **Target:** v0.1.3 (patch — documentation only).

#### F-02 (F-create-perm) — `+ New` toolbar button doesn't consult `permissions.default.addChildren` for top-level add

- **Severity:** 🔸 Medium
- **Dimension:** Public API + Robustness
- **Location:** [`src/registry/components/data/todo-tree/parts/todo-tree-toolbar.tsx`](../../../../src/registry/components/data/todo-tree/parts/todo-tree-toolbar.tsx) (`canAdd` derivation)
- **Observed:** `canAdd = editable && !readOnly && factory != null`. Permissions matrix is not consulted. A consumer setting `permissions={{ default: { addChildren: false } }}` to lock the tree against new top-level rows would still see the `+ New` button. (Semantically `addChildren` is the closest existing rule key; the tree treats top-level adds as "adding a child of the implicit root".)
- **Suggested fix:** Either (a) add an explicit `permissions.canAddRoot?: boolean` derived from `permissions.default.addChildren` at top-level, OR (b) document that top-level add is governed only by `editable` + `readOnly`, NOT by `permissions.addChildren` — the matrix is per-node and there's no implicit root node. Option (b) is the simpler closure and matches how `<TodoTree>` already treats roots as a parentless array, not a virtual node.
- **Target:** v0.1.3 (doc note in guide § Adding new items) OR v0.2.0 (introduce a `canAddRoot` predicate / rule). Author's preference: v0.1.3 doc closure; the predicate-level orchestration belongs to consumers wiring their own toolbar already.

#### F-03 (F-strip-flush) — `statusIndicator="strip"` no longer flush with row's left edge after F-grip-clip fix

- **Severity:** 🔹 Low (cosmetic)
- **Dimension:** Component code internals
- **Location:** [`src/registry/components/data/todo-tree/parts/todo-tree-row.tsx`](../../../../src/registry/components/data/todo-tree/parts/todo-tree-row.tsx) + [`parts/todo-tree-row-content.tsx`](../../../../src/registry/components/data/todo-tree/parts/todo-tree-row-content.tsx) (strip render at row body's `left-0`, body is now 16px inset from row outer edge after the grip column was lifted into row flow)
- **Observed:** Strip indicator now appears 16px in from the row's outer-left edge instead of flush. Visually fine in dev; only matters for consumers who want a hard-edge strip flush against the list container.
- **Suggested fix:** Lift the strip indicator from `todo-tree-row-content.tsx` into `todo-tree-row.tsx` so it positions relative to the OUTER row (not the body), or render the strip as a `::before` pseudo-element on the row. Acceptable to ship as-is — the visual softening is arguably more legible.
- **Target:** v0.1.x cosmetic when next touching strip rendering.

#### F-04 (F-create-default-name) — Built-in factory's `name: "Untitled"` lands a non-empty default that bypasses an "empty → placeholder" UX

- **Severity:** 🔹 Low
- **Dimension:** Public API + Robustness
- **Location:** [`src/registry/components/data/todo-tree/parts/todo-tree-toolbar.tsx`](../../../../src/registry/components/data/todo-tree/parts/todo-tree-toolbar.tsx) (built-in factory)
- **Observed:** The fallback factory emits `name: "Untitled"`. In the deferred-commit flow this lands in the dialog with "Untitled" already filled in — the user has to clear it before typing. Some consumers prefer `name: ""` so the inline rename / dialog text field starts empty.
- **Suggested fix:** Either flip the default to `name: ""` (will need confirmation that downstream readers tolerate empty `name`; the TodoItem schema declares `name: string` non-optional), or expose a `defaultName?: string` toolbar prop. Or document that consumers wanting empty-default should provide their own `createItem`. Author's preference: leave fallback as "Untitled" + document the override pattern. Closes by docs in v0.1.3.
- **Target:** v0.1.3 (doc closure).

### Rotating dimension

- **Chosen:** Dim 3 ‚Äî Component code (sealed folder)
- **Why:** video-player-01 wraps a notoriously defect-prone surface (HTML5 `<video>` element + custom controls + cross-frame autoplay policies + event-listener hygiene). The `useReducer` state machine + rAF-throttled time updates + stable callback refs are exactly the patterns that *should* be there ‚Äî verifying they're implemented correctly is the load-bearing risk for a v0.1 video primitive.
- **Verified?** yes (with 3 small findings)
- **Notes:**
  - **Folder shape:** host pattern (`<slug>.tsx` + 2 hooks + 2 parts + types/index/dummy/meta) ‚Äî appropriate for the substrate complexity. Plan-vs-code 1:1.
  - **State machine** (`hooks/use-video-state.ts`): `useReducer` with no-op early-bails on every action (`state.isPlaying ? state : ...`) ‚Äî prevents reducer-driven re-renders when the action would be a no-op. ‚úì idiomatic and efficient.
  - **rAF-throttled `onTimeUpdate`** (line 144-153): single-pending-rAF guard via `rafRef.current` prevents handler flooding from `<video>`'s native timeupdate event. rAF cleanup on unmount (line 105-110). ‚úì exactly the right pattern.
  - **Stable callback refs** (lines 89-102): all 6 consumer callbacks mirrored to refs in always-running useEffect; handlers use refs instead of deps ‚Üí empty-deps `useCallback` everywhere ‚Üí handlers never recreate, event listeners never re-attach. ‚úì best-practice for `<video>` event-handler hygiene.
  - **DOM-as-source-of-truth for muted**: `toggleMute` mutates `el.muted = !el.muted`; the browser's `volumechange` event fires synchronously and dispatches `set-muted` ‚Üí reducer. ‚úì avoids state-DOM divergence.
  - **`isActive` auto-pause** owned by hook (lines 113-115). Single-dep effect; no manual pause/resume coordination at consumer level. ‚úì
  - **`loadstart` clears stale state** on src change (per meta) ‚Äî verified: reducer's `loadstart` returns `{ ...initialState, isMuted: state.isMuted }` so muted preference persists across src swaps. ‚úì
  - **`useDoubleTap`** is exported as a public hook for non-video consumers (`expandable-text-01`, etc.). Reusable cleanly. ‚úì
  - Three small findings ‚Äî see below.

## Findings (3)

### F-01 ‚Äî `useDoubleTap` uses `Date.now()` instead of `performance.now()`

- **Severity:** üîπ Low
- **Dimension:** 3 ‚Äî Component code
- **Location:** `src/registry/components/media/video-player-01/hooks/use-double-tap.ts:25`
- **Observed:** `const now = Date.now()` for the tap-window comparison. Same shape as story-viewer-01 F-01 ‚Äî `Date.now()` is wall-clock and can jump on NTP/DST corrections; `performance.now()` is monotonic and cheaper. For a 300ms double-tap window the practical risk is zero (system clock corrections within that window are essentially impossible), but `performance.now()` is the principled choice for any "elapsed-since" calculation.
- **Suggested fix:** Replace with `performance.now()`. One-line change. Consider doing both video-player-01 F-01 and story-viewer-01 F-01 (matching shape) in the same v0.1.1 patch.

### F-02 ‚Äî `play()` rejection silently swallowed regardless of error type

- **Severity:** üîπ Low
- **Dimension:** 3 ‚Äî Component code (error handling)
- **Location:** `src/registry/components/media/video-player-01/hooks/use-video-state.ts:196-198`
- **Observed:** `void el.play().catch(() => { /* autoplay policy may reject */ })`. The catch swallows ALL `play()` rejections. The common case is `NotAllowedError` (browser autoplay policy ‚Äî expected, correctly ignored). But `play()` can also reject with `NotSupportedError` (codec/format unsupported) or `AbortError` (load aborted) ‚Äî these are real failures the consumer's `onError` should surface. Per the review-guide's Dim 3 bad signals: *"No swallowed errors. `catch (e) {}` is a smell. Even `console.warn` is better than silence."* The current silence is correct *for autoplay policy* but wrong *for hardware/codec failures*.
- **Suggested fix:** Distinguish error names: `void el.play().catch((err) => { if (err.name === "NotAllowedError") return; onErrorRef.current?.(el.error ?? null); })`. Routes only autoplay-policy rejections through silent swallow; surfaces real failures via the existing `onError` callback. The comment can be retained explaining why NotAllowedError is silent.

### F-03 ‚Äî `onError` callback receives `MediaError | null`; null case undocumented

- **Severity:** üîπ Low
- **Dimension:** 3 ‚Äî Component code (API contract clarity)
- **Location:** `src/registry/components/media/video-player-01/hooks/use-video-state.ts:155-161` + `types.ts` (onError prop signature)
- **Observed:** `handleVideoError` invokes `onErrorRef.current?.(el.error)` ‚Äî `el.error` (`HTMLMediaElement.error`) is typed `MediaError | null` per the HTMLMediaElement spec. Most error events do populate it, but `null` is allowable per spec when the error has been cleared or the element is in an indeterminate error state. Consumers reading the prop type as `(error: MediaError) => void` (non-null) would be wrong. The plan/guide doesn't call this out.
- **Suggested fix:** Either (a) update the prop type to `onError?: (error: MediaError | null) => void` (preferred ‚Äî TypeScript-enforced contract); OR (b) coerce in the handler (`if (el.error) onErrorRef.current?.(el.error)`) so consumers always receive a populated error (cleaner DX but loses the "error-cleared" signal). Option (a) matches the underlying browser API; option (b) trades fidelity for consumer ergonomics. Document the choice.

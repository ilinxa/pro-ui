## Findings

#### F-01 — Icon helper missed forwardRef component objects

- **Severity:** ⚠️ High (closed in `52e5f33` before close)
- **Dimension:** Verification
- **Location:** [`parts/icon.tsx:22-30`](../../../../src/registry/components/navigation/rich-sidebar/parts/icon.tsx#L22-L30)
- **Observed:** `parts/icon.tsx` used `typeof icon === "function"` to detect component-type icons, but lucide-react v0.475+ ships icons as `React.forwardRef` objects (`typeof === "object"`). The component object fell through to the ReactNode branch and crashed the static prerender with `Objects are not valid as a React child (found: object with keys {$$typeof, render, displayName})`. Latent since C8; surfaced by the first prod build run since C7.
- **Suggested fix:** Add an `isComponentObject` guard alongside the function check: `typeof icon === "object" && icon !== null && "$$typeof" in icon`. **Already landed in `52e5f33`.**

#### F-03 — Skip-link container `position: relative` only on desktop nav

- **Severity:** 🔹 Low (cosmetic / latent)
- **Dimension:** Public API
- **Location:** [`rich-sidebar.tsx:646`](../../../../src/registry/components/navigation/rich-sidebar/rich-sidebar.tsx#L646) and the Sheet branch at [`rich-sidebar.tsx:692`](../../../../src/registry/components/navigation/rich-sidebar/rich-sidebar.tsx#L692)
- **Observed:** Both branches use `className="relative …"` so the skip-link's `focus:absolute focus:left-2 focus:top-2` anchors correctly. If a consumer overrides the outer `className` to remove `relative`, the skip link will anchor to the next-up positioned ancestor (likely the page body) and appear in an unexpected location.
- **Suggested fix:** Document in guide.md that skip-link positioning depends on the nav root being a positioning context. Alternatively, force `relative` via `cn("relative", className)` — already done in the desktop branch; the Sheet branch is also `relative` via `cn("relative …", "flex w-72 …")`. Verified in both render paths. **No code change needed; guide-level callout only.**

#### F-04 — `data-side` flips for `mobileDrawerSide` not handled in RTL

- **Severity:** 🔹 Low (acceptable v0.1 limitation)
- **Dimension:** Public API
- **Location:** [`rich-sidebar.tsx:686-689`](../../../../src/registry/components/navigation/rich-sidebar/rich-sidebar.tsx#L686-L689)
- **Observed:** Sheet's `side={drawerSide}` is a physical CSS position (`left` / `right`) — under `dir="rtl"` it doesn't auto-flip. A consumer who sets `side="left"` expecting their sidebar to slide-in from the inline-start edge will see it slide from the physical left even in RTL.
- **Suggested fix:** Either (a) auto-flip `drawerSide` when document direction is RTL via a `useMatchMedia("(direction: rtl)")` style check, or (b) document the limitation and recommend `side="start"` semantics in v0.2. Defer to v0.2.
- **Owner:** assistant. **Target:** v0.2.0 (logical-direction sweep).

#### F-05 — Active variant `left-bar` / `right-bar` use physical sides not logical

- **Severity:** 🔹 Low (variant name suggests physical; consumer can choose)
- **Dimension:** Public API
- **Location:** [`lib/active-variant-classes.ts:27-40`](../../../../src/registry/components/navigation/rich-sidebar/lib/active-variant-classes.ts#L27-L40)
- **Observed:** Variant `"left-bar"` literally renders `before:left-0` and `"right-bar"` renders `after:right-0`. In RTL with `dir="rtl"`, these don't auto-flip — `left-bar` always appears physically left of the row, regardless of writing direction. Since the variant *name* implies physical side, this is arguably correct behavior. A future logical variant pair would be `"start-bar"` / `"end-bar"`.
- **Suggested fix:** Document explicitly in guide.md; defer the logical-direction variant pair to v0.2.
- **Owner:** assistant. **Target:** guide.md note now + v0.2 logical variants.

### Rotating dimension

- **Chosen:** **Public API** (justified by C2 reason discriminator + C4 onClick widening — both touch user-visible contracts; L54 + L56 promote new public surface; L55 codifies the renderItem `<li>` contract).
- **Why:** v0.3 is a fix-and-tighten minor bump that touches the public API in 4 ways: (a) `RichSidebarHandle.{open,close,toggle}Mobile` widen to accept optional `reason?: RichSidebarMobileOpenReason` (additive — preserves v0.2.x callers); (b) `NavUserMenuItem.onClick` signature widens from `React.MouseEvent` to `Event | React.MouseEvent` (TypeScript-BREAKING — migration note in guide); (c) `NavUserMenuItemSelectEvent` new exported type alias; (d) `onMobileOpenChange.reason` discriminator behavior changes from permanent `"imperative"` to actual reason carry-through (no API change, behavior change). The contract must be honest about each — especially L55's "library owns `<li>` wrapping" invariant that depends on the C1 SidebarNavRow refactor.
- **Verified?** **yes**
- **Notes:**
  - **L53 `lastMobileOpenReason` reducer state** — verified at [`lib/sidebar-reducer.ts`](../../../src/registry/components/navigation/rich-sidebar/lib/sidebar-reducer.ts). Field added to `SidebarReducerState`; `createInitialState` initializes to `null`; `SET_MOBILE_OPEN` writes `action.reason` ONLY when transition is non-no-op (no-op guard at the front of the case preserves prior reason — load-bearing for the L53 "no double-fire on re-entry" guarantee); `TOGGLE_MOBILE` widened with optional `reason?` and writes `action.reason ?? "imperative"`; `EXTERNAL_SYNC` resets to `"imperative"` ONLY on transition-causing syncs (R23 mitigation — prevents stale `"item-click"` leaking into prop-driven close).
  - **L54 handle `reason?` params** — verified at [`types.ts`](../../../src/registry/components/navigation/rich-sidebar/types.ts) (`RichSidebarHandle.openMobile` / `closeMobile` / `toggleMobile` all accept optional `RichSidebarMobileOpenReason`) and at [`lib/build-handle.ts`](../../../src/registry/components/navigation/rich-sidebar/lib/build-handle.ts) where the methods pass `reason ?? "imperative"` through to dispatch. **Critical detail**: `toggleMobile` dispatches `TOGGLE_MOBILE` (NOT a translated SET_MOBILE_OPEN) so the reducer reads fresh `state.mobileOpen` and handles rapid same-tick double-clicks correctly. The race fix is documented in plan §4 C2 + R21 mitigation.
  - **L55 `<li>` ownership inversion** — verified at [`parts/sidebar-nav-row.tsx`](../../../src/registry/components/navigation/rich-sidebar/parts/sidebar-nav-row.tsx) (returns only `<TooltipWrapper>…{linkEl}…</TooltipWrapper>` — no outer `<li>`) and [`parts/sidebar-nav-list.tsx`](../../../src/registry/components/navigation/rich-sidebar/parts/sidebar-nav-list.tsx) (single `<li className={cn("list-none", item.className)} data-testid={item["data-testid"]} key={item.id}>` wrapper applied to BOTH the default path AND the `renderItem`-slot path, with the slot's return as the body). #### F-01 — Path-b consumer-tsc smoke deferred to post-deploy

- **Owner / bump target:** Author / v0.3.1 if smoke surfaces anything; otherwise closed at post-deploy verification.

#### F-02 — Pre-existing `showDesktopHeader` unused-var warning in rich-sidebar.tsx

- **Severity:** 🔹 Low
- **Dimension:** Verification (lint hygiene)
- **Location:** [`rich-sidebar.tsx:537`](../../../src/registry/components/navigation/rich-sidebar/rich-sidebar.tsx#L537) — `const showDesktopHeader = !!headerSlot || !!resolvedBrand || desktopAccessory !== null;`
- **Observed:** Pre-existing in v0.2.x (the variable was computed but only `showMobileHeader` is consumed inside `renderInnerChrome`; desktop path computes its own `shouldShowBrandRow`). Not introduced by v0.3.0 work but surfaced in C5's lint run.
- **Suggested fix:** Delete the line (trivial cleanup); does not warrant blocking v0.3.0.
- **Owner / bump target:** Author / v0.3.1 patch.

#### F-03 — Storage rehydration does not fire `onCollapsedChange` (DOCUMENTED as intended)

- **Severity:** 🔹 Low
- **Dimension:** Public API (docs)
- **Location:** [`types.ts`](../../../src/registry/components/navigation/rich-sidebar/types.ts) `onCollapsedChange` JSDoc + [`hooks/use-storage-sync.ts`](../../../src/registry/components/navigation/rich-sidebar/hooks/use-storage-sync.ts)
- **Observed:** v0.3.0 documents this behavior as intended (consumer's persisted state is already reflected at mount; firing the callback would be confusing). However, a consumer with analytics wired to ALL collapse-state changes (including rehydration) is silently underreporting. Not a bug per the JSDoc lock, but worth surfacing.
- **Suggested fix:** Consider a `fireOnRehydrate?: boolean` opt-in prop for consumers who DO want the rehydration event. Deferred per plan §"Out of scope": "Revisit if a real consumer files a feature request."
- **Owner / bump target:** Author / deferred to v0.4 or later.

#### F-04 — F-cross-13 graceful-degradation path for `<SheetContent>` handlers untested

- **Severity:** 🔹 Low
- **Dimension:** Robustness
- **Location:** [`rich-sidebar.tsx`](../../../src/registry/components/navigation/rich-sidebar/rich-sidebar.tsx) `<SheetContent onPointerDownOutside>` + `onEscapeKeyDown` wiring
- **Observed:** The plan §6 R16 mitigation says "if shadcn migrates to Base UI and these handlers don't fire, `onOpenChange` fallback fires with `reason: \"imperative\"`". The fallback IS wired and the no-op guard IS in place, but there's no actual Base UI test setup to confirm graceful degradation. We're trusting the architecture.
- **Suggested fix:** When/if shadcn ships a Base UI Sheet variant, run a smoke test against that primitive specifically. Until then, the F-cross-13 pattern is the safety net.
- **Owner / bump target:** Author / when shadcn migrates Sheet primitive (no specific date).

## Findings

### F-01 ‚Äî `useVirtualizer` informational warning (√ó2)

üîπ **Low** ‚Äî `src/registry/components/navigation/file-manager/hooks/use-list-virtual.ts` (and the existing one on file-tree's `use-tree-virtual.ts`).

Same as file-tree's F-01 ‚Äî React Compiler skips memoizing `useVirtualizer`. Functionally correct; no stale-UI risk in our usage. Document in guide so future contributors don't try to "fix" it.

**Severity:** Low ‚Äî informational only.

### F-02 ‚Äî `useImperativeHandle` deps include the full `state` and `actions` objects

üî∏ **Medium** ‚Äî `src/registry/components/navigation/file-manager/file-manager.tsx`

The imperative handle's `useImperativeHandle(ref, () => ({ state, actions }), [state, actions])` rebuilds the handle on every render because `state` and `actions` are themselves rebuilt on each render (memoization breaks via React Compiler skip on `useVirtualizer` ‚Äî see F-01). Consumers comparing `ref.current` between renders see a fresh object every time.

This is the same finding pattern as pdf-viewer's F-02 ("actions identity"). Mitigation lands in v0.1.1: stabilize via lazy getters or `useImperativeHandle(ref, () => stableHandleRef.current, [])` where `stableHandleRef.current = { state, actions }` is updated in an effect.

**Severity:** Medium ‚Äî doesn't break functionality, but breaks identity-based comparison in consumer code (`useEffect(() => ..., [ref.current])` would re-fire every render). Bump-target: **v0.1.1**.

### F-03 ‚Äî Cut + paste into the same folder fires `onPaste` (no-op move)

üî∏ **Medium** ‚Äî `src/registry/components/navigation/file-manager/file-manager.tsx` (`paste` callback)

Same finding pattern as file-tree's F-03 ‚Äî when the cut source's parent is already the current folder, paste fires `onPaste` anyway, leaving the consumer to detect the no-op. Document the behavior in guide.md (already done) and add a guard in v0.1.1.

**Severity:** Medium ‚Äî not a correctness bug; consumer can guard. Bump-target: **v0.1.1**.

## Verdict

**Pass with follow-ups.**

Largest component in the library to date (32 sealed-folder files + 1 `_shared/` primitive); largest public API surface (~85 props, 28 actions, 6 slots, 18 callbacks). Verification all clean (40/40 meta-deps, build succeeds, registry artifacts regenerate). Three findings are all v0.1.1 follow-ups; F-04 is procedural.

In the same review session I caught + fixed (no findings logged):
- React Compiler `react-hooks/refs` violation in `use-navigation-history.ts` ‚Äî refactored from `useRef` to `useState` for back/forward stacks.
- `react-hooks/set-state-in-effect` violation in `use-selection.ts`'s clear-on-navigate effect ‚Äî moved the clear logic into the navigate-action wrappers in the orchestrator (cleaner; consumer-driven `currentFolderId` mutation does NOT clear, documented in usage gotchas).
- `react-hooks/immutability` violation in `use-visible-items.ts` (totalSize reassignment in map) ‚Äî switched to `reduce`.
- `react-hooks/refs` forward-reference issue in `use-marquee.ts` ‚Äî defined the move/up handlers inside the pointer-down callback so refs don't precede declaration.
- `react-hooks/set-state-in-effect` (kickoff) for `use-lazy-load.ts` ‚Äî wrapped `triggerLoad` in `queueMicrotask` so the synchronous setLoadingIds doesn't violate during the effect's body.
- Unused `useRef` import in `file-manager-content-pane.tsx`, unused `getVisibleIds` arg in `use-marquee.ts`, unused `_additive` in callbacks.
- Tailwind canonical-class warning: `min-w-[280px]` ‚Üí `min-w-70`.

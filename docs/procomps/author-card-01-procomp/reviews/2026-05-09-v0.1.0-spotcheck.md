The load-bearing risk is **token discipline**: does it hold the line on `--font-serif` / `--primary` / `--card` / lime-accent restraint, or does it drift toward generic Tailwind defaults? Marketing components especially are at risk for AI-aesthetic drift (purple gradients, system-font fallbacks).

- **Notes:**
  - **Fonts:** uses `font-serif` for the heading (line 50). Project token resolution: `--font-serif: var(--font-playfair-display), "Lora", ui-serif, Georgia, serif;` (verified in `src/app/globals.css:19`). Consistent with the editorial-news family. ✓

  - **`primary` tone** uses `bg-primary/10` + `text-primary` (faint lime tint with lime icon). Pairs the project's signal-lime correctly; chroma stays at 0.20 token. ✓
  - **`accent` tone** uses shadcn's `bg-accent` + `text-accent-foreground` — that's the *neutral* shadcn accent, NOT the project's lime. Naming overload — see F-01.
  - **`muted` tone** uses `bg-muted` + `text-muted-foreground` ✓ matches CLAUDE.md "subtle surface" mandate.
  - **Card surface:** `bg-card rounded-2xl border-border/50` ✓. Lifted off the cool off-white in light, raised on graphite-cool in dark — matches the design system's surface elevation rules.
  - **Focus ring:** `focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-2` (when polymorphic root is clickable). ✓ project focus token.
  - **Hover:** `hover:border-border` (gentle border darken) + `transition-colors`. Subtle, restrained, matches the "one orchestrated reveal per page" mandate ✓.

## Findings (2)

### F-01 — Tone naming `accent` collides with conceptual "brand accent" expectation

- **Severity:** 🔹 Low
- **Dimension:** 6 — Design system (API + token clarity)
- **Location:** `src/registry/components/marketing/author-card-01/types.ts:3` + `parts/tone-resolver.ts:8-12`
- **Observed:** `tone: "primary" | "accent" | "muted"` maps to:
  - `primary` → `bg-primary/10` + `text-primary` (faint lime tint + lime icon)
  - `accent` → `bg-accent` + `text-accent-foreground` (shadcn's neutral hover/active surface)
  - `muted` → `bg-muted` + `text-muted-foreground` (subtle surface)
  
  In the project's design-system mandate (CLAUDE.md), "accent" intuitively reads as "the brand accent" (signal-lime). But here `accent` maps to shadcn's neutral `--accent` — a low-chroma surface, often near-invisible against `bg-card`. A consumer reading the API expects `tone="accent"` to dial up the brand-lime presence; they actually get something visually quieter than `tone="primary"`. The current `primary` is the lime tone; `accent` is the surface tone.
- **Suggested fix:** Rename in v0.2 (breaking) to `primary | neutral | muted` (or `lime | surface | muted` for explicit brand mapping). Current naming ships; v0.2 migration note documents the rename. Until then, the guide should call out this naming quirk explicitly.

### F-02 — Editorial section heading "About the author" precedes the name; semantic ordering is unusual

- **Severity:** 🔹 Low
- **Dimension:** 6 — Design system (editorial pattern) / Dim 1 (doc clarity)
- **Location:** `src/registry/components/marketing/author-card-01/author-card-01.tsx:48-55` + `types.ts:27-29`
- **Observed:** The component renders `<HeadingTag>{labels.heading}</HeadingTag>` (default `<h3>"About the author"</h3>`) ABOVE the name+role row. The author's name is a `<p>` (line 65-70). Most author-card patterns use the name as the heading (`<h3>{name}</h3>` with role + bio in a sub-block). This component inverts that — the heading is a section label, the name is body text. For SR users navigating by headings, the level appears as "About the author" rather than the actual author name; the name itself is non-headed. Configurable via `labels.heading`, but the structural inversion is locked.
- **Suggested fix:** Either (a) document the editorial-pattern choice prominently in the guide so consumers understand why navigating by H3 surfaces "About the author" not the author's name (preferred — the choice is intentional for sidebars where author is one of many sections), or (b) v0.2 add `headingMode: "section" | "name"` to let consumers pick the structural rhythm. The current implementation is correct for sidebar editorial use; just document the trade-off.

## 1. Executive summary

`kanban-board-01` v0.2 ships a high-quality renderer-registry kanban with 34 source files, complete planning docs, robust DnD wiring, atomic auto-expand-on-drop, and the marquee rich-card adapter pattern fully demonstrated. The v0.2 additive changes (`dragHandle: "shell" | "header"`, native column scroll, rich-card demo) are non-breaking and well-implemented. Verification: `pnpm tsc --noEmit` clean, `pnpm lint` clean (1 pre-existing rich-card warning, no kanban warnings), `pnpm registry:build` clean, smoke install **passes** in 11.7s against the production Vercel registry.

**Headline finding:** the built-in `kanban-note` renderer hardcodes Tailwind palette colors (`amber-500`, `amber-50`, `amber-950`, `amber-900`, `amber-200`) in direct violation of the description-§6.6 success criterion ("no hardcoded colors — semantic Tailwind tokens only"), and the `KanbanNoteData.color` field declared in `types.ts` is unused at render time. Six findings total — two High, four Medium, no Blockers.

**Recommended next step:** ship a **v0.2.1 patch** addressing F-01 (kanban-note color tokens + unused `color` field), F-03 (dead-code `use-keyboard-actions.ts`), F-04 (usage import paths), F-05 (demo `col-review` swatch). F-02 (STATUS.md staleness) and F-06 (drag-over visual feedback) can land in next-normal-cadence.

## 2. Strengths

Locking these in for future components:

- **Renderer-registry pattern delivers on its marquee promise.** The v0.2 rich-card adapter in [`demo.tsx:15-41`](../../../src/registry/components/data/kanban-board-01/demo.tsx) proves heterogeneous columns work end-to-end — three renderer kinds (kanban-card, kanban-note, rich-card) sharing a single column with full feature passthrough on the rich-card body. This is the pattern future renderer-host components (workspace, flow-canvas-01) should mirror.
- **Atomic auto-expand-on-drop in the `move-item` action.** [`lib/data.ts:57`](../../../src/registry/components/data/kanban-board-01/lib/data.ts) bundles collapse-clearing into the same dispatch — explicitly justified in plan §5.7 as avoiding the controlled-mode race ("two separate dispatches in the same tick race because consumer's `setData(next1)` then `setData(next2)` both compute from the same captured render-state"). Worth replicating for any host with collapsible regions.
- **Sealed `lib/` purity rule respected.** `lib/data.ts`, `lib/permissions.ts`, `lib/palette.ts`, `lib/ids.ts`, `lib/reducer.ts` — zero React imports, all pure functions. Built-in renderers correctly live in `parts/` despite being publicly exported (because their `render`/`editForm` fields return JSX) — deviation justified in plan Q-P10.
- **Plan-vs-code drift tracked explicitly.** Plan Q-P4 was originally "Auto-expand after 600ms hover during drag-over"; during implementation the simpler "drop-on-collapsed → expand AND accept drop" was found sufficient. The plan doc carries an explicit `(shipped)` annotation explaining the deviation. This is the right way to handle plan drift — loud, not silent.
- **Public helper API is open and consistent.** `findRenderer`, `newItemId`, `newColumnId`, `validateData`, `DEFAULT_PALETTE`, `KanbanCardView`, `KanbanNoteView` — all re-exported from [`index.ts`](../../../src/registry/components/data/kanban-board-01/index.ts). Consumers writing custom inline editors or visual variants have access to the same primitives the component uses internally.
- **A11y wiring is thorough.** `aria-roledescription` per renderer label, `role="article"` (correctly chosen over `role="button"` to permit nested Edit/Delete buttons — see plan §11 deviation note), focus-visible rings, dnd-kit's keyboard sensor + announcer integration, and a graceful `<MissingRendererFallback>` fallback for unknown renderer ids.
- **`MissingRendererFallback` + once-per-session warn pattern.** [`parts/missing-renderer.tsx:17-28`](../../../src/registry/components/data/kanban-board-01/parts/missing-renderer.tsx) — graceful degradation with developer feedback. Right pattern for any renderer-registry host.
- **Demo dataset is realistic and heterogeneous.** [`demo.tsx`](../../../src/registry/components/data/kanban-board-01/demo.tsx) seeds 4 columns × 2 swimlanes with 3 rich-card items (every predefined RichCard key — codearea, image, table, quote, list — plus nested children + `__rcmeta`) + 6 kanban-card + 2 kanban-note items. Exercises everything the description §5 promises. Not three lorem items.
- **Read-only mode short-circuits DndContext.** [`kanban-board-01.tsx:143-164`](../../../src/registry/components/data/kanban-board-01/kanban-board-01.tsx) skips the entire DnD wiring tree in readOnly mode rather than attempting to disable each sensor. Cleaner contract, lower runtime cost.
- **Smoke install pass.** `pnpm dlx shadcn@latest add @ilinxa/kanban-board-01` against the production Vercel registry succeeds in 11.7s. Locked target convention verified — every file `type: "registry:component"`, `target: "components/kanban-board-01/<sub-path>"`. None of the 8 shadcn primitives or 4 npm deps fail to resolve.

### 3.2 High (⚠️)

#### F-01 — `kanban-note` renderer hardcodes Tailwind palette colors; violates "no hardcoded colors" success criterion

- **Severity:** ⚠️ High
- **Dimension:** 6 (Design system)
- **Location:** [`src/registry/components/data/kanban-board-01/parts/kanban-note.tsx:20-21,27,60`](../../../src/registry/components/data/kanban-board-01/parts/kanban-note.tsx)
- **Observed:** The `KanbanNoteView` and `KanbanNoteEditForm` components hardcode Tailwind palette amber values: `border-l-amber-500/60`, `bg-amber-50/30`, `dark:bg-amber-950/20`, `text-amber-900 dark:text-amber-200`. None of these resolve to project semantic tokens (`--card`, `--border`, `--primary`, `--chart-*`, etc.). Additionally, `KanbanNoteData.color?: string` is declared in [`types.ts:118`](../../../src/registry/components/data/kanban-board-01/types.ts) but the renderer never reads it — consumer-supplied colors are silently ignored.
- **Why it matters:** Description §6 success criterion #6 is explicit: *"No hardcoded colors — semantic Tailwind tokens only; column color palette is built from design tokens."* Plan §6 specifies the palette exclusively uses CSS variables. Hardcoded amber values fragment the visual vocabulary across the library and fail the dark-mode design-token audit. The unused `color` field is also a misleading public API: the type promises configurability that the runtime ignores.
- **Suggested fix:** Replace the amber Tailwind classes with semantic-token equivalents in one of two ways:
  1. Introduce a project-level `--note-accent` (or `--note-foreground`/`--note-bg`) CSS variable in `globals.css` and use `var()` in inline styles — same pattern the column-color picker uses.
  2. Use the existing palette swatch system: read `KanbanNoteData.color` (mapping to a `KanbanPaletteSwatch.id`), resolve via `findSwatch + swatchCssColor`, render with `style={{ borderLeftColor: ... }}`. Default to a neutral semantic token (e.g. `--muted`) when no color provided.
  
  Either way, removing the unused `color` field from `KanbanNoteData` (or wiring it up) is required to align the type with reality.

### 3.3 Medium (🔸)

#### F-03 — `hooks/use-keyboard-actions.ts` is unused dead code shipped to consumers

- **Severity:** 🔸 Medium
- **Dimension:** 3 (Component code)
- **Location:** [`src/registry/components/data/kanban-board-01/hooks/use-keyboard-actions.ts`](../../../src/registry/components/data/kanban-board-01/hooks/use-keyboard-actions.ts)
- **Observed:** The file exports `useKeyboardActions()` with an empty body and a comment "Reserved for future keyboard shortcuts… Intentionally empty in v0.1." No file in the kanban folder imports this hook (verified against `kanban-board-01.tsx` and all parts/hooks). The file is included in `registry.json`'s base item (line 1340), so consumers receive an empty stub hook in their installed component folder.
- **Why it matters:** The project's "no half-finished implementations" rule and the review-guide §3 cleanness criteria call out dead code as a smell. Shipping it to consumers is worse — they'll see the empty hook and either delete it (extra cleanup) or be confused about whether they're missing functionality. The "future shortcuts" justification is speculative scope creep — the existing `parts/item-shell.tsx` Enter→onClick handler already covers per-item keyboard, and dnd-kit's KeyboardSensor handles the rest.
- **Suggested fix:** Delete `hooks/use-keyboard-actions.ts`. Remove the corresponding entry from `registry.json` files-array (line 1340). Re-run `pnpm registry:build`. If keyboard actions are added in v0.3, author the file then.

#### F-04 — `usage.tsx` mixes producer-side and consumer-side import paths

- **Severity:** 🔸 Medium
- **Dimension:** 4 (Demo + usage)
- **Location:** [`src/registry/components/data/kanban-board-01/usage.tsx:40-42 vs 105`](../../../src/registry/components/data/kanban-board-01/usage.tsx)
- **Observed:** The "Basic example" code block (lines 40-42) imports from `@/registry/components/data/kanban-board-01` (the producer's docs-site path), which won't resolve in a consumer's installed app. The "Mixing rich cards" code block on line 105 correctly uses `@ilinxa/rich-card` (consumer-side namespace path). The same file ships two incompatible import conventions.
- **Why it matters:** A consumer copy-pasting the basic example into their app immediately hits an unresolved-module error. The first impression is broken-out-of-the-box. The internal inconsistency also makes the docs feel half-edited.
- **Suggested fix:** Standardize on the consumer-side path. Replace lines 40-42 with:
  ```ts
  import { KanbanBoard, kanbanCardRenderer, kanbanNoteRenderer } from "@/components/kanban-board-01";
  ```
  After `pnpm dlx shadcn add @ilinxa/kanban-board-01`, the install lands at `components/kanban-board-01/` per the locked target convention, and the barrel `index.ts` re-exports all three.

#### F-05 — Demo's `col-review` uses non-existent palette swatch `"indigo"`

- **Severity:** 🔸 Medium
- **Dimension:** 4 (Demo + usage)
- **Location:** [`src/registry/components/data/kanban-board-01/demo.tsx:193`](../../../src/registry/components/data/kanban-board-01/demo.tsx)
- **Observed:** The demo's third column declares `color: "indigo"`. The DEFAULT_PALETTE in [`lib/palette.ts:3-10`](../../../src/registry/components/data/kanban-board-01/lib/palette.ts) defines exactly 6 swatches: `slate`, `lime`, `amber`, `emerald`, `sky`, `rose`. There is no `indigo` swatch. `findSwatch("indigo")` returns undefined; `swatchCssColor(undefined)` returns undefined; the column receives no `borderLeftColor` style and renders without the intended accent. The demo silently no-ops — no error, no warning, just an undecorated column.
- **Why it matters:** The demo is the docs-site's marquee marketing surface for the component. A column that visibly lacks the color it tried to declare gives a poor first impression and undermines the "color picker" feature being demonstrated. It also masks the underlying graceful-degradation behavior (which IS the right behavior; the bug is in the demo data).
- **Suggested fix:** Change `col-review`'s `color: "indigo"` to one of the six valid swatch ids — likely `"sky"` for "review" semantics (info / blocked-by-others per plan §6 visual-intent table). Alternative: extend `DEFAULT_PALETTE` to include an `indigo` swatch mapping to a project token (e.g. `--chart-4` if one is reserved).

#### F-06 — `onDragOver` visual rejection feedback is incomplete vs plan §3.5

- **Severity:** 🔸 Medium
- **Dimension:** 7 (Accessibility — UX feedback)
- **Location:** [`hooks/use-drag-handlers.ts:49-51`](../../../src/registry/components/data/kanban-board-01/hooks/use-drag-handlers.ts), [`parts/column-body.tsx:31-34`](../../../src/registry/components/data/kanban-board-01/parts/column-body.tsx), [`parts/swimlane-cell.tsx:51-52`](../../../src/registry/components/data/kanban-board-01/parts/swimlane-cell.tsx)
- **Observed:** Plan §3.5 specifies: *"`canDrop` runs in `onDragOver` (for live UI feedback — invalid target gets a red ring + cursor: not-allowed) AND in `onDragEnd` (final check before dispatch)."* The actual `onDragOver` handler (lines 49-51) is empty with the comment *"Visual rejection is handled inside SwimlaneCell via column.acceptsRendererIds + activeRendererId."* But the `rejectDrop` computed in `column-body.tsx:31-34` only checks `column.acceptsRendererIds`, NOT the full `canDrop` predicate. Consequence: `allowReorder: false` / `allowIncoming: false` / `allowOutgoing: false` columns and `item.locked: true` items do NOT get a red rejection ring during drag-over — only a renderer-id mismatch does. The drop is correctly rejected at `onDragEnd` (canDrop runs there), but the visual feedback during the gesture is partial.
- **Why it matters:** Drag-and-drop UX depends heavily on real-time feedback — users learn the rules of the system by what they see. A user dragging an item over an `allowIncoming: false` column sees no warning during drag, then has the drop silently rejected at end-of-gesture, with the item snapping back. Compared to a column that visibly turns red mid-drag (per plan), this is a measurably worse experience. The accessibility live region announces the result but the visible feedback lags.
- **Suggested fix:** Either (a) wire the full `canDrop` predicate into `onDragOver` and pipe the result down to `column-body`/`swimlane-cell`'s `rejectDrop` computation, OR (b) update plan §3.5 to document the deviation explicitly (similar to Q-P4's `(shipped)` annotation) and track the missing visual feedback as a v0.3 follow-up. Either path is acceptable; silent drift is not.

## 8. Reviewer reflection

Things that surfaced about the review system itself during the pilot:

- The 133-item checklist is dense but tractable in a focused review block; the per-dimension grouping helps maintain mental locality. No items felt redundant.
- The smoke harness install loop (37 × ~5-15s avg, plus a few outliers like engagement-bar-01 at 51s) is the right tradeoff vs manual per-component install. Took ~5 minutes total. The expected-fail classification for `flow-canvas-01` worked as designed; the 5 unexpected-fails are real signal the system caught.
- F-01 (kanban-note hardcoded colors) was the kind of finding the review system is specifically designed to catch — it's been in the code since v0.1.0, passes tsc + lint, and would never surface without an explicit Dimension-6 dive. Validates the system's value.
- The plan-vs-code drift on F-06 is exactly the kind of subtle issue that requires reading both plan AND code carefully — a checklist alone wouldn't catch it. The eight-step review order's "read planning docs first" step proved essential.
- The pilot took ~120 minutes of focused effort excluding the harness scaffolding (which itself took ~15 minutes once `create-next-app` + `shadcn init` completed in background). Confirms the review-process.md estimate of 90-120 min for Tier 1 reviews.
- One template ergonomics note: review-guide.md and review-process.md are excellent reference but the report template fields could use slightly more inline guidance for first-time reviewers (e.g. what the "Trigger" field expects). Minor — not worth changing the templates mid-sweep, but worth capturing for any post-sweep template revision.

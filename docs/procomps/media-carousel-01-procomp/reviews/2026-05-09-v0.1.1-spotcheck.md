### Rotating dimension

- **Chosen:** Dim 7 — Accessibility
- **Why:** Carousels are a notorious WAI-ARIA failure point — slide announcement, focus management when slides change, keyboard nav, indicator semantics, and inactive-slide focus-trapping all distinguish a real APG-pattern carousel from a "just a swipeable strip." `meta.context` claims **"WAI-ARIA APG carousel pattern (region / slide group / aria-current on indicators)"** — verifying that claim holds for the user-facing surfaces is exactly Dim 7's job.
- **Verified?** yes (with 2 substantive Medium findings)
- **Notes:**
  - **Region semantics ✓** (`carousel-track.tsx:209-213`): `role="region"` + `aria-roledescription="carousel"` + `aria-label={labels.carouselLabel}`.
  - **Slide group semantics ✓** (`carousel-track.tsx:96-99`): `role="group"` + `aria-roledescription="slide"` + per-slide aria-label formatted as "Slide {index} of {total}" via `labels.slideAriaLabel`.
  - **Nav button semantics ✓** (`parts/nav-buttons.tsx:21,31`): `aria-label={labels.previousSlide}` / `aria-label={labels.nextSlide}`. Chevron icons properly `aria-hidden="true"`. RTL flip via `rtl:rotate-180`.
  - **Indicator dot semantics ✓** (`parts/indicator-dots.tsx:23-35`): native `<button>` per dot with `aria-label="${goToSlide} ${i+1}"` + `aria-current="true"` on active. Focus-visible ring. List-of-buttons pattern (vs tablist) — acceptable per WAI-ARIA APG carousel pattern.
  - **No autoplay** → no pause control needed, no `aria-live` region needed (APG only requires those for autoplay carousels). ✓ correctly absent.
  - **`<img loading="lazy">`** for image items (perf-friendly, default).
  - **Cross-folder video composition** via `useDoubleTap` from `@/registry/components/media/video-player-01` (line 6); video items use `VideoPlayer01`'s built-in controls + `isActive` propagation (per `meta.features`).
  - Two real a11y gaps surfaced — see findings.

## Findings (2)

### F-01 — Embla keyboard plugin not enabled; arrow-key slide navigation does not work when the track is focused

- **Severity:** 🔸 Medium
- **Dimension:** 7 — Accessibility (keyboard navigation)
- **Location:** `src/registry/components/media/media-carousel-01/parts/carousel-track.tsx:171-181` (`emblaOptions` memo)
- **Observed:** The Embla configuration sets `align`, `containScroll`, `loop`, `direction` — but does NOT enable the `keyboard` plugin (Embla's `embla-carousel-keyboard` package or the equivalent option). When a keyboard user Tabs into the carousel track region, arrow keys do nothing — they're not intercepted by Embla. The user can navigate slides only via:
  - Click/Tab to the prev/next chevron buttons (visible only when `showNavButtons=true`, default `true`)
  - Click/Tab to the indicator dots (visible only when `showIndicators=true`, default `true`)
  - Touch-swipe on the track itself
  
  Per WAI-ARIA APG carousel pattern: *"Carousels SHOULD support keyboard navigation including arrow keys when the slides region has focus."* The current implementation requires the user to find the prev/next buttons or indicators — adequate for keyboard-only users but slower than direct arrow-key nav.
- **Suggested fix:** Either (a) install `embla-carousel-keyboard` (~1KB) and add to `emblaOptions.plugins` (preferred — gives APG-conformant keyboard nav for free), OR (b) attach a `keydown` handler on the track wrapper that calls `scrollPrev()` / `scrollNext()` on `ArrowLeft` / `ArrowRight` (avoids the dep but reimplements). Option (a) is the lower-friction fix.

### F-02 — Inactive slides remain focusable; keyboard users can Tab into off-screen slide content

- **Severity:** 🔸 Medium
- **Dimension:** 7 — Accessibility (focus management)
- **Location:** `src/registry/components/media/media-carousel-01/parts/carousel-track.tsx:96-126` (the slide wrapper for gallery variant) + `:130-148` (linear variant)
- **Observed:** All slides render simultaneously (Embla uses `transform` to slide them in/out of view, not unmount). When a slide contains interactive content — most notably `<VideoPlayer01>` controls (play, mute, fullscreen, double-tap, etc.) — those controls are STILL in the tab order even when the slide is off-screen. A keyboard user navigating the page can Tab into a video player on slide 4 even though slide 1 is currently visible, leading to the focus indicator visually disappearing (it's on the off-screen slide). Per APG: *"Inactive slides should be removed from the tab order or marked `inert`."* The current implementation does neither — `aria-hidden` is also absent on inactive slides.
- **Suggested fix:** Add the HTML5 `inert` attribute to non-current slide wrappers when `!isActive`. The `inert` attribute removes the entire subtree from the tab order, hides it from assistive tech, and disables pointer events — a single attribute does all the focus management. Applied conditionally:
  ```tsx
  <div role="group" aria-roledescription="slide" inert={!isActive ? "" : undefined} ...>
  ```
  Browser support is universal (Chrome 102+, Safari 15.5+, Firefox 112+). React 19 supports the `inert` prop natively.

## Verdict

**Verdict:** Pass with follow-ups

**Justification:** media-carousel-01 v0.1.1 implements most of WAI-ARIA APG carousel pattern correctly — region/carousel/slide roledescriptions, per-slide aria-labels, indicator-dot aria-current, nav-button aria-labels, RTL chevron flip, focus-visible rings. Two genuine a11y gaps (F-01 keyboard nav inside track + F-02 focus-management for inactive slides) are 🔸 Medium and worth addressing in v0.1.2.

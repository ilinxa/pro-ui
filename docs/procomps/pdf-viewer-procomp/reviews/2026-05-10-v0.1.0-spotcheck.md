### Rotating dimension

- **Chosen:** dim 6 — Architecture / composition pattern
- **Why:** This is the most architecturally novel component in the registry — internal Context + standalone toolbar parts + slot overrides + imperative ref + hooks-heavy state machine. Higher than usual chance of architectural drift surfacing during the review.
- **Verified?** yes
- **Notes:**
  - Internal `PdfViewerContext` cleanly carries state + actions. Toolbar parts (`PdfPageNav`, `PdfZoomControls`, `PdfActionMenu`, `PdfPageIndicator`) all read via `usePdfViewer()` — no prop drilling, parts are fully composable inside `renderToolbar`.
  - State sources are clearly separated: `usePdfDocument` owns load lifecycle; `usePdfZoom` owns scale/fit; `usePdfPageTracker` owns current page; `usePdfDrop` owns drag state. No single hook is a god-object.
  - The slot pattern (`renderToolbar` / `renderContextMenu` / `renderPasswordPrompt`) gives consumers full takeover; default implementations are exported as parts so partial composition is also clean.
  - `useImperativeHandle` exposes a typed `PdfViewerHandle` for external control surfaces — verified via the demo's "Imperative ref" tab.
  - `"use client"` is on every file that needs it (parts, hooks, top-level). Lib files are pure — server-renderable in principle (though not used as such today).
  - One small architectural smell: `docState` and `actions` are passed *both* as part of `contextValue` AND as `useImperativeHandle` deps; both objects are rebuilt every render (Compiler-memoized but referentially unstable across renders that change any input). Acceptable today; flagged as F-02 for v0.1.1.

## Findings

### F-01 — Worker default uses CDN, contradicts description's "offline-friendly" preference

- **Severity:** ⚠️ High
- **Dimension:** verification (substrate decision drift)
- **Location:** [`src/registry/components/media/pdf-viewer/lib/worker-config.ts`](../../../../src/registry/components/media/pdf-viewer/lib/worker-config.ts)
- **Observed:** The plan locked `?url` import of `pdfjs-dist/build/pdf.worker.min.mjs` as the default worker source — chosen specifically for offline-friendliness. During implementation, Turbopack failed to resolve `new URL("pdfjs-dist/build/pdf.worker.min.mjs", import.meta.url)` (bare-specifier in `new URL()` not supported by Turbopack's resolver). Fallback shipped: load worker from `https://unpkg.com/pdfjs-dist@${version}/build/pdf.worker.min.mjs`. This contradicts description Q14 which committed to "no CDN dep at runtime."
- **Suggested fix:** Two paths:
  - **(a)** Investigate `?url` query syntax in Turbopack (separate from `new URL()` form) — may already work for transitive deps; declare ambient `*?url` module if needed.
  - **(b)** Ship a `postinstall` recipe in the consumer guide so consumers can copy the worker into their `public/` and pass `workerSrc="/pdf.worker.min.mjs"`. Document this as the recommended path; fall back to CDN only if neither (a) nor consumer setup applies.
  - **Bump target:** v0.2.0 (substrate change — worker resolution is foundational; not a patch-level fix).
- **Owner:** author

### F-02 — `actions` object rebuilt on every render; imperative handle deps unstable

- **Severity:** 🔸 Medium
- **Dimension:** architecture / composition
- **Location:** [`src/registry/components/media/pdf-viewer/pdf-viewer.tsx`](../../../../src/registry/components/media/pdf-viewer/pdf-viewer.tsx) — the `actions = useMemo(() => ({...}), [...])` block
- **Observed:** The `actions` object's `useMemo` deps include `pageTracker`, `zoom`, and `printer` — each of which returns a new object on every parent render under React Compiler memoization assumptions. Net result: `actions` rebuilds nearly every render, which causes `useImperativeHandle`'s identity check to also run, which causes the consumer's `ref.current` to point at a fresh object on each parent render. Not a correctness bug — values are still right — but it defeats the consumer's `useEffect(() => {...}, [ref.current])` patterns if they rely on stable identity.
- **Suggested fix:** Either (a) wrap each hook's returned methods in stable `useCallback` (with refs for state-reading deps) so `pageTracker`, `zoom`, `printer` return identity-stable methods, OR (b) hoist `actions` construction into `useImperativeHandle`'s factory so only one identity is exposed externally, even if the internal object churns. Option (b) is smaller.
  - **Bump target:** v0.1.1 (non-breaking fix).
- **Owner:** author

### F-03 — Print path lacks "currently printing" UX; user could click again

- **Severity:** 🔹 Low
- **Dimension:** architecture / robustness
- **Location:** [`src/registry/components/media/pdf-viewer/parts/pdf-action-menu.tsx`](../../../../src/registry/components/media/pdf-viewer/parts/pdf-action-menu.tsx) — the print Button
- **Observed:** `usePdfPrint` correctly tracks `isPrinting` via state; the print() function early-returns if already printing. But the print button doesn't reflect this state — no spinner, no disabled flag, no "Preparing print…" microcopy. For a 100-page PDF, the canvas-render loop can take 10+ seconds, during which the user might assume nothing is happening and click again (no-op, but confusing).
- **Suggested fix:** Read `isPrinting` from context (would need to expose it from `usePdfPrint` → context value), then disable the Print button + show an inline progress indicator while it's true. Document the per-page render time in the guide as "expect ~100ms/page during print."
  - **Bump target:** v0.1.1.
- **Owner:** author

## Post-deploy smoke addendum (same session, 2026-05-10)

After commit `1da6485` pushed and Vercel redeployed, the smoke harness was extended to include `pdf-viewer` and run against `https://ilinxa-proui.vercel.app/r/pdf-viewer.json`. The harness's automated install path timed out at 120s (heavy deps; expected for react-pdf + pdfjs-dist), so install was reproduced manually via `pnpm dlx shadcn@4.6.0 add @ilinxa/pdf-viewer --yes --overwrite`. Install **succeeded** (all 28 base files + 8 shadcn primitives copied). Consumer-side `pnpm tsc --noEmit` then surfaced **3 type errors** that the local producer tsc had not — exactly the F-cross-11 brittleness signal the harness is designed to catch:

1. **`pdfjs-dist` version skew** — registry.json's bare-name `pdfjs-dist` declaration caused consumer's pnpm to resolve to `5.7.284` (latest), while react-pdf transitively pulled `5.4.296`. Two versions installed; type imports resolved against one but react-pdf's `<Document onLoadSuccess>` was typed against the other → incompatible-types error.
2. **`TooltipProvider delayDuration` not portable** — local producer's tooltip (`radix-ui`) accepts `delayDuration`, but consumer's shadcn-installed tooltip uses `@base-ui/react/tooltip` which expects `delay`. Passing `delayDuration={300}` from `<PdfToolbar>` failed type-check on the consumer side.
3. **Cascading type errors** in `pdf-viewer.tsx` from #1 propagated into the `Document onLoadSuccess` callback signature.

**Resolution → v0.1.1:**

- Removed `pdfjs-dist` from `meta.ts dependencies.npm` and from `registry.json` `pdf-viewer` item's `dependencies` array. Consumer now gets pdfjs-dist only via react-pdf's transitive — single version, no skew. Type imports `from "pdfjs-dist"` still resolve via pnpm's default hoisting of transitive deps.
- Removed `delayDuration={300}` from `PdfToolbar`'s `<TooltipProvider>` — uses default (0) which works across all shadcn tooltip versions (radix-ui and base-ui).

Re-verified locally: tsc clean, lint clean, validate-meta-deps 38/38 clean. Component bumped to v0.1.1 and re-pushed. Smoke harness re-run pending second Vercel redeploy.

**F-04 status:** closed by v0.1.1; surfaced 2 real issues that local checks missed — exactly the value F-cross-11 path-b was added to deliver.

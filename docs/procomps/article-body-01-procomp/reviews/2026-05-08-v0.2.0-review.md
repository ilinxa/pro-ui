## 1. Executive summary

`article-body-01` v0.2 is the library's first Plate-based component (per the `project_richtext_substrate.md` memory: "Plate adopted as WYSIWYG substrate"). Two exports from one folder: **`<ArticleBodyEditor>`** (`'use client'`, ~165KB gzip, fixed top toolbar + selection-anchored floating toolbar via `@floating-ui/react`) and **`<ArticleBodyViewer>`** (server-renderable via `platejs/static`, ~32KB gzip). Both consume the same Plate `Value` JSON shape; storage is JSON, not HTML round-trips. v0.2 added 4 features (lowlight 15-language code highlighting, image resize + caption, floating toolbar, async server-only `serializeArticleBodyToHtml` escape hatch) — all wired correctly and explicitly documented in both the description (`> **v0.2 changelog**`) and guide (`## What's new in v0.2`).

**Headline finding:** `meta.ts.dependencies.npm["lucide-react"]: "^0.x"` — non-standard semver notation **and** wrong major version. Producer's actual `package.json` pin is `^1.11.0`. When a consumer installs, the `^0.x` range resolves to ancient lucide releases and the icons import would fail or pull a v0.x build with different exports. Same root pattern as rich-card's F-02 (session 2). This is now affirmed across two components — F-cross-07 (cross-component dep version drift) is real and systemic.

Two findings total — one High, one Medium. No Blockers. Smoke passed in session 1's run (13.9s). The component is otherwise clean: no `process.env` violations (description §6 #10 honored — only one guarded `navigator.platform` use for Cmd/Ctrl key descriptor), platform-aware key bindings, echo-guarded value-prop sync, robust floating-toolbar selection tracking with `autoUpdate` + scroll/resize listeners, server-only HTML export with configurable strip options, comprehensive Plate plugin set wired in `plugins/editor-kit.ts` + `plugins/viewer-kit.ts`.

**Recommended next step:** v0.2.1 patch fixing F-01 (`lucide-react: "^1.11.0"` precise pin) + F-02 (`usage.tsx` import-path drift). Roll into the same fix-up cycle as kanban v0.2.1 + rich-card v0.4.1 + flow-canvas-01 v0.1.1 since all four share the meta-deps + import-path patterns.

## 2. Strengths

- **Editor + Viewer split honors the Plate convention.** Two exports from one folder (`ArticleBodyEditor` + `ArticleBodyViewer`) — both consume same `ArticleBodyValue` JSON. Editor is `'use client'`; viewer is server-renderable. This is the canonical pattern documented in `project_richtext_substrate.md` and worth promoting to `docs/component-guide.md` as the standard for editor-substrate components.
- **JSON-as-storage discipline.** Description and guide both call out: don't round-trip through HTML for storage; keep JSON canonical. The `serializeArticleBodyToHtml(value)` escape hatch is explicitly **async** and **server-only** ("don't call from client components") — surface design matches Plate's storage guidance verbatim.
- **Echo-guarded value-prop sync.** [`article-body-01.tsx:51-57`](../../../src/registry/components/data/article-body-01/article-body-01.tsx) uses `lastSyncedValueRef` to detect external `value` prop changes without re-emitting `onChange`. This is the same pattern markdown-editor uses (per STATUS.md context) and prevents the infinite-loop footgun if a consumer calls `setValue` from inside their `onChange`. Worth promoting to a documented convention.
- **Platform-aware Cmd/Ctrl key descriptor with SSR guard.** [`article-body-01.tsx:16-19`](../../../src/registry/components/data/article-body-01/article-body-01.tsx): `typeof navigator !== "undefined" && /Mac/i.test(navigator.platform)` correctly guards SSR (no `navigator` on server) and uses the standard `navigator.platform` heuristic for Mac vs other. The visible `<kbd>Cmd+S</kbd>` / `<kbd>Ctrl+S</kbd>` rendered in the footer matches the actual key handler. Small detail but high polish.
- **Floating toolbar handles real edge cases.** [`parts/floating-toolbar.tsx`](../../../src/registry/components/data/article-body-01/parts/floating-toolbar.tsx):
  - `getSelectionRect()` handles SSR (`typeof window === "undefined"`), no-selection (`!sel`), zero-range (`rangeCount === 0`), and zero-rect (`rect.width === 0 && rect.height === 0`) cases.
  - `autoUpdate` from `@floating-ui/react` keeps the toolbar attached to selection during scroll/resize.
  - `onMouseDown preventDefault` on toolbar buttons prevents selection loss when clicking — the canonical floating-toolbar UX trick.
  - `role="toolbar"` + `aria-label="Selection actions"` for screen readers.
  - eslint-disable comments are **justified inline** (`react-hooks/set-state-in-effect` is genuinely required because the source of truth is a DOM-level Selection that React doesn't track; `react-hooks/refs` is a false positive on `setFloating` which is a stable ref-callback). Worth showcasing as the right way to bypass lint with documentation.
- **`process.env` discipline honored.** Zero occurrences in shipped code. Description §6 #10 satisfied here, in contrast to flow-canvas-01's F-03 (4 occurrences). Worth flagging as a positive comparison when discussing F-cross-08.
- **v0.2 changelog explicitly documented in BOTH description and guide.** Description has `> **v0.2 changelog (2026-05-02):**` block with the 4 features + bundle size estimates. Guide has `## What's new in v0.2 (2026-05-02)` section + the kanban-style "Closed in v0.2 ✅" callout. This is the gold-standard pattern for major-feature bumps — kanban v0.2 followed it; flow-canvas-01 v0.1 didn't have a v0.x section yet (initial ship); rich-card v0.4 had a per-version plan file. Three different conventions across four components; would benefit from a single documented standard in `docs/component-guide.md`.
- **Smoke install passes** — `pnpm dlx shadcn@latest add @ilinxa/article-body-01 --yes` succeeded in session 1's run at 13.9s. All declared deps resolved, including the 11-package Plate dep family.

## 3. Findings

### 3.1 Blockers (🚫)

### 3.2 High (⚠️)

#### F-01 — `meta.ts.dependencies.npm["lucide-react"]: "^0.x"` is wrong (non-standard semver + wrong major)

- **Severity:** ⚠️ High
- **Dimension:** 5 (Dependencies)
- **Location:** [`meta.ts:68`](../../../src/registry/components/data/article-body-01/meta.ts) — declares `"lucide-react": "^0.x"`
- **Observed:** Two issues with the same line:
  1. **Non-standard semver notation.** `^0.x` is not canonical semver. Standard ranges are `^0.1.0`, `^1.0.0`, etc. The `^N.x` form may not resolve cleanly across all package managers.
  2. **Wrong major version.** Producer's `package.json` has `"lucide-react": "^1.11.0"`. There is no `lucide-react@0.x` released this year — the package's modern major is 1.x. When a consumer installs article-body-01, the shadcn CLI writes the `^0.x` constraint into the consumer's package.json, which would resolve to an old lucide build with different exports than the icons used (`Bold`, `Code`, `Highlighter`, etc. — these are 1.x export names; 0.x had different naming).
- **Why it matters:** This is the second confirmation of F-cross-07 (cross-component dep version drift). Rich-card's F-02 had the same pattern with `@dnd-kit/sortable: "^11.x"` (wrong major + non-standard format). Affirmed across **2 of 4** Tier 1 components reviewed so far → systemic. A consumer installing article-body-01 + rich-card together gets two conflicting deps via shadcn-add (one ancient lucide, one nonexistent dnd-kit-sortable v11).
- **Suggested fix:** Replace line 68 with `"lucide-react": "^1.11.0"` (matching producer's pin). Re-run `pnpm registry:build` so the artifact at `public/r/article-body-01.json` reflects the correction. Cross-component coherence: F-cross-07's recommended `pnpm validate:meta-deps` lint check would catch both this and rich-card's similar error.

### 3.3 Medium (🔸)

#### F-02 — `usage.tsx` uses producer-side import path (F-cross-06 pattern, now 3/3)

- **Severity:** 🔸 Medium
- **Dimension:** 4 (Demo + usage)
- **Location:** [`usage.tsx:29-32`](../../../src/registry/components/data/article-body-01/usage.tsx) and line 62
- **Observed:** Both code blocks import from `@/registry/components/data/article-body-01` (the producer's docs-site path). After consumer install via `pnpm dlx shadcn add @ilinxa/article-body-01`, the install lands at `components/article-body-01/`, so the consumer's correct import is `@/components/article-body-01` (or via the `@ilinxa/article-body-01` namespace alias).
- **Why it matters:** Same pattern as kanban-board-01 F-04 (session 1) and rich-card F-04 (session 2). **Now affirmed across 3 of 4 Tier 1 components** — F-cross-06 (sweep-wide usage import-path drift) is conclusively systemic. A consumer copy-pasting either example into their app immediately hits an unresolved-module error.
- **Suggested fix:** Standardize on consumer-side path. Lines 29-32:
  ```ts
  import {
    ArticleBodyEditor,
    ArticleBodyViewer,
    type ArticleBodyValue,
  } from "@/components/article-body-01";
  ```
  Line 62 similarly. Recommend escalating F-cross-06 to a sweep-wide single-commit fix at the session 7 mid-sweep checkpoint — fixing all 36 components' usage.tsx import paths together is more efficient than per-component.

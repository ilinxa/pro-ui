## Findings (max 5)

#### F-01 — Forward-only wire-format (old reader can't read new envelope)
- **Severity:** 🔹 Low
- **Dimension:** Public API / wire-format
- **Location:** `lib/clipboard.ts`
- **Observed:** A v0.4.0 copy (envelope) pasted into a *pre-v0.4.0* rich-card consumer would fail (the old reader expects a bare item). Reverse direction works (back-compat read). Clipboard is ephemeral (no cross-version persistence), so practical risk is minimal.
- **Suggested fix:** Note the unified envelope in the guide; no code change needed (back-compat covers the realistic upgrade path).

## Addendum — deep adversarial re-review (2026-06-23)

A second adversarial pass caught two real correctness issues (both pre-existing latent bugs that the cross-surface unification amplifies); **both fixed before close**:

- **🔸 was-Medium → FIXED:** the 3 copy sites (imperative handle, keyboard hook, action-menu) serialized `node.item` directly, but `item.children` goes **stale after in-session edits** (`childNodes` is the source of truth; `denormalize` reconstructs — see normalize.ts:324). So copying an edited subtree carried stale children into the cross-surface clipboard. Now all three copy `denormalize(node)`.
- **🔸 was-Medium → FIXED:** the legacy bare-item back-compat fence (`looksLikeTodoItem`) only checked `id` + `name`, so foreign `{id, name}` JSON false-positived as a task. Tightened to require `status` (string) + `active` (boolean) + `setAt` (string) — matching json-io's original `validate` contract. Verified: legacy full items still parse, partial/foreign JSON now rejected (5/5 fence test).
- **Verified safe (no change):** no double-reassign — rich-card's reducer `add-child` re-ids (reassignIds) so paste correctly does NOT re-id at the call site. Removed json-io clipboard fns have zero remaining importers; `serialize`/`fromDataTransfer` still used by card.tsx.

Re-gated after fixes: **tsc 0 · lint 0 (touched) · meta-deps 57/57 · build 66/66**.

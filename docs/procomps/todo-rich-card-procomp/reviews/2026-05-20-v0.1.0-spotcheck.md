## Findings

#### F-01 — Hydration warning suppressed instead of avoided

- **Severity:** 🔸 Medium
- **Dimension:** Public API + Verification
- **Location:** `src/registry/components/data/todo-rich-card/parts/card.tsx:101` (`suppressHydrationWarning`)
- **Observed:** The original plan (§5.5 / §11) called for first-paint deferral via `useLayoutEffect` to avoid hydration mismatch when `now` isn't frozen. During implementation, the React Compiler's `react-hooks/incompatible-library` rule flagged the `setState`-in-`useEffect` pattern as cascading-render unsafe. We pivoted to `suppressHydrationWarning` on the article root, which mutes the warning but doesn't *avoid* the underlying server/client divergence on `style.borderColor`.
- **Suggested fix:** v0.2 — switch to a `useSyncExternalStore` pattern with a server snapshot that always returns `null` for `borderColor`, so server-rendered HTML omits the inline style entirely and only the client adds it after hydration. Avoids the mismatch without suppression. Documented in guide.md "SSR + the `now` prop" so consumers know to pass frozen `now` for clean SSR.

#### F-02 — Four post-sign-off deviations from signed-off description / plan

- **Severity:** 🔸 Medium (bumped from Low — scope expansion is now one of the items)
- **Dimension:** Public API + A11y + UX
- **Locations:**
  - `parts/card.tsx:142-146` — `aria-disabled` pivot
  - `lib/reducer.ts:13` + `todo-rich-card.tsx:34` — `createInitialState` signature
  - `parts/card.tsx:174-233` — `<Popover>` → `<Dialog>` for color override
  - `parts/card-header.tsx:25-44` + `parts/card.tsx:163-172` + reducer + state — **per-card collapsibility added**
- **Observed:** Four deviations from signed-off docs, two forced by lint/spec compliance, one forced by a UX race condition, one a user-requested scope addition during the live-demo session:
  1. **`aria-disabled` → `data-*` attributes** (lint-driven) — description.md §2 A11y + success-criterion #11, plan.md §3.2 + §14 all specify `aria-disabled={locked || !active}` on the `<article>` element. `jsx-a11y/role-supports-aria-props` flagged this — per WAI-ARIA spec, the `article` role doesn't support `aria-disabled`. Pivoted to `data-locked="true"` + `data-inactive="true"`. The visual + behavioral signal (opacity, saturate-50 classes, permission denials) is preserved.
  2. **`createInitialState(input)` signature** (lint-driven) — plan §3.3 said `createInitialState(input, props)`; the `_props` arg was unused and lint flagged it; removed. If v0.2 needs props-derived initial state, reintroduce loudly.
  3. **Color-override `<Popover>` → `<Dialog>`** (race-condition fix) — plan §6.4 + §9 listed Popover with a sr-only trigger anchored invisibly to the card. In live testing the popover opened-and-immediately-dismissed because the DropdownMenu's lingering click bubbled up and triggered the popover's outside-click handler. Pivoted to `<Dialog>` (modal overlay captures all clicks; no race). `popover` dropped from `meta.shadcn` and `registry.json`'s `registryDependencies`. The dialog footer now carries a prominent **Auto** button (with Sparkles icon) that clears the override and hands control back to the time engine; the previous "Reset to time-engine" wording is gone.
  4. **Per-card collapsibility added** (scope expansion) — description doc did NOT list collapsibility in v0.1 scope (rich-card has it; we deferred for parity-tightness). User requested it during live demo. Added: `collapsedIds: ReadonlySet<string>` slice on `State`, `toggle-collapse` action on the reducer, `isCollapsed(id)` accessor on `TodoCardContextValue`, chevron button at the left of every card header (`<ChevronDown>` / `<ChevronRight>`), conditional render in `parts/card.tsx` that hides `<CardBody>` + nested children when collapsed. UI-only state — not stored in `TodoItem`, not affecting JSON I/O round-trip. Default expanded. No programmatic `setCollapsed` on the imperative handle yet (v0.2 candidate).
- **Suggested fix:** v0.1 ships as-is — all four deviations are correct under their respective forcing functions (lint, race condition, user ask). Description.md + plan.md stay frozen. When v0.2 bumps the gate docs (e.g. for F-01's SSR strategy switch), fold ALL four updates into the next-revision loud-deviation banner: aria-disabled, createInitialState signature, color-override Dialog, and the scope-expanding collapsibility. The guide.md, usage.tsx, and meta.ts are updated to match current behavior in this same session.

#### F-03 — No `removeItem(id)` on imperative handle

- **Severity:** 🔹 Low
- **Dimension:** Public API
- **Location:** `src/registry/components/data/todo-rich-card/todo-rich-card.tsx:175` (useImperativeHandle block)
- **Observed:** Handle exposes `setBorderColor`, `toggleActive`, `setLocked`, `openEdit`, `closeEdit`, `copy`, `paste`, `focusItem`, `markClean` — but no programmatic remove. The only path to remove an item is via the action menu (which fires `onItemRemoved`). Symmetric with `addItem` (also missing programmatically, but at least `paste()` covers add-as-child).
- **Suggested fix:** v0.1.1 patch — add `removeItem(id): void` + `addChild(parentId, item, index?): void` to the handle. Additive, non-breaking.

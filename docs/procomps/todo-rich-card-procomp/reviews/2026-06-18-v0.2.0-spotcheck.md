#### F-04 — R10 Select narrowing broke Base UI consumers (smoke-caught) ✅ FIXED in v0.2.1

- **Severity:** ⚠️ High (regression) → **Fixed**
- **Dimension:** Public API / cross-backend portability
- **Location:** `parts/edit-inline.tsx` + `parts/edit-popup.tsx` (status `Select` `onValueChange`)
- **Observed:** The report's R10 (drop the `(v: string | null)` annotation as "dead defensive code") was analyzed against the **producer's Radix** Select (`onValueChange: (string) => void`). But a **Base UI** consumer's Select passes `(value: string | null, eventDetails)`, so the narrowed `(next: string)` handler is not assignable → 2 consumer-tsc errors. **The report's R10 recommendation was wrong** — the nullable annotation is load-bearing cross-backend, not dead.
- **Fix (v0.2.1):** restored `(v: string | null) => … ?? ""` with a comment documenting the Radix/Base-UI divergence. Re-smoke from the live registry: clean.

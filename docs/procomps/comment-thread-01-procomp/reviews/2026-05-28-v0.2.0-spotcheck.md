### Rotating dimension

- **Chosen:** **Public API**
- **Why:** The v0.2.0 change adds three new public API entries (`Comment.edited`, `CommentMenuItem.separatorBefore`, `CommentThreadLabels.editedSuffix`) — the latter two are reusable across host components (post-card-01 already consumes `separatorBefore` in its v0.3.0 moderator section). Correctness here means the shape extensions are strictly additive, the default behavior under absence is identical to v0.1, and the realtime-vs-first-paint semantics are consistent.
- **Verified?** yes
- **Notes:**
  - `Comment.edited` is optional + checked via `comment.edited ? ... : null` truthy check — `undefined` and `false` both suppress the suffix; only `true` renders. ✓
  - The realtime delta arm in `use-comment-state.ts:129` now flips `edited: true` alongside the content patch. Host that wanted the v0.1 behavior (content patch without marking edited) had no way to do so before — semantics shift here is a strict improvement (the v0.1 behavior was a bug: a realtime edit didn't surface as edited in the UI). No consumer is depending on the absence of the flag. ✓
  - `DEFAULT_COMMENT_THREAD_LABELS` extension keeps `Required<Omit<CommentThreadLabels, "formatRelativeTime">>` contract — TS would fail if any required key was missing from the merged defaults, so this is statically guaranteed. ✓
  - The `(edited)` suffix renders inside the same `<span>` as the timestamp, separated by a single space + a slightly muted inner `<span>` (`text-muted-foreground/80`) to visually de-emphasize. Matches FB/Twitter convention. Wrapping with `<>...</>` Fragment + leading space text node is verbose but TS-safe under strict whitespace rules — kept intentional vs. tucking the space into the conditional render.
  - `CommentMenuItem.separatorBefore` is documented as optional + cosmetic — comment-thread-01's own default `CommentKebab` does NOT currently consume it (it has no section concept). The flag is purely additive surface for host-grouped kebabs. The same pattern is already idiomatic in the codebase (`flow-canvas-01/types.ts:113` has the same flag on its `ContextMenuItem`).
  - Existing comment dummy data — only one comment now carries `edited: true`; all other dummy fixtures unchanged. Visual coverage in the docs `/components/comment-thread-01` page without regression risk on the existing demos. ✓

#### F-02 — `dispatch({ kind: "patch-content" })` does NOT flip `edited`

- **Severity:** 🔹 Low
- **Dimension:** Public API
- **Location:** `src/registry/components/data/comment-thread-01/hooks/use-comment-state.ts:111-116`
- **Observed:** Intentional asymmetry between two escape hatches that both patch comment content:
  - `subscribe-delta { kind: "edited" }` → flips `edited: true` (correct realtime semantics, per ILX-4)
  - `dispatch({ kind: "patch-content" })` → does NOT flip `edited: true` (host might patch content for non-edit reasons — refetch with corrected payload, sanitization, etc.)
- This asymmetry is correct UX but undocumented at the dispatch action's JSDoc.
- **Suggested fix:** Add a 2-line JSDoc note on `CommentLocalAction { kind: "patch-content" }` clarifying scope. Or extend the action with `{ kind: "patch-content"; commentId: string; content: string; markEdited?: boolean }` so the host can opt in. Lean toward (a) docs-only since (b) bloats the action union.
- **Owner:** Author.
- **Bump target:** v0.2.1 (docs only; no behavior change).

#### F-03 — Standalone `<CommentKebab>` doesn't currently consume `separatorBefore`

- **Severity:** 🔹 Low
- **Dimension:** Composition integrity (touched at the API boundary)
- **Location:** `src/registry/components/data/comment-thread-01/parts/comment-kebab.tsx:90-103`
- **Observed:** The default `<CommentKebab>` part (used by `<CommentNode>`) iterates items and renders them via `<DropdownMenuItem>` with no separator branching at all — it ignores both `separatorBefore` AND `destructive` boundaries. (Comment kebabs are short — usually Report + Delete — so the absence of a separator inside the row's own kebab is acceptable today.) The host-side post-card-01 `<PostHeader>` is the one that actually composes `separatorBefore` + destructive-boundary together.
- The flag is therefore "available to host kebabs" but "ignored by the built-in kebab" — slightly asymmetric. Documented behavior is consistent with that asymmetry (the v0.2.0 type JSDoc says "Used by host components that group items into visual sections"), but a strict reader might expect parity.
- **Suggested fix:** Optional v0.2.1 — bring CommentKebab's render in line with PostHeader's separator composition (cheap: ~6 lines). OR leave as-is and document the asymmetry explicitly. Either is acceptable; not a blocker for ILX-4 closure.
- **Owner:** Author.
- **Bump target:** v0.2.1 candidate (depends on whether a consumer reports it).

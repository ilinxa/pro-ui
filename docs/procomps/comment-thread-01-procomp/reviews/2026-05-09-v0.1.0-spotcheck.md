- **Why:** comment-thread-01 is recursive (tree-of-comments with `replies?: Comment[]` per node), supports realtime via Subscribe<CommentDelta>, and ships an optimistic-flow reducer that handles add/like/delete/swap-temp + page-append. Threads can grow unbounded; hot-feed comments at 100+/sec deltas is plausible. Reducer cost + render-cost-per-tree-node + memoization hygiene are the load-bearing concerns.
- **Verified?** yes (with 2 small Low findings)
- **Notes:**
  - **`CommentNode` IS wrapped in `React.memo`** (`parts/comment-node.tsx:297`). ✓ defensive against full-tree re-renders for NPM consumers without React Compiler. Strong perf hygiene.
  - **`comments.map((c) => <CommentNode ... />)`** at line 242 of root — every CommentNode receives a stable bundle of useCallback-wrapped handlers (handleLike / handleDelete / handleReport / openReply / cancelReply / submitReply). ✓ all callbacks `useCallback`-wrapped with proper deps.
  - **`useCommentState` reducer** + `Subscribe<CommentDelta>` channel. Reducer fires on every realtime delta. Reducer actions like `like-toggle` and `swap-temp` walk the tree by ID — O(N) per action. Acceptable for typical N (≤ 1000 comments per thread); flag at scale.
  - **`requestAnimationFrame` for focus restoration** (lines 196, 206) — appropriate use of rAF for post-render DOM operations after reply submission/cancel. ✓
  - **`crypto.randomUUID()` with fallback** to `Date.now() + Math.random()` (line 33-38) — handles older browsers + insecure contexts. ✓
  - **Pagination via `pageSize: 10` + `onLoadMore(page)`** — `dispatch({ kind: "append-page", comments: result })` appends to the tree. `hasMore` is derived from `result.length === pageSize`. See F-02 for the heuristic concern.
  - **`composerRef` + `composerEmptyState` slot** — composer mounts conditionally on `currentUser` presence. Not a perf concern but worth noting for tree-render hygiene.
  - **No virtualization** — see F-01.

## Findings (2)

### F-01 — No virtualization; deep / wide threads with N > 200 walk full DOM tree per render

- **Severity:** 🔹 Low
- **Dimension:** 8 — Performance
- **Location:** `src/registry/components/data/comment-thread-01/comment-thread-01.tsx:241-273` (the `comments.map((c) => <CommentNode ... />)` block)
- **Observed:** All comments render via `comments.map(c => <CommentNode ... />)` at the root level, with each CommentNode recursively rendering its own `replies` (depth-bounded by `maxDepth` default 2, then "view N replies" inline-expand). For a feed thread with 50 top-level comments × 4 replies each = 200 nodes, that's 200 CommentNode instances mounted simultaneously. With memo, they don't re-render on unrelated state changes — but they all stay mounted. For high-traffic posts with 1000+ comments paginated in via `onLoadMore`, the in-DOM tree grows monotonically. No virtualization (e.g., `react-virtual`) is wired, and there's no `meta.npm` dep on a virtualization library. Acceptable for v0.1's typical-feed use case; worth flagging at scale.
- **Suggested fix:** No v0.1.1 fix. Document the threshold in the guide ("v0.2 candidate: virtualize at N > 200 visible nodes") and note as a v0.2 roadmap item. If a real consumer hits deep-thread perf issues, wire `@tanstack/react-virtual` for the top-level list (replies stay un-virtualized — they're depth-bounded). The `dispatch({ kind: "append-page" })` + page boundaries naturally delineate virtualization windows.

### F-02 — `hasMore` heuristic relies on `result.length === pageSize` exactly

- **Severity:** 🔹 Low
- **Dimension:** 8 — Performance / API contract
- **Location:** `src/registry/components/data/comment-thread-01/comment-thread-01.tsx:96, 226`
- **Observed:** Two sites use the same heuristic:
  - Line 96 (initial state): `useState(initialComments.length === pageSize)`
  - Line 226 (after page load): `setHasMore(result.length === pageSize)`
  
  Edge cases:
  - Backend returns `< pageSize` on the last page (correct: `hasMore=false`). ✓
  - Backend returns `> pageSize` (misconfiguration; `hasMore=false`, but there ARE more) — wrong.
  - Backend returns exactly `pageSize` and there's nothing more to load (e.g., total=20, pageSize=10, page-2 returns 10 with no more pages — `hasMore=true` incorrectly).
  
  The heuristic works in 95% of cases but produces a UI bug ("Load more" button visible when there's nothing more) in the exact-page-boundary case. Backend would need to provide an explicit `hasMore` flag for accuracy. Minor.
- **Suggested fix:** Either (a) change `onLoadMore` signature to return `{ comments, hasMore }` (breaking, v0.2) so the host can be authoritative; OR (b) document the heuristic + edge case prominently in the guide. Option (b) is the v0.1.1 doc-only fix.

## Findings (2)

### F-01 — `text-white` on `from-primary via-primary/95 to-primary/80` gradient may violate the design-system white-on-lime mandate

- **Severity:** 🔸 Medium
- **Dimension:** 14 — Cross-component coherence (and Dim 6 — Design system)
- **Location:** `src/registry/components/marketing/page-hero-news-01/page-hero-news-01.tsx:91, 106` (`text-white text-4xl` on title; `text-white/80` on description)
- **Observed:** CLAUDE.md design mandate is explicit: *"Always paired with near-black `--primary-foreground` (lime is too bright for white text)."* page-hero-news-01 renders the title with `text-white` and description with `text-white/80` on a gradient that starts at full `--primary` (lime, oklch(0.80) light / oklch(0.86) dark). At the gradient's lighter end (`primary/80` opacity), the lime is even more transmissive — white text fades into the cool off-white page background. The gradient's darker end (full `primary`) is the explicit "lime is too bright for white text" failure mode the mandate calls out. The migration origin (kasder NewsHero used a different brand color where white worked); pro-ui's signal-lime needs different foreground treatment.
- **Suggested fix:** Two reasonable paths: (a) Replace `text-white` / `text-white/80` with `text-primary-foreground` / `text-primary-foreground/80` (the mandated near-black; matches the rest of the lime-paired pattern in the library — kanban-board-01 uses lime-with-near-black for active cards). (b) If a darker hero is preferred, define a custom gradient via `--page-hero-*` tokens (e.g., a darker brand color or a graphite gradient) so white text becomes legible. Option (a) is the smaller diff; option (b) is the higher-effort visual choice. Recommend (a) for v0.1.1.

### F-02 — Inline base64-encoded SVG noise pattern embedded in className

- **Severity:** 🔹 Low
- **Dimension:** 14 — Cross-component coherence (cleanliness)
- **Location:** `src/registry/components/marketing/page-hero-news-01/page-hero-news-01.tsx:61`
- **Observed:** The subtle noise overlay is rendered via a 132-character base64-encoded SVG embedded in a Tailwind `bg-[url('data:image/svg+xml;base64,...')]` arbitrary-value class. This works but is unusual in the library — most decorative SVG patterns either live in CSS (as utility classes via globals.css) or as separate `.svg` assets imported. The inline-base64 approach makes the className unscannable, complicates future style edits, and bloats the source file's diff width.
- **Suggested fix:** Either (a) extract to a CSS utility class `bg-hero-noise` defined in globals.css (cleanest; reusable across other hero-style components), OR (b) extract to a separate `parts/noise-pattern.svg` and reference it. Option (a) keeps the asset inline (no asset bundling complexity) and makes the className readable.

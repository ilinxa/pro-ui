## Findings (3)

### F-01 — `registry.json` declares `react-day-picker` + `date-fns` as direct deps; `meta.ts` does not

- **Severity:** 🔹 Low
- **Dimension:** 5 — Dependencies
- **Location:** `registry.json:504` vs `src/registry/components/forms/filter-bar-01/meta.ts:35-39`
- **Observed:** `registry.json` line 504 declares `"dependencies": ["lucide-react", "react-day-picker", "date-fns"]` (the npm deps shadcn-add will install in the consumer). `meta.ts.dependencies.npm` declares only `{ "lucide-react": "^1.11.0" }`. Both are correct *for their respective scopes* — `meta.ts` reflects only direct imports (per F-cross-07 closure: `react-day-picker` and `date-fns` come transitively via shadcn `Calendar`, so dropping them from meta was the right call); `registry.json` lists them so the consumer install gets a working calendar even if shadcn's preset transitive resolution is unreliable. But the divergence is undocumented anywhere — a future contributor reading the two files side-by-side will see what looks like drift.
- **Suggested fix:** Either (a) add a one-line comment in `meta.ts` near the `npm` field noting the transitive deps are declared in `registry.json` (preferred — keeps the audit-clean meta but signals the divergence), or (b) verify shadcn's `@/components/ui/calendar` already brings `react-day-picker` + `date-fns` into a fresh consumer install and drop them from `registry.json.dependencies` (test required). Option (a) is the lower-risk fix.

### F-02 — Guide should document the transitive dep surface for consumer expectations

- **Severity:** 🔹 Low
- **Dimension:** 1 — Procomp planning docs
- **Location:** `docs/procomps/filter-bar-01-procomp/filter-bar-01-procomp-guide.md`
- **Observed:** When a consumer runs `pnpm dlx shadcn add @ilinxa/filter-bar-01`, their `package.json` gains `react-day-picker` + `date-fns` (per `registry.json:504`). The procomp's guide doesn't mention this; consumers seeing two new deps in their lockfile will wonder where they came from. `meta.context` and `meta.features` mention the transitive chain at the registry-listing level, but not in the consumer-facing guide.
- **Suggested fix:** Add a 2-3 sentence paragraph to the guide's "Reference" section (or "Gotchas") stating that installing this procomp also adds `react-day-picker` + `date-fns` to the consumer's project as transitive deps of shadcn `Calendar`. Consumers who already have these (or a different version) should review the lockfile diff. No code change needed.

### F-03 — `useEffect` combined-onChange fires on initial mount even when no user interaction has occurred

- **Severity:** 🔹 Low
- **Dimension:** 1 — Procomp planning docs (plan-vs-code clarity)
- **Location:** `src/registry/components/forms/filter-bar-01/filter-bar-01.tsx:117-122`
- **Observed:** The combined `onChange` fires on mount with the initial values (deps include `search`, `category`, `dateRange`, `onChange` — all stable on mount, so the effect runs once and dispatches). Comment in source explicitly documents this: *"We intentionally fire on every change (including initial mount). Consumers wanting initial-render skip should track it themselves."* The plan documents the controlled-or-uncontrolled posture but does not discuss the mount-fire posture as a deliberate decision — a future reviewer or consumer reading only the plan/guide could miss it and hit the gotcha.
- **Suggested fix:** Add a short sentence to the guide's "Gotchas" or "API contracts" section: *"`onChange` fires on initial mount with the initial sub-control values. Skip via a ref guard if your handler triggers expensive downstream work."* The implementation itself is fine — this is a doc-completeness nit.

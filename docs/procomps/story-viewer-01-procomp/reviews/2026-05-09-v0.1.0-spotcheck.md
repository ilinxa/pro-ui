  - **Pause-preserve accumulator timer** (`hooks/use-story-progress.ts`) implements the documented kasder-bug fix correctly: snapshot `Date.now() - startTime` into `accumulatedRef.current` on pause; reset `startTimeRef` to `null`; resume re-takes Date.now() and adds prior accumulator.   - **`setInterval(50ms)` = 20Hz progress tick.** Each tick calls `setProgress(pct)` triggering re-render of `StoryViewer01Inner`. CSS `transition-[width]` smooths visually between ticks; the JS state itself doesn't aim for 60Hz. ✓ acceptable for the smoothness target.
  - **Keyboard-nav listener** (`hooks/use-story-keyboard-nav.ts`) attaches only on `isOpen=true`; refs-mirror pattern keeps the listener stable; teardown on close. No leaks. ✓
  - **`completedRef` guard** in the tick callback prevents double-invocation of `onComplete` when the interval fires on the exact 100% boundary. ✓
  - **ID-anchored cursor** (per meta) means insertions/removals don't desync; the `itemKey` resets the timer accumulator on every navigation. ✓
  - **`videoMetadataMs` state** is a per-id cache; `setVideoMetadataMs` early-bails when entry already exists. Bounded growth in practice. ✓
  - One real-world-rare clock concern + two minor perf footguns surfaced — see findings.

## Findings (3)

### F-01 — Progress tick uses `Date.now()` instead of `performance.now()` (theoretical clock-jump fragility)

- **Severity:** 🔹 Low
- **Dimension:** 8 — Performance / robustness
- **Location:** `src/registry/components/media/story-viewer-01/hooks/use-story-progress.ts:76, 87, 91`
- **Observed:** The accumulator math uses `Date.now()` for both the pause-snapshot (`accumulatedRef.current += Date.now() - startTimeRef.current`) and the per-tick elapsed calculation (`Date.now() - startedAt + accumulatedRef.current`). `Date.now()` is wall-clock and can jump forward/backward on NTP sync, DST transitions, or system time changes. `performance.now()` is monotonic and immune to all three; it's also slightly cheaper. For 5-second story durations the practical risk is negligible (the user would have to be unlucky enough to trigger an NTP correction mid-story), but the principled choice is `performance.now()`.
- **Suggested fix:** Replace both `Date.now()` calls with `performance.now()` (modern browsers; `globalThis.performance.now()` for SSR-safety paranoia, though the hook is `'use client'` so SSR doesn't apply). One-line change per call site.

### F-02 — 20 Hz progress tick re-renders `StoryViewer01Inner` and all children that aren't memoized

- **Severity:** 🔹 Low
- **Dimension:** 8 — Performance
- **Location:** `src/registry/components/media/story-viewer-01/story-viewer-01.tsx:142-148` (hook call) + the parts/ children
- **Observed:** `useStoryProgress` returns the `progress` state, which is read inside `StoryViewer01Inner` and passed to `<ProgressBars>`. Every 50ms tick triggers `setProgress(pct)` → re-render of `StoryViewer01Inner` → re-render of all 6 part children unless React Compiler / explicit `memo` halts the cascade. In-repo, React Compiler covers it; for NPM consumers without the Compiler, every tick walks 6 part subtrees (TapZones / NavArrows / ViewerHeader / ItemView re-render unnecessarily — they don't depend on `progress`).
- **Suggested fix:** Either (a) wrap each part in `React.memo` defensively, OR (b) move the progress-state ownership down into a sub-component that wraps `<ProgressBars>` only. Option (b) is the cleaner state-isolation pattern; option (a) is a smaller-diff defensive fix. Document which the project prefers.

### F-03 — `useImperativeHandle` deps assume callback-stability of `useStoryViewerState` returns

- **Severity:** 🔹 Low
- **Dimension:** 8 — Performance / API contract
- **Location:** `src/registry/components/media/story-viewer-01/story-viewer-01.tsx:96-111`
- **Observed:** `useImperativeHandle(ref, () => ({...}), [goToStoryIndex, goToItemIndex, setPaused, reset, dispatch])` — the deps array references 5 functions returned by `useStoryViewerState`. If any of these aren't `useCallback`-stable across renders (e.g., they're inline `(idx) => ...` declarations re-created each call), the imperative handle re-creates every render. Hosts that key effects on `ref.current` would re-run them every tick. Without reading `use-story-viewer-state.ts`, I can't confirm — but this is a recurring pattern-trap.
- **Suggested fix:** Verify each function from `useStoryViewerState` is wrapped in `useCallback`. If any aren't, wrap them. The deps array on `useImperativeHandle` is correct *given* stable callbacks; the assumption is the load-bearing piece.

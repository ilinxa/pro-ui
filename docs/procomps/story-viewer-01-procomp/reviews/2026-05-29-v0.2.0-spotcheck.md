## Findings (max 5)

Severity legend: 🚫 Blocker · ⚠️ High · 🔸 Medium · 🔹 Low.

#### F-01 — `triggerReply` content arg is documented but stubbed

- **Severity:** 🔹 Low
- **Dimension:** Public API
- **Location:** `src/registry/components/media/story-viewer-01/story-viewer-01.tsx` (useImperativeHandle block) + types.ts L489
- **Observed:** The `triggerReply(content?: string)` parameter is accepted by the handle, but the underlying `CommentComposerHandle` (comment-thread-01 v0.2.1) only exposes `focus()` — there is no `setValue()` yet. The current implementation focuses the composer and silently drops `content`. The JSDoc says "optionally pre-fill content" which over-promises.
- **Suggested fix:** Either (a) extend `CommentComposerHandle` upstream in comment-thread-01 v0.2.2 to add `setValue()` and wire it here, OR (b) tighten the JSDoc to clarify pre-fill is a no-op until comment-thread-01 v0.2.2 lands. **Bump target:** v0.2.1 (b) immediately; v0.2.2 (a) as the proper fix.

#### F-02 — Owner overlay viewers cast `as any` at LikersStrip boundary

- **Severity:** 🔹 Low
- **Dimension:** Public API
- **Location:** `src/registry/components/media/story-viewer-01/parts/owner-overlay.tsx`
- **Observed:** `ViewerListItem` and engagement-bar-01's `EngagementLikerProfile` are structurally compatible but distinct types post inline-copy (F-S1 Bug 3). The LikersStrip composition uses an `as any` cast at the boundary.
- **Suggested fix:** Documented cast — fine for v0.2.0. **Bump target:** consider a discrete `LikersAdapterProps` helper type in v0.3.0 to avoid the cast (low priority).

#### F-03 — Engagement overlay coexists with reply composer but does not visually offset

- **Severity:** 🔸 Medium
- **Dimension:** Public API
- **Location:** `src/registry/components/media/story-viewer-01/parts/engagement-overlay.tsx` (`bottom-24`) + `parts/reply-composer.tsx`
- **Observed:** Stacked engagement-bar sits at `bottom-24` regardless of whether the reply composer (which sits at the bottom edge) is present. On very short viewports the bookmark / kebab buttons may overlap the composer.
- **Suggested fix:** Either add a `bottom-32` shift when composer is mounted, or document the constraint. **Bump target:** v0.2.1 patch; visual-test in a dev browser.

#### F-04 — Long-press pause may double-fire with tap-pause on slow taps

- **Severity:** 🔹 Low
- **Dimension:** Public API
- **Location:** `src/registry/components/media/story-viewer-01/hooks/use-long-press-pause.ts`
- **Observed:** If a user taps the middle tap-zone with a hold barely under 200ms, the tap-zone's middle-pause fires AND the long-press timer just barely doesn't, but on a slower tap (250ms total, lifted before threshold) the timer is cleared (no pause from long-press) and the tap-zone's click STILL fires the middle-pause — single net pause as expected. The race is correct but undocumented.
- **Suggested fix:** Add a guide note: "If both gestures fire, only one pause net-effect — TapZones fires on click; long-press fires on hold-past-threshold." **Bump target:** v0.2.1 docs.

#### F-05 — `engagementOverlayMounted` gate hides kebab fallback when host disables BOTH engagement AND sets viewerMode

- **Severity:** 🔹 Low
- **Dimension:** Public API
- **Location:** `src/registry/components/media/story-viewer-01/story-viewer-01.tsx` (header fallback condition)
- **Observed:** The header fallback kebab is gated on `viewerMode && disableEngagement && kebabItems.length > 0`. If host opts into `viewerMode="viewer"` but disables engagement entirely AND supplies no kebabActions/handlers, the kebab disappears with no visual fallback — intended, but the guide doesn't explicitly call this out.
- **Suggested fix:** Guide note: "Setting `disableEngagement` without wiring kebab handlers + matching permissions leaves the kebab affordance absent — this is intentional; pair `disableEngagement` with explicit kebab wiring." **Bump target:** v0.2.1 docs.

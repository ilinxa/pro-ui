### Rotating dimension

- **Chosen:** Dim 8 ‚Äî Performance
- **Why:** people-grid-01 renders an N-image grid at avatar-display sizes (48px / 64px / 96px). Image-loading footprint, memoization hygiene, and re-render cost across N PersonCard instances are the load-bearing perf concerns. Use cases include conference speakers + team pages + course instructors ‚Äî N can scale to 100+.
- **Verified?** yes (with 2 small Low findings)
- **Notes:**
  - **`<img loading="lazy">`** ‚úì on each person image (line 59 of person-card.tsx). Native browser lazy-loading; off-screen images don't fetch until scroll-near.
  - **`PeopleGrid01` IS `React.memo` wrapped** ‚úì (line 97 of root). Stable item-array refs prevent re-renders.
  - **`PersonCard` is NOT `React.memo` wrapped** ‚Äî see F-01.
  - **Module-scope `COLUMNS_MAP`** (line 13 of root) ‚úì ‚Äî class-string lookup table allocated once, not per-render.
  - **`labels` `useMemo`** ‚úì (line 39 of root) ‚Äî defaults merge re-runs only when consumer's `labels` prop changes.
  - **`getInitials(item.name)`** is a pure function called per-render ‚Äî cheap O(name length); no memoization needed.
  - **CSS Grid layout** (`grid-cols-N`) ‚úì ‚Äî browser native, no JS layout calculation.
  - **`motion-safe:group-hover:text-primary transition-colors`** ‚úì ‚Äî CSS-only hover, `prefers-reduced-motion` respected.
  - **No `srcset`/`sizes`** on `<img>` ‚Äî see F-02.

## Findings (2)

### F-01 ‚Äî `PersonCard` not wrapped in `React.memo`; large grids re-render every card on any state change

- **Severity:** üîπ Low
- **Dimension:** 8 ‚Äî Performance
- **Location:** `src/registry/components/data/people-grid-01/parts/person-card.tsx:29` (`export function PersonCard(...)`) ‚Äî no memo wrap
- **Observed:** `PeopleGrid01` (root) is `React.memo` wrapped (line 97) so unrelated parent re-renders don't cascade in. But `PersonCard` (the child) is not memo-wrapped, so when `PeopleGrid01` itself does re-render (e.g., consumer updates `labels` or `columns`), every PersonCard in the grid re-renders unconditionally. With React Compiler in-repo this auto-memoizes; for NPM consumers without the compiler, large grids (50+) walk every card on every state change. Same shape as session 9 story-viewer-01 F-02 (recurring perf-polish pattern). Not a defect for typical 10-30 speaker grids; matters at scale.
- **Suggested fix:** Wrap defensively: `export const PersonCard = memo(PersonCardInner);` ‚Äî same pattern used in `comment-thread-01/parts/comment-node.tsx:297`. One-line change. NPM consumers benefit immediately; in-repo behavior unchanged.

### F-02 ‚Äî `<img>` lacks `srcset`/`sizes`; full-resolution images load regardless of display size

- **Severity:** üîπ Low
- **Dimension:** 8 ‚Äî Performance (bandwidth / network)
- **Location:** `src/registry/components/data/people-grid-01/parts/person-card.tsx:55-61`
- **Observed:** `<img src={item.image} ... loading="lazy" className="w-full h-full object-cover" />` ‚Äî single `src`, no `srcset` for responsive resolution, no `sizes` for the browser's resolution-picking heuristic. Avatar display sizes are 48√ó48px (sm) / 64√ó64px (md) / 96√ó96px (lg) ‚Äî but source images are typically 200√ó200 to 500√ó500 (headshot photography conventions). Browser downloads full-resolution, scales down. For 100 people √ó ~80KB per avatar = ~8MB wasted bandwidth on a single page. With Retina displays the math gets worse (browsers prefer 2√ó / 3√ó density when available, but full-resolution is the only option offered).
- **Suggested fix:** Either (a) extend `PeopleGridItem.image` from `string` to `string | { src: string; srcSet?: string; sizes?: string }` ‚Äî backwards compatible (string still works as src); v0.1.x can ship as-is with consumer responsibility. OR (b) document in the guide that consumers should pre-size their avatar images (server-side resize / CDN auto-format) for typical 96√ó96 display. Option (b) is the v0.1.x doc fix; (a) is a v0.2 API expansion.

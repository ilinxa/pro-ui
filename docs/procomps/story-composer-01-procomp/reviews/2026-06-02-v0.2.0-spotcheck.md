## Findings (max 5)

#### F-01 — `editorBackground` prop accepted but not forwarded

- **Severity:** 🔸 Medium
- **Dimension:** Public API (semantic completeness)
- **Location:** [story-composer-01.tsx:80](../../../../src/registry/components/media/story-composer-01/story-composer-01.tsx#L80) (prop accepted) and the `<MediaEditor01 .../>` mount below (not passed through)
- **Observed:** v0.1.5 had `editorBackground?: string` (default `#000`) which threaded into `ComposerShell` to set the modal backdrop / canvas behind. The wrapper destructures the prop but doesn't forward it to MediaEditor01 (which has no equivalent prop). Net effect: consumers passing a non-default `editorBackground` see no change in v0.2.0; the canvas always uses MediaEditor01's `#000` default.
- **Suggested fix:** Either (a) expose a `background?: string` prop on MediaEditor01 + thread through (cleanest), or (b) wrap MediaEditor01 in a `<div style={{ background: editorBackground }}>` in the wrapper (works for inline; less clean for dialog). (a) is preferred; tag v0.2.1.

#### F-02 — `renderPublishingOverlay` slot is a no-op (CLOSED in-review)

- **Severity:** 🔹 Low
- **Dimension:** Public API (semantic completeness)
- **Location:** [story-composer-01.tsx:78](../../../../src/registry/components/media/story-composer-01/story-composer-01.tsx#L78) — prop accepted but not forwarded; default `<PublishingProgressOverlay>` always rendered during upload
- **Observed:** v0.1.5 `renderPublishingOverlay` let consumers override the default publishing UI with their own component. Wrapper destructured but didn't honor the slot. Documented as a known gap in C16 commit message.
- **Fix applied (in-review):** Added `renderPublishingOverlay` to the wrapper's props destructure and wrapped the publishing-overlay JSX as `renderPublishingOverlay ? renderPublishingOverlay({ progress: uploadProgress, mode }) : <PublishingProgressOverlay .../>`. The v0.1.5 ctx signature `(ctx: { progress: number; mode: ComposerMode }) => ReactNode` is preserved.

#### F-03 — `setMode` workaround via `loadState` in `toComposerCtx`

- **Severity:** 🔸 Medium
- **Dimension:** Public API (handle completeness — cross-cuts media-editor-01)
- **Location:** [story-composer-01.tsx:200-205](../../../../src/registry/components/media/story-composer-01/story-composer-01.tsx#L200) (`editorRef.current?.loadState({ ...getState(), mode: m })`)
- **Observed:** `ComposerCtx.setMode` is part of the v0.1.5 slot-ctx contract. MediaEditor01's handle has no public `setMode` (description §7 lock = 22 explicit methods). Wrapper round-trips through `loadState` to set mode externally. Functional but hacky; if a future MediaEditorState shape adds a new field, the spread loses defaults.
- **Suggested fix:** Cross-references media-editor-01 v0.1.0 review F-02. Resolution path: add `setMode(mode: ComposerMode | null) => void` to `MediaEditor01Handle` in v0.3.0; story-composer wrapper drops the workaround. Same v0.3 cohort as F-02 there.

#### F-04 — Story-composer-01 planning docs not updated for v0.2.0

- **Severity:** 🔹 Low (doc currency)
- **Dimension:** Procomp planning docs
- **Location:** [docs/procomps/story-composer-01-procomp/](../) — description.md / plan.md / guide.md all reference the v0.1.5 self-contained architecture
- **Observed:** All three planning docs still describe story-composer-01 as a self-contained Konva-based editor. The v0.2.0 wrapper architecture (consumes @ilinxa/media-editor-01) is documented in meta.ts + the media-editor-01 procomp's plan/decision files, but NOT in story-composer-01's own planning trio. A consumer reading `story-composer-01-procomp-description.md` sees stale architecture context.
- **Suggested fix:** Add a v0.2.0 amendment section to each of description.md / plan.md / guide.md pointing at the wrapper refactor + cross-referencing media-editor-01-procomp. Quick doc sweep; tag v0.2.1 docs.

#### F-05 — Visual regression vs v0.1.5 not verified

- **Dimension:** Verification
- **Location:** Plan §C18 deliverable
- **Observed:** Same as media-editor-01 v0.1.0 F-04. Plan §C18 calls for walking every v0.1.5 demo tab post-wrapper to confirm identical behavior. Deferred to user post-push.

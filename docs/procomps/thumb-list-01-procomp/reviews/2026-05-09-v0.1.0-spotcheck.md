## Findings (2)

### F-01 — Demo tabs lack explanatory text describing what each variant shows

- **Severity:** 🔹 Low
- **Dimension:** 4 — Demo + usage
- **Location:** `src/registry/components/data/thumb-list-01/demo.tsx` (5 sub-tabs, none have explanatory `<p>`)
- **Observed:** Other component demos in the library (share-bar-01, article-meta-01, page-hero-news-01) include an explanatory `<p className="mt-3 text-xs text-muted-foreground">...</p>` after each tab's demo, telling the consumer what to look for ("Click Copy link — icon flips to a check for 2s.", "Hover the byline...", etc.). thumb-list-01's demo just shows the component without commentary. Consumers comparing tabs have to infer the difference from the visuals alone. Per Dim 4 review-guide: *"The demo is what the docs site shows; it's the consumer's first impression."* Annotation lifts comprehension without changing the component.
- **Suggested fix:** Add a `<p className="mt-3 text-xs text-muted-foreground">...</p>` after each `<ThumbList01>` instance, briefly describing what the tab demonstrates. Examples:
  - Default: "Standard sidebar list — frame on, default header icon."
  - No frame: "Borderless variant for inline placement (e.g., inside another card)."
  - Custom meta: "Use the `renderMeta` slot for dates / badges / scores."
  - Empty state: "Custom `emptyText` via the labels prop."
  - i18n: "Custom `headerIcon` + Turkish heading via labels."
  
  ~5-line change per tab. Lifts demo polish to match the rest of the library.

### F-02 — `headerIcon={null}` and custom `linkComponent` not demonstrated despite being public API

- **Severity:** 🔹 Low
- **Dimension:** 4 — Demo + usage (variant coverage)
- **Location:** `src/registry/components/data/thumb-list-01/demo.tsx` + `meta.ts:14, 16` (declared features)
- **Observed:** Two public API surfaces have no demo coverage:
  1. **`headerIcon={null}`** — `meta.features` says *"pass null to hide"* the header icon. None of the 5 tabs demonstrates this — all have either default BookOpen or custom Bookmark icon.
  2. **Custom `linkComponent`** — polymorphic root for NextLink / RemixLink / etc. None of the 5 tabs demonstrates this.
  
  Consumers reading the demo learn the visible features; missing demos mean consumers might not realize these knobs exist. Common scenario: a consumer wants a heading-less sidebar list and resorts to passing an empty string (which fails differently than `null`).
- **Suggested fix:** Either (a) add a 6th sub-tab "Hidden icon + custom link" demonstrating `headerIcon={null}` + a custom linkComponent (e.g., a wrapper that logs clicks), OR (b) extend an existing tab to incorporate one or both. Option (a) is the more discoverable fix.

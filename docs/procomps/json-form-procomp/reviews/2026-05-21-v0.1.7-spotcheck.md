### Rotating dimension

- **Chosen:** Public API
- **Why:** v0.1.7 adds **5 new public surface items** at once: `field.dependsOn` typed prop, `useJsonFormFieldValue<T>(name)` + `useJsonFormFieldsValue<T>(names)` hooks, `defineFieldRenderer<TValue, TConfig>(...)` factory, `JsonFormDevtools` component + `JsonFormDevtoolsProps` type. That's the largest single-bump API expansion since v0.1.0 first ship. Audit-worthy.
- **Verified?** yes
- **Notes:** All five additions match the plan exactly:
  - `dependsOn` declared on `FieldDefinition` as `ReadonlyArray<string>` per Q-P1 locked semantics (`undefined` / `[]` / non-empty). v0.1.7 ships typed-only — runtime behavior is the unchanged full-bag watch. JSDoc explicitly states "v0.1.7 — typed metadata + schema-lint validation only. The runtime watch-gating that honors this flag ships in v0.2.0."
  - `useJsonFormFieldValue<T = unknown>(name): T` + `useJsonFormFieldsValue<T extends Record<string, unknown> = Record<string, unknown>>(names): T` — symmetric typed assertions per Q-P7 lock. Both wrap RHF's `useWatch` against the active context. Multi-field variant rehydrates via `setByPath` (matches the v0.1.6 use-conditional pattern).
  - `defineFieldRenderer<TValue, TConfig>({ displayName?, impl })` — returns plain `FieldRenderer` (Q-P5 lock); attaches `displayName` as non-enumerable property for devtools rendering. Type-narrowing only.
  - `<JsonFormDevtools>` — loader stub + lazy body chunk via `React.lazy()`; prod no-op via `process.env.NODE_ENV` guard + `force` override (Q-P4 lock). Floating-by-default, `inline` prop opt-out (Q-P3 lock). Four tabs (schema / values / conditionals / errors), `Ctrl+Shift+J` shortcut overridable via `shortcut` prop.
  - `compileStructural` + `injectStrings` split landed but kept **internal** (not in `index.ts` exports per F-06 fix in the plan). `compileSchema` wrapper preserved for any internal callsite that wants the one-shot. `useJsonForm` now memoizes the two stages independently.

## Findings

#### F-01 — Smoke harness path-b not exercised against v0.1.7

- **Severity:** 🔸 Medium
- **Dimension:** Verification
- **Location:** `e:/tmp/ilinxa-smoke-consumer/` (consumer-side workflow)
- **Observed:** v0.1.6 spotcheck ran path-b smoke and surfaced the F-S1 cross-procomp `/types` bug that drove the v0.1.5 patch. v0.1.7 adds 4 new files including a lazy-import boundary (`json-form-devtools.tsx` → `import("./json-form-devtools-body")`) — shadcn 4.6.0's path rewriter may or may not preserve dynamic-import paths through the producer → consumer install. Until smoked, this is unverified.
- **Suggested fix:** Run `pnpm dlx shadcn@4.6.0 add @ilinxa/json-form @ilinxa/json-form-fixtures` from the smoke harness against deployed Vercel artifacts post-push, then `pnpm tsc --noEmit`. If the lazy-import path rewrites cleanly, no action; if not, switch the dynamic-import to a relative `./json-form-devtools-body` (already what we wrote — but the rewriter may inject `@/` aliases regardless).
- **Owner:** Author. **Bump target:** `v0.1.7` (pre-push smoke before deploy) OR `v0.1.8` (post-push, fast-follow).

#### F-02 — `dependsOn` runtime behavior deferred to v0.2.0 may confuse consumers

- **Severity:** 🔹 Low
- **Dimension:** Public API (documentation/expectation mismatch)
- **Location:** [`types.ts`](../../../../src/registry/components/forms/json-form/types.ts) `FieldDefinition.dependsOn` JSDoc
- **Observed:** The JSDoc opens with bold "**v0.1.7 — typed metadata + schema-lint validation only. The runtime watch-gating that honors this flag ships in v0.2.0.**" but then describes the FULL semantics including the three-state behavior. A consumer skimming the doc may set `dependsOn` and expect perf wins in v0.1.7. The `validateSchemaDev` warn for dangling refs DOES fire, so the consumer gets SOMETHING out of it.
- **Suggested fix:** Add an explicit "Until v0.2.0, setting this flag has only one observable effect: the schema-lint warns if any name in the list doesn't match a `schema.fields[].name`." to the JSDoc opener. Already mostly conveyed via the bold prefix — consider this a minor polish for v0.1.8.
- **Owner:** Author. **Bump target:** `v0.1.8` (doc-only).

#### F-03 — Devtools `<form>` parent assumption

- **Severity:** 🔹 Low
- **Dimension:** Robustness
- **Location:** [`parts/json-form-devtools-body.tsx`](../../../../src/registry/components/forms/json-form/parts/json-form-devtools-body.tsx)
- **Observed:** The panel reads `useJsonFormContext()` which throws if rendered outside a `<JsonFormProvider>`. For `inline` mode this is correct — devtools placed inside a form work fine. But a consumer who places `<JsonFormDevtools />` OUTSIDE the form (as a sibling of `<JsonForm>`) won't have the context. The Provider is mounted INSIDE `<JsonForm>`, so external placement fails.
- **Suggested fix:** Document the constraint in usage.tsx ("must be a descendant of `<JsonForm>` OR `<JsonFormProvider>`"). Optionally: render a soft warning panel (instead of throwing) when the context is missing — easier to debug for consumers. Defer the soft-warning to v0.1.8.
- **Owner:** Author. **Bump target:** `v0.1.8`.

#### F-04 — `defineFieldRenderer<TValue, TConfig>` type narrowing is shallow

- **Severity:** 🔹 Low
- **Dimension:** Public API / typing
- **Location:** [`lib/define-field-renderer.ts`](../../../../src/registry/components/forms/json-form/lib/define-field-renderer.ts)
- **Observed:** `NarrowedRendererArgs<TValue, TConfig>` types `value: TValue` and `field.config?: { [key: string]: TConfig }`. The `[key: string]` index signature means accessing `field.config?.color?.palette` yields `TConfig | undefined` — but the key lookup itself isn't type-narrowed to specific config buckets (e.g., `code` / `date` / `rating` from the existing `FieldConfig`). A consumer reading `field.config?.color` gets `TConfig | undefined` regardless of whether `'color'` is a registered key.
- **Suggested fix:** Defer. Tighter narrowing would require a third generic for the config key, which adds boilerplate for marginal type-safety gain. Document the current shape in usage.tsx; revisit if real consumers hit ergonomic blockers.
- **Owner:** Author. **Bump target:** `v0.2.x` (only if real-consumer feedback drives it).

#### F-05 — Devtools `prettyReplacer` doesn't handle `Date` or `BigInt`

- **Severity:** 🔹 Low
- **Dimension:** Robustness
- **Location:** [`parts/json-form-devtools-body.tsx`](../../../../src/registry/components/forms/json-form/parts/json-form-devtools-body.tsx) `prettyReplacer`
- **Observed:** The JSON.stringify replacer converts functions / Map / Set, but `Date` instances stringify to ISO via JSON.stringify default (acceptable), and `BigInt` throws. A consumer schema with `BigInt` defaults or values would crash the devtools Schema/Values tab.
- **Suggested fix:** Add `if (typeof value === "bigint") return value.toString() + "n";` to the replacer. ~1 LOC.
- **Owner:** Author. **Bump target:** `v0.1.8`.

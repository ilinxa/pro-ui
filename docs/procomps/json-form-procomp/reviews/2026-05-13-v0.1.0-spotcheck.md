### Rotating dimension

- **Chosen:** Public API
- **Why:** This is a high-surface-area procomp — 38-line top-level `JsonFormProps`, generic `<TValues>`, 12-method imperative handle, 6 standalone parts, 11-operator Condition DSL, 24 field types. Surface this big is exactly the place where post-ship API regret lands hardest. The plan-stage P-01 through P-14 review already paid down a lot of that risk, but the implementation pass surfaces shape mismatches the plan can't.
- **Verified?** yes
- **Notes:** API surface matches the plan with two additive refinements: `JsonFormContextValue` gained a `fieldRegistry` member (needed for `<JsonFormField>` to resolve renderers without re-merging in every standalone-part call site), and `JsonFormContextValue` + `useJsonFormContext` + `JsonFormProvider` generics now have `extends Record<string, any> = Record<string, any>` (needed to satisfy RHF's `FieldValues` constraint — the foundation `types.ts` had already locked the same idiom on `JsonFormProps<TValues>`). Both are additive and consumer-invisible. `'form'` was removed from shadcn deps because the implementation uses `Slot.Root` + RHF `Controller` directly rather than wrapping the shadcn `Form` primitive — same end-state, less indirection.

## Findings

#### F-01 — Smoke harness path-b not exercised

- **Severity:** 🔸 Medium
- **Dimension:** Verification
- **Location:** `e:/tmp/ilinxa-smoke-consumer/` (consumer-side workflow)
- **Observed:** This first-ship review verified local tsc + lint + build + registry:build clean, but did not run `pnpm dlx shadcn@latest add @ilinxa/json-form` from the smoke consumer + a follow-up `pnpm tsc --noEmit`. The F-cross-11 lesson is that local-side cleanliness ≠ consumer-side cleanliness — the path-b consumer-tsc catches barrel mis-exports, missing target rewrites, and accidentally-shipped docs files.
- **Suggested fix:** Run the path-b smoke harness against `@ilinxa/json-form` (against the Vercel-deployed `https://ilinxa-proui.vercel.app/r/json-form.json` once master pushes) before bumping to `v0.1.1`. If it surfaces issues, fix them and re-review.
- **Owner:** Author. **Bump target:** `v0.1.0` (pre-push smoke before deploy) OR `v0.1.1` (post-push, fast-follow).

#### F-02 — `requireWhen` doesn't trigger Zod validation until first submit

- **Severity:** 🔹 Low
- **Dimension:** Public API
- **Location:** [`src/registry/components/forms/json-form/lib/schema-compiler.ts:321-348`](../../../../src/registry/components/forms/json-form/lib/schema-compiler.ts#L321-L348)
- **Observed:** `requiredWhen` is enforced via an outer `.superRefine()` on the ZodObject, but the resolver only runs when RHF triggers validation — which is `onTouched` by default. A field that becomes conditionally-required mid-form will only surface the error on next blur or submit. This matches the plan's "Field gains the required asterisk + Zod re-validates on next submit attempt" edge-case treatment, but is mildly surprising for consumers expecting "true reactive validation."
- **Suggested fix:** Document the timing more explicitly in the FAQ. Optionally, in v0.1.x: when a field's `requiredWhen` toggles true, internally `form.trigger(name)` to surface the error immediately if the field already failed touched-validation. Costs ~10 LOC + a watcher.
- **Owner:** Author. **Bump target:** `v0.1.x` (FAQ in next patch; reactive trigger in next minor).

#### F-03 — `'use client'` directive missing from `lib/default-registry.ts`-internal lazy import

- **Severity:** 🔹 Low
- **Dimension:** Public API / portability
- **Location:** [`src/registry/components/forms/json-form/lib/default-registry.ts:1`](../../../../src/registry/components/forms/json-form/lib/default-registry.ts#L1)
- **Observed:** File starts with `"use client"` (good), but the registry shape — a plain object of function references — doesn't strictly require it. Including the directive is safe but pulls the file into the client bundle even when only a server consumer imports the types. This is the equivalent of code-block's "lib stays pure .ts" rule. Minor.
- **Suggested fix:** No action in v0.1.0 — the registry imports the `parts/*` modules which are all `'use client'` anyway, so the directive is informationally correct. Revisit if the registry-vs-types split sharpens in v0.2.
- **Owner:** Author. **Bump target:** v0.2.0 (only if split refactor lands).

#### F-04 — `JsonFormProvider` headless example in `demo.tsx` hardcodes a `strings` literal

- **Severity:** 🔹 Low
- **Dimension:** Public API (demo as consumer reference)
- **Location:** [`src/registry/components/forms/json-form/demo.tsx:225-262`](../../../../src/registry/components/forms/json-form/demo.tsx#L225-L262)
- **Observed:** The "fully-headless" demo subsection passes a verbose hand-written `strings` object instead of using `defaultJsonFormStrings`. Consumers copying this pattern would unnecessarily duplicate the dictionary. Cosmetic — both produce the same UI.
- **Suggested fix:** Swap the literal for `...defaultJsonFormStrings` spread + overrides only. Saves ~20 lines of demo.tsx and teaches the canonical pattern. Trivial follow-up.
- **Owner:** Author. **Bump target:** `v0.1.1`.

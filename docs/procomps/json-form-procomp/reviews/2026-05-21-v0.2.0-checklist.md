## Findings

### F-01 — `dependsOn` stable-identity guidance missing — 🔸 Medium

**Location:** [`types.ts`](../../../../src/registry/components/forms/json-form/types.ts) `FieldDefinition.dependsOn` JSDoc.

**Description:** The JSDoc explains the three semantic states but doesn't note that the array reference should be stable (module-scope const or `useMemo`-cached). An unstable reference causes the underlying `useWatch({ control, name: dependsOn })` to re-subscribe on every render, which would defeat the perf win and could in theory cause subscription churn warnings from RHF.

**Evidence:** Today's audit of `parts/field-wrapper.tsx` after the v0.2.0 change — `dependsOn` is forwarded directly into `useWatch`'s `name` prop without intermediate memoization. RHF's internal handling reconciles by content equality at the subscriber level, so the practical perf impact of an unstable array is small (RHF de-dupes subscription targets), but the JSDoc should still note the convention.

**Suggested fix (v0.2.1):** Add to the JSDoc on `FieldDefinition.dependsOn`:

```
* The array reference should be stable across renders — declare it at module
* scope or `useMemo`-cache it. RHF de-dupes subscription targets by content
* so the practical penalty is small, but stable refs avoid wasted reconciles.
```

**Owner:** ilinxa.
**Bump target:** v0.2.1 (patch, JSDoc-only).

### F-02 — `getValues()` snapshot freshness in snapshot mode — 🔹 Low

**Location:** [`parts/field-wrapper.tsx`](../../../../src/registry/components/forms/json-form/parts/field-wrapper.tsx) lines ~70–80.

**Description:** In snapshot mode, `allValues = getValues()` is read at render time. The field's `Controller` triggers a re-render when its OWN value changes, so per-field snapshots are fresh on every keystroke into that field. But the snapshot DOES NOT reflect changes to OTHER fields between renders — which is the entire point of snapshot mode. A consumer who reads `allValues.foo` from a snapshot-mode renderer that happens to be a built-in `text` will see a stale value if `foo` changed elsewhere. This is documented behavior (the whitelist exists because we audited the built-in renderers don't read `allValues`), but a consumer-registered renderer at the `text` slot correctly opts out (identity check), so the practical surface is narrow.

**Evidence:** Audit of all 25 built-in renderers (recorded in v0.1.7 plan and re-confirmed today) — none read `args.allValues` directly. `computed`'s renderer destructures only `{ field, value, onChange, disabled, ariaProps }`; the values-reading happens inside `useComputed()`'s own narrow `useWatch`, which is independent.

**Suggested fix (defer; document only):** No code change. The behavior is correct by design; the JSDoc on `FieldRendererArgs.allValues` could note "in snapshot mode (built-in default renderers + `dependsOn: []`), this is a render-time snapshot; subscribe via `useJsonFormFieldValue` or `useJsonFormFieldsValue` for live values from a custom renderer."

**Owner:** ilinxa.
**Bump target:** v0.2.1 (patch, JSDoc-only).

### F-03 — Default-registry whitelist drift mitigation pending — ⚠️ High (deferred to v0.2.0+1)

**Location:** [`lib/default-registry.ts`](../../../../src/registry/components/forms/json-form/lib/default-registry.ts) `BUILTIN_RENDERER_TYPES_SKIPPING_ALL_VALUES`.

**Description:** If a future built-in renderer adds `args.allValues` access but the author forgets to remove it from the whitelist, that built-in's renders will silently miss values updates. The v0.2.0 plan flagged this as the highest-severity risk in the register.

**Suggested fix:** A `validate-meta-deps`-style lint (or a node script chained into `pnpm vercel-build`) that grep-scans each whitelisted renderer's source file for `args.allValues` / `args\.allValues` access and surfaces a tracker error if a whitelisted renderer reads it.

**Owner:** ilinxa.
**Bump target:** v0.2.1 or v0.3.0 — scoped to a follow-up. Until then: hand-audit at each renderer-change PR.

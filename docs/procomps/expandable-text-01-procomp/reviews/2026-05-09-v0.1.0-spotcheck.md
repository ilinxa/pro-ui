## Findings (2)

### F-01 — Cross-folder import pattern confirms post-card-01 F-01 as a library-wide latent failure mode

- **Severity:** 🔸 Medium
- **Dimension:** 14 — Cross-component coherence (and Dim 9 — Registry distribution)
- **Location:** post-card-01.tsx:14-20 + comment-thread-01/parts/comment-node.tsx:12 (cross-folder importers) + expandable-text-01/index.ts (barrel re-exports `useLineClampDetect` from `hooks/use-line-clamp-detect.ts`)
- **Observed:** Per session 10's investigation of post-card-01 F-01, shadcn's path-rewrite logic transforms producer-side `@/registry/components/data/<slug>` (folder) → consumer-side `@/components/<slug>/<slug>` (filename-direct, bypassing `index.ts` barrel). Today this works for expandable-text-01's component import because:
  - post-card-01 imports `ExpandableText01` from `@/registry/components/data/expandable-text-01` → rewrites to `@/components/expandable-text-01/expandable-text-01` → resolves to the component (it IS in `<slug>.tsx`). ✓
  - Same for comment-thread-01's CommentNode import.
  
  **But the brittleness is real and visible:** `useLineClampDetect` lives in `hooks/use-line-clamp-detect.ts` and is re-exported via `index.ts`. If a future consumer (e.g., `article-body-01` per `meta.related`, or a new component) imports it cross-folder via `@/registry/components/data/expandable-text-01` (folder), shadcn rewrites to `@/components/expandable-text-01/expandable-text-01` (filename-direct) — and `<slug>.tsx` does NOT export `useLineClampDetect`. Result: producer-side `tsc` passes (resolves through index.ts barrel); consumer-side install ships broken; smoke harness passes (only checks `shadcn add` succeeds, not consumer tsc).
  
  This **confirms post-card-01 F-01 (s10) as a systemic library-wide pattern, not a post-card-only concern.** 2 of 16 reviewed Tier 2 components (post-card-01, expandable-text-01) demonstrate the brittleness shape. Worth escalating to F-cross-NN at sweep close (s13).
- **Suggested fix:** This is no longer a per-component fix — it's a library-wide concern. Three paths (same as s10 F-01):
  1. **Document the constraint** in `docs/component-guide.md`: cross-folder consumers MUST import from `<slug>.tsx` directly (not from sub-folders re-exported via barrel). All public-helper symbols intended for cross-folder consumption must live in `<slug>.tsx`. (Cheapest. Locks the convention.)
  2. **Extend the smoke harness** with consumer-side `pnpm tsc --noEmit` after install. (Catches the brittleness at smoke time.)
  3. **Realign cross-folder import paths** to consumer-side style (`@/components/<slug>`). (Biggest change, but most robust.)
  
  Recommendation: tackle at sweep close (s13) as a F-cross-NN escalation per the audit-systematic-scope memory. 2 confirmed shapes is enough signal.

- **Severity:** 🔹 Low
- **Dimension:** 14 — Cross-component coherence (related-list honesty)

Optionally include `meta.consumedBy` as a separate field if the project wants forward-vs-actual distinction (no current procomp uses this; defer the structural change).

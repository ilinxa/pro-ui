## Findings

### F-01 — F-cross-13 carrier `delayDuration={150}` dropped from `<TooltipProvider>` pre-ship

- **Severity:** 🔹 Low (pre-empted)
- **Dimension:** Robustness — primitive divergence
- **Location:** `src/registry/components/marketing/pricing-table-01/pricing-table-01.tsx:93`
- **Observed:** The original implementation rendered `<TooltipProvider delayDuration={150}>`. `delayDuration` is the Radix prop name; shadcn@4.6.0 installs Base UI tooltip whose equivalent prop is `delay`. Producer-side tsc passes because producer ships Radix-based `src/components/ui/tooltip.tsx`, but consumer-side install would have surfaced this as either a tsc error or a silently-dropped prop. Same F-cross-13 carrier pattern surfaced + fixed in `rich-card-in-flow@v0.2.0` (B4 commit `5684bc5`) and `todo-rich-card@v0.1.1`.
- **Suggested fix:** **Applied pre-ship.** Dropped the prop entirely with an inline comment documenting the rationale. Default delay (~700ms Radix / ~600ms Base UI) is acceptable for the pricing-table tooltip use case (feature-row hints). If consumers want a snappier delay, they wrap their own `<TooltipProvider delay={NN}>` at app-root level.

### F-02 — `<TooltipProvider>` is internal-scoped — consumer tree may end up with nested providers

- **Severity:** 🔹 Low
- **Dimension:** Robustness
- **Location:** `src/registry/components/marketing/pricing-table-01/pricing-table-01.tsx:93-159`
- **Observed:** The component wraps its body in `<TooltipProvider>` so it works out-of-the-box without a consumer-supplied provider. If the consumer's app shell already mounts a `<TooltipProvider>` (common in shadcn apps via the root layout), the pricing-table will nest a second provider inside it. Nested providers are a no-op in Radix and Base UI (no visible bug), but the tree is slightly wider than necessary.
- **Suggested fix:** Acceptable for v0.1.0 — out-of-the-box-works trumps tree-width pedantry. v0.2 candidate: detect whether a TooltipProvider is in scope via a sentinel context and conditionally skip the inner provider. Track as v0.2 candidate if a real consumer asks.

### F-03 — Dev-mode tier-count warning fires once per render

- **Severity:** 🔹 Low
- **Dimension:** Robustness
- **Location:** `src/registry/components/marketing/pricing-table-01/pricing-table-01.tsx:75-88`
- **Observed:** The `useEffect` warning block lists `[tiers]` as the dep, so the warn fires whenever the consumer changes the `tiers` array reference. Memoized consumers fire once; consumers passing inline `tiers={[ ... ]}` arrays will fire on every parent render. Console noise rather than a real bug.
- **Suggested fix:** Acceptable for v0.1.0 — `console.warn` is dev-only (`process.env.NODE_ENV` check). If a real consumer reports the noise, dedupe via a `useRef<string>(JSON.stringify(...))` guard. v0.2 candidate.

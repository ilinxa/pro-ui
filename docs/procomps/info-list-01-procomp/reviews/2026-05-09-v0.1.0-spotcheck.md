## Findings (2)

### F-01 — `dummy-data.tsx`'s `MapLinkAction` is a fake-interactive `<span>` (hover styling but no role / onClick / keyboard handlers)

- **Severity:** 🔹 Low
- **Dimension:** 7 — Accessibility (and Dim 4 — Demo + usage; the demo demonstrates the bad pattern)
- **Location:** `src/registry/components/data/info-list-01/dummy-data.tsx:12-22` (English + Turkish variants)
- **Observed:** Both `MapLinkAction` and `MapLinkActionTr` are JSX `<span>` elements styled with `cursor-pointer text-primary hover:underline` — visually they look like clickable links — but they have NO `onClick` handler, NO `role="button"` / `role="link"`, NO `tabIndex` or keyboard handler. SR users get no announcement that this is interactive; keyboard users can't reach it; sighted users would click it expecting navigation but nothing happens. Per Dim 7 review-guide bad signal: *"`<div onClick>` instead of `<button onClick>`"* — same shape (visual interactivity without semantic interactivity). The dummy data ships in `info-list-01-fixtures` (per registry pattern), so consumers installing the fixtures inherit this bad example. Even more importantly, consumers reading the demo at `/components/info-list-01` are likely to copy this `action` slot pattern verbatim into their own code, propagating the fake-interactive shape.
- **Suggested fix:** Replace the `<span>` with a real `<button type="button">` or `<a href="#">` (placeholder href is acceptable in dummy data). E.g.:
  ```tsx
  const MapLinkAction = (
    <button type="button" className="inline-flex items-center gap-1 text-sm text-primary mt-1 hover:underline">
      View on map <ExternalLink aria-hidden="true" className="w-3 h-3" />
    </button>
  );
  ```
  Plus mirror in `MapLinkActionTr`. Demo will continue to compile + render identically, but consumers copying the pattern get correct semantics. ~6-line edit total.

### F-02 — `compact` variant silently drops `secondary` and `action` fields

- **Severity:** 🔹 Low
- **Dimension:** 7 — Accessibility (information loss for SR users) / Dim 2 — Public API (silent contract)
- **Location:** `src/registry/components/data/info-list-01/parts/info-row.tsx:25-52` (compact variant only renders `primary`)
- **Observed:** Per `meta.features` *"Compact variant intentionally ignores secondary + action (documented limitation)"*, the design choice is locked. Implementation correctly drops both fields in compact mode (line 25-52). But there's no dev-warn — a consumer passing `compact` items with `secondary: "423 registered"` gets the data silently dropped. Sighted users see only the primary; SR users miss the secondary entirely. The consumer's intent (provide more context) goes unfulfilled without indication. Common in copy-paste scenarios (consumer iterates the same items array across two sections, one comfortable + one compact).
- **Suggested fix:** Add a dev-only `console.warn` in non-production builds: when an item has `secondary` or `action` set AND `variant === "compact"`, warn: `"[info-list-01] item '${id}' has secondary/action but compact variant ignores them. Use 'comfortable' variant to render."` Production builds strip the warning per F-cross-08 closure. Bundlers tree-shake the dead code.

# flow-canvas-01 — pre-v0.2 baseline measurements

> **Status:** baseline numbers for `flow-canvas-01@0.1.3` (master tip pre-v0.2-perf-work). All subsequent tier-measurement files compare against this.
> **Date:** 2026-05-14
> **Protocol:** [2026-05-14-measurement-protocol.md](2026-05-14-measurement-protocol.md)
> **Driving description:** [flow-canvas-01-v0.2.0-perf-description.md](../flow-canvas-01-v0.2.0-perf-description.md)
> **Tooling:** dev-server stress page at `/sandbox/flow-stress`, FPS overlay (eyeballing — NOT cited in success criteria per protocol §9)
> **Measurement window:** continuous pan-drag for several seconds, FPS range captured from overlay (rolling 1-second average)

## Headline findings

1. **The `custom-json` built-in fallback renderer is the perf bottleneck**, not node count. At N=200 the light fixture (custom-json) drops to 7-15 FPS; at the same N the heavy fixture (synthetic) renders at 50-60 FPS. **5-10× difference** in favor of the more visually complex renderer. **This is a bug.**
2. **Viewport culling (`onlyRenderVisibleElements`) is transformative at scale.** At N=5000 with heavy fixture: 5-6 FPS (off) → 60-70 FPS (on). **~12× lift.** At small N (100-500) the lever's effect is marginal because there's not enough off-viewport work to skip.
3. **Perf flattens once you're past the viewport cap.** N=200 heavy with culling ≈ N=5000 heavy with culling (both ~65 FPS). Once culling is on, the bottleneck becomes "viewport-visible node count," not "total N."
4. **xyflow + the heavy renderer + culling clearly does 5,000 nodes smooth.** Probably more. This raises the question of whether the description doc's "Tier 4 sibling procomp at >2k" threshold should move much higher.

| N | Fixture | Off → On lift | Notes |
|---:|---|---|---|
| 100 | light | 70-110 → 40-90 (lower avg, smoother feel) | Toggle hurts top-end FPS at small N but stabilizes the curve |

- Below ~200 nodes: marginal or neutral.

- 5000 nodes: order-of-magnitude. This is where the lever earns its keep.

## Implications for v0.2 planning

### Re-prioritization vs. the description's tier ladder

The description's Tier-1 plausibility claim was "300-500 nodes smooth (light); 200-300 smooth (heavy)." The data says:

- **Light** is unusable at N=200 even with the lever on. Tier 1's lever cannot save it; the custom-json renderer needs to be fixed first. **Light should not be a benchmark fixture for tier success criteria until custom-json is fixed.**
- **Heavy** with the lever on is smooth at N=5000 already, without any Tier 1, 2, or 3 work. Tier-1's plausibility for heavy was severely conservative.

### A new P0 finding: custom-json renderer perf bug

The built-in `custom-json` fallback ships with every consumer install. Every user who hasn't registered a custom renderer for their `__type` gets the bug. This is **shipped-component bug territory**, not tier-ladder territory. Plausible candidates for the cause:

1. Per-node `useState(expanded)` and a per-node `onClick` arrow function — the latter creates a new function identity every render. But the component is `memo`'d, so this shouldn't matter at the React level unless `data` prop ref is changing per tick.
2. The `cn(...)` calls with `expanded && "shadow-sm"` and `expanded && "border-b border-border"` are cheap.
3. The `min-w-45 max-w-90` Tailwind classes — need to verify these are valid Tailwind v4 utilities (project uses Tailwind v4.2.x without a config file; arbitrary numeric values should resolve via the default spacing scale).
4. Possible interaction with xyflow's node-measurement loop. xyflow measures every node on mount to compute layout; if the custom-json node has expensive layout (CSS classes resolving to many style rules), this could compound.
5. Something specific to how `<NodeShell>` + `<CustomJsonNode>` compose — a memo barrier that's leaking.

**Next action:** read `custom-json-node.tsx` carefully with a perf-bug hypothesis in mind, possibly compare against the synthetic `heavy-stress` renderer to identify what's different in the React reconciliation path.

### Suggested v0.2 path adjustment

1. **First**: investigate + fix the `custom-json` bug. Likely lands as **v0.1.4 patch** (bug fix, not a perf-tier feature). Re-measure light fixture against the same baseline matrix.
2. **Then**: Tier 1 default flip ships as v0.2.0 with the confidence that the renderer-side bug is no longer dominating.
3. **Then**: re-evaluate Tier 2/3 scope based on post-Tier-1 numbers.

### Sibling procomp threshold

The description doc said: *"if a consumer's use case needs >2,000 simultaneous interactive nodes, point them at the sibling procomp."* The data shows the heavy renderer with viewport culling on already does **5,000 nodes at 60-70 FPS** with no Tier 2 or Tier 3 work applied. The sibling-procomp threshold should likely move to **>10,000 or >20,000**, OR be reframed as **"when you genuinely need zero DOM per node"** rather than a node count.

This is a description-doc update worth doing as part of the v0.2 plan-stage work, NOT a separate fix now — the description doc is signed off, changes after sign-off should be loud and intentional.

## Patch attempt log

### 2026-05-14 — `<details>`/`<summary>` rewrite — REVERTED (wrong hypothesis)

**Hypothesis:** per-node `useState(expanded)` + `<CustomJsonNode>` memo-wrapper were the bottleneck.

**Result:** vis-off regressed 4-5× at N=100; vis-on improved 2-3× at N=200/500. **Tradeoff, not a fix.** Real root cause was elsewhere.

**Reverted to v0.1.3** before discovering the actual bug.

### 2026-05-14 — Missing port handles in `CustomJsonNode` — SHIPPED v0.1.4 ✅

**Root cause:** the `CustomJsonNode` component **never rendered `<Handle>` elements**, even when `data.ports` declared ports. The fixture data has `ports: [{id: "in"}, {id: "out"}]` and edges reference `s0:out → s1:in`, but the DOM had no matching handles. Every render frame, xyflow's edge layer called `getEdgePosition` for every edge, failed because the handles didn't exist in the DOM, and logged a warning. At N=200 × ~2 edges × 60 fps ≈ **~24,000 failed lookups + console warnings per second** — that's the CPU cost masquerading as a perf bug.

**Discovery:** user spotted it visually (nodes had no port circles in the rendered canvas) + flagged the flood of `[React Flow]: Couldn't create edge for source handle id: "out"` console warnings during pan.

**Fix:** added 4 `<PortsAt>` calls (one per side: left / right / top / bottom) to `CustomJsonNodeImpl`, plus a `relative` class on the outer wrapper so xyflow's absolute-positioned handles anchor correctly. Total diff: ~6 lines added, no behavior change to non-port-bearing data.

**Re-measurement (same hardware, same protocol):**

| N | Vis off — before → AFTER | Vis on — before → AFTER |
|---:|---|---|
| 100 | 70-110 (laggy) → **155** | 40-90 → **160** |
| 200 | **7-15 (very laggy)** → **90-100** | **7-15 (very laggy)** → **100-115** |
| 500 | **3-4 (dead)** → **50-70** | **3-4 (dead)** → **80-90** |
| 1000 | (not prior measured) → **20-30** | (not prior measured) → **60-80** |

**~15-20× improvement** at the N values where the bug was hitting. Light fixture is now faster than heavy at the same N (fewer DOM nodes per renderer — as expected). Console warnings during pan: **gone**.

**Verdict:** real bug fix, strict improvement. Shipped as **v0.1.4** (patch bump — bug fix, no public API change, no behavior break for callers that didn't use port-bearing data). Per the readiness-review rule, patch bumps don't trigger a review file.

### Implications for the v0.2 description's tier ladder

The description's Tier-1 plausibility ("light: 300-500 nodes smooth") is **already exceeded with just the v0.1.4 bug fix** — vis-on at N=1000 hits 60-80 FPS without any Tier 1 implementation work. Specifically:

- **Tier 1's success criterion** (light ≥50 FPS at N=500): already met by v0.1.4 + vis-on at 80-90 FPS.
- **Tier 1 itself** becomes mostly the default-flip + the `fireOnChange` batching. The headline lever (vis-on) is now demonstrably worthwhile (12-20× improvement at scale).
- **Tier 2's urgency reduces.** `useShallow` + CSS selection still has value, but the light fixture is comfortably smooth at N=1000 already. Tier 2 is for "tighten Tier 1's results at 2k-3k nodes," not "make 1k usable."
- **Tier 3 (canvas edges + LOD) needs a fresh use-case justification.** If 1k light is smooth and 5k heavy is smooth (with culling), what's the user-facing need for Tier 3? Plan stage should re-examine this.

The description doc should NOT be edited (signed off; changes after sign-off are loud and intentional). Instead, the v0.2 Stage 2 **plan doc** captures this updated context as planning inputs.

## Caveats

- Not strict-protocol measurements (no DevTools Performance trace, overlay-only).
- Single machine (CPU/RAM/GPU not logged).
- Single observer (one set of eyes on the overlay).
- N=1000 and N=2000 not measured for heavy.
- Light fixture not measured beyond N=500.

A formal re-measurement against the protocol §3 + §4 should happen before any tier ships, especially to back the v0.2.0 release-notes claim "tested at N=…". This file is "directionally accurate but not citation-grade."

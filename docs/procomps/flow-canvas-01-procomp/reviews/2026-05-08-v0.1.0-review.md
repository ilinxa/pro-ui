## 1. Executive summary

`flow-canvas-01` v0.1.0 is the library's xyflow substrate component — a node-and-edge canvas with three keystone registries (renderers / port types / edge types), recursive port discovery via tree-walker, source-vs-canvas-data export boundary, drop / paste / file-drop pipeline with sub-object extract gesture, typed connection validation (Strict mode + same-`type` + opposite-direction + `multi: false` enforcement), three right-click context menus, and a 200-node stress demo. The implementation honors the canonical xyflow-react-pro footgun fix (module-scope `NODE_TYPES` / `EDGE_TYPES`) and the v0.1 plan's three-stage validator pipeline. Verification: `pnpm tsc --noEmit` clean, `pnpm lint` clean (1 pre-existing rich-card warning), `pnpm registry:build` correctly skips this component (it's not in `registry.json`).

**Headline finding:** Two compounding High-severity issues sit between this component and shipping:
1. **F-01 (carries F-cross-03):** the component is **absent from `registry.json`** — `pnpm dlx shadcn@latest add @ilinxa/flow-canvas-01` returns 404. Smoke harness confirmed `expected-fail` at 3.0s. Until the registry entry is authored, the component exists only on the docs site, never installable by external consumers.
2. **F-02:** `meta.ts.dependencies` significantly under-declares actual usage — claims **0 shadcn primitives** and **only `@xyflow/react`** as npm dep, but the shipped code uses **4 shadcn primitives** (`dialog`, `button`, `textarea`, `context-menu`) and **`lucide-react`**. When F-01 is fixed and F-02 is not, the consumer install will succeed at copy-time but break at compile-time when missing primitives don't resolve.

Seven findings total — three High, three Medium, one Low. No Blockers (F-01 is High because the not-yet-shipped state is an *intended* freeze, not a breakage; the registry entry is a planned follow-up). The `process.env.NODE_ENV` use in 4 files violates description §6 success criterion #10 ("no `process.env`") — bundlers compile out the dev-warn blocks in production, so consumer-runtime impact is zero, but the explicit criterion is violated and worth either fixing or formally relaxing.

**Recommended next step:** v0.1.1 patch addressing F-02 (declared deps) + F-03 (process.env) + F-04 (stale `related: ["force-graph"]`) + F-06 (STATUS file-count claim). Then ship the `registry.json` entries (base + fixtures) per the locked target convention; that closes F-01 + F-cross-03 in one move. Browser validation deferred per the project-wide pre-existing gap.

## 2. Strengths

Locking these in for future components:

- **Canonical xyflow-react-pro footgun fix is correctly applied.** [`parts/canvas.tsx:31-32`](../../../src/registry/components/data/flow-canvas-01/parts/canvas.tsx) — `NODE_TYPES` and `EDGE_TYPES` are module-scope const objects. The skill at `.claude/skills/xyflow-react-pro/` calls this out as the #1 footgun (recreating the type maps in render tears down + remounts every node every render). The implementation gets it right. Worth showcasing.
- **Single dispatch-by-internal-type pattern.** All nodes route through `'ilinxa-node'` xyflow type → `NodeAdapter` → renderer registry lookup by `data.__type`. All edges route through `'ilinxa-edge'` xyflow type → renderer registry lookup by `EdgeRecord.type`. xyflow's nodeTypes/edgeTypes maps stay tiny and stable; consumer extensibility happens at the inner registry layer. Same shape kanban-board-01 and workspace use for renderer dispatch — consistent across host components. Worth promoting in `docs/component-guide.md` as the canonical pattern.
- **Comprehensive isValidConnection.** [`parts/canvas.tsx:116-154`](../../../src/registry/components/data/flow-canvas-01/parts/canvas.tsx) — runs Strict mode + opposite-direction check (`out` → `in`) + same-`type` match + `multi: false` enforcement (counts existing edges at the candidate port). Per-handle override hooks are layered through `usePortType`'s color routing. Plan §3.5 (typed connections) delivered faithfully.
- **Three-MIME drop pipeline + sub-object extraction.** [`hooks/use-drop-pipeline.ts:16-23`](../../../src/registry/components/data/flow-canvas-01/hooks/use-drop-pipeline.ts) accepts `application/json` (primary), `application/reactflow` (xyflow's own sidebar pattern — accepted for compatibility), `text/plain` (fallback), plus `application/x-ilinxa-subobject` (custom MIME for in-canvas drag-extract carrying `parentId|path` metadata). File-drop is also wired. The Alt-key move-vs-copy gesture lands at the drop handler. Paste with editable-target skip (don't steal Cmd+V from focused inputs/textareas/contenteditables). viewport-center fallback for paste position when no mouse coords. All matches plan §3.7.
- **Public helper API is open and consistent.** [`index.ts`](../../../src/registry/components/data/flow-canvas-01/index.ts) re-exports 17 types + 11 helpers/components: `customJsonRenderer`, `findRenderer`, `mergeRenderers`, `useRenderer`, `defaultPortTypes`, `findPortType`, `usePortType`, `defaultEdgeRenderer`, `findEdgeType`, `mergeEdgeTypes`, `useEdgeType`, `PortsAt`, `PortHandle`, `findPortInTree`, `countEdgesAtPort`, `emitSubObjectDrag`. Consumers writing custom renderers, custom edge types, or sub-object drag handlers have access to every primitive the canvas itself uses internally. Solid open-API discipline.
- **Source-vs-canvas-data export boundary lands cleanly.** [`hooks/use-export.ts`](../../../src/registry/components/data/flow-canvas-01/hooks/use-export.ts) exposes the `exportRef.export({ withPorts })` toggle promised in description §2 + §6 #5. The `withPorts: false` path strips ports + edges for clean source export; `true` returns full canvas state.
- **`@xyflow/react` is the only new peer dep.** [`meta.ts:45`](../../../src/registry/components/data/flow-canvas-01/meta.ts) — single explicit version pin (`^12.10.2`, standard semver, not `^N.x`). MIT license — consumer install requires no Pro subscription. Earlier "license note" framing in STATUS.md was wrong (corrected this session — see follow-up section).
- **Read-only mode is a clean toggle.** `interactive = !readOnly` flips drop/paste handlers, `nodesDraggable`, `nodesConnectable`, `elementsSelectable`, and `deleteKeyCode` in one consistent path. Pan + zoom remain enabled (per plan §5.9 and description §2).
- **Stress data exists.** `makeStressData` in `dummy-data.ts` is a 200-node fixture for performance verification (description §6 #8: "60fps on a mid-tier laptop"). The `onlyRenderVisibleElements` toggle is exposed as a prop (description §2) for high-N consumers.

- **Severity:** ⚠️ High
- **Dimension:** 5 (Dependencies)
- **Location:** [`meta.ts:43-47`](../../../src/registry/components/data/flow-canvas-01/meta.ts)
- **Observed:** `meta.ts.dependencies` declares:
  ```

    shadcn: [],
    npm: { "@xyflow/react": "^12.10.2" },
    internal: [],

  dependencies: {
    shadcn: ["button", "context-menu", "dialog", "textarea"],
    npm: {
      "@xyflow/react": "^12.10.2",
      "lucide-react": "^1.11.0",  // match producer's pin
    },
    internal: [],
  }
  ```
  Then verify the corresponding `registryDependencies` in `registry.json` (when authored per F-01's fix). Re-run `pnpm registry:build`. Cross-component coherence: future components must include their full dep declarations; consider adding a `pnpm validate:meta-deps` script that grep-walks each component for imports and cross-checks with meta.

#### F-03 — `process.env.NODE_ENV` use violates description §6 success criterion #10

- **Severity:** ⚠️ High
- **Dimension:** 3 (Component code) + plan-vs-code drift
- **Location:** Four files use `process.env.NODE_ENV`:
  - [`flow-canvas-01.tsx:21`](../../../src/registry/components/data/flow-canvas-01/flow-canvas-01.tsx) — `mergePortTypes` collision warning
  - [`registries/renderer-registry.tsx:29`](../../../src/registry/components/data/flow-canvas-01/registries/renderer-registry.tsx) — renderer collision warning
  - [`registries/edge-type-registry.tsx:27`](../../../src/registry/components/data/flow-canvas-01/registries/edge-type-registry.tsx) — edge-type collision warning
  - [`lib/parse-json.ts:10`](../../../src/registry/components/data/flow-canvas-01/lib/parse-json.ts) — JSON parse-error warning
- **Observed:** Description §6 success criterion #10 reads: *"Portability: zero `next/*` imports, no `process.env`, no app-context coupling."* — explicit prohibition on `process.env`. The plan §5 dependencies block reiterates "no env-specific code". The actual code uses `process.env.NODE_ENV !== "production"` to gate dev-only collision warnings. This is standard React-ecosystem practice and bundlers (esbuild, swc, vite, webpack) compile out the dev branches in production builds — so the runtime impact in a consumer's production bundle is zero. But the criterion is explicit, and the project has other non-`process.env` patterns to choose from.
- **Why it matters:** The criterion was set deliberately for portability; relaxing it silently weakens the project's "no env-specific code" discipline and makes future audits inconsistent. The current dev-warn pattern is reasonable but the criterion violation should be addressed loudly: either (a) update the criterion to allow `process.env.NODE_ENV` for dev-warns specifically, or (b) replace the four uses with an alternative pattern. Silent drift is the wrong outcome.
- **Suggested fix:** Two options — pick one and apply consistently:
  1. **Relax the criterion (recommended).** Update description §6 #10 to: *"Portability: zero `next/*` imports, no app-context coupling, no `process.env` *outside of standard `process.env.NODE_ENV` dev-warn gates*."* Document in plan §5 that dev-warn-only `process.env.NODE_ENV` checks are permitted. This codifies prevailing React-ecosystem practice without surprises.
  2. **Replace with a config flag.** Introduce a project-level `IS_DEV` constant in `@/lib/utils` or similar, computed once via `process.env.NODE_ENV !== "production"` at build time. Components import the constant instead of reading `process.env` directly. Keeps the criterion strict but adds boilerplate.
  
  Either way, all four use sites should be updated together with a one-line plan-doc note.

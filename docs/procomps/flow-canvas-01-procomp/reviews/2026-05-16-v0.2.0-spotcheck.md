#### F-01 — Tier 1+2 success-criteria evidence is directional, not protocol-compliant

- **Severity:** 🔸 Medium
- **Dimension:** Performance (rotating)
- **Location:** plan §3 F-03 (success bars) + plan §6 step 1 (re-baseline measurement)
- **Observed:** Path 1 chosen at sign-off (skip formal re-baseline; use the 2026-05-14 baseline as the comparison floor; ship Tier 1+2; defer the formal 4-cell post-Tier-1+2 DevTools Performance trace). Plan §9 risks explicitly authorize this scenario ("If v0.1.4's accidental gains were so large that Tier 1's default-flip adds <5 FPS at the N matrix, ship anyway — the default-flip is a 'make it free for every consumer' lever, not a 'lift the FPS curve' lever"). Code-level changes match the plan exactly, but the success-criteria evidence at ship-time is not citation-grade — the F-03 bars (light ≥50 FPS at N=2000, heavy ≥50 FPS at N=1000) are inferred from the v0.1.4 numbers + the lever's known scale-dependence, not measured against Tier 1+2 directly.
- **Suggested fix:** schedule a 4-cell formal post-Tier-1+2 measurement (light vis-on N=200, N=2000; heavy vis-on N=200, N=1000) per protocol §3 in a follow-up session. File as `docs/procomps/flow-canvas-01-procomp/research/<date>-tier2-postship.md`. Anti-regression check (N=200 stays ≥ v0.1.4 baseline). Owner: assistant + user. Target version: v0.2.1 patch (or filed as a standalone research artifact without a code bump, if the numbers confirm Tier 1+2's claims and no fix is needed).

## Findings

### F-01 — `useVirtualizer` informational warning

🔹 **Low** — `src/registry/components/navigation/file-tree/hooks/use-tree-virtual.ts:29`

Lint output reports a single warning:

> Compilation Skipped: Use of incompatible library — TanStack Virtual's `useVirtualizer()` API returns functions that cannot be memoized safely

**Evidence:** ESLint `react-hooks/incompatible-library`. The React Compiler skips memoizing the wrapping hook. Functionally correct — virtualizer values flow into `FileTreeRowList` which doesn't memoize them; no stale-UI risk. The warning is an *advisory* about the React Compiler's optimization scope, not a bug.

**Suggested fix:** none required. If we ever wrap a part of the virtualizer's output in `React.memo`, revisit. Document the warning in guide.md so future contributors don't try to "fix" it.

**Severity:** Low — informational only.

### F-02 — `onContainerContextMenu` prop in `FileTreeRowList` is unused

🔹 **Low** — `src/registry/components/navigation/file-tree/parts/file-tree-row-list.tsx`

`FileTreeRowList` declares an `onContainerContextMenu?: (e) => void` prop and forwards it to the scroll container's `onContextMenu`. The orchestrator never passes this prop because right-click is handled by the wrapping shadcn `<ContextMenu>` (which sits *above* the row-list). The prop is dead plumbing.

**Suggested fix:** remove `onContainerContextMenu` from `FileTreeRowListProps` + the two `onContextMenu` bindings in the JSX. Bump-target: **v0.1.1** (non-breaking — purely internal cleanup).

**Severity:** Low — does not affect runtime; clutters the surface.

### F-03 — Drop-target row not visually distinguished when `dropPosition === 'inside'` and dragged ids include the target's parent path

🔸 **Medium** — `src/registry/components/navigation/file-tree/lib/validation.ts` + `src/registry/components/navigation/file-tree/hooks/use-tree-drag.ts`

Self-walked the `isLegalDrop` flow with a contrived test case: drag folder `a/b` onto file `a/b/c.txt` (same parent as descendant). With `position='before'`, `validation.ts` re-runs `isCycle` against the file's parent (`a/b`); since `a/b` IS in `srcIds`, the drop is correctly refused. Verified mentally; not exercised in demo.

**However**, when the drag source is `a/b` and the user hovers over `a` itself (the source's parent), the `dropPosition='inside'` case checks `isCycle([a/b], a, ...)` — walks `a`'s parent chain, doesn't find `a/b`. So the cycle check returns `false` (no cycle), and the drop is *legal* — meaning we'd move `a/b` into its own current parent, which is a no-op move. The tree fires `onMove` for this no-op; consumer's `onMove` handler has to detect and ignore.

**Evidence:** code reading; no exercised demo case.

**Suggested fix:** add a "no-op move" guard: if `position === 'inside'` and `targetId === srcNode.parentId`, OR if `position !== 'inside'` and the drop target is already the next/prev sibling, skip the `onMove` call. Or: document that no-op moves can fire and consumer should detect.

**Severity:** Medium — not a correctness bug (consumer can guard), but the API contract is "we only fire `onMove` for legal *meaningful* drops." Pure no-ops fall in a gray zone. Bump-target: **v0.1.1**.

## Verdict

**Pass with follow-ups.**

The component meets all four core-dimension bars and the chosen rotating-dimension bar (robustness). The two real findings (F-02 dead plumbing, F-03 no-op drop fires `onMove`) are both v0.1.1 fixes, neither blocking. F-01 is informational; F-04 is procedural. None require revision before push.

In the same review session I caught + fixed (no findings logged):
- Roving-tabindex a11y: root had `tabIndex={0}` and a focused row also had `tabIndex={0}` simultaneously. Fixed: root `tabIndex={focusedId === null ? 0 : -1}`.
- `nodeIndex` was being recomputed on every `dragover` (~60Hz). Fixed: memoized at hook entry; callbacks read from the memo.
- `renderRow` slot's outer wrapper was double-nesting around `FileTreeRowView`'s own outer (broke virtualized positioning). Fixed: single wrapping div in slot path; default path passes `outerStyle` directly to the row view.
- Missing `<TooltipProvider>` caused SSR build to fail at `/components/file-tree`. Fixed: wrapped the tree root in `<TooltipProvider delayDuration={400}>`.
- `useTreeState`'s exposed `selectionAnchorId: ref.current` violated the `react-hooks/refs` rule. Fixed: replaced the value field with a `getSelectionAnchorId()` getter.

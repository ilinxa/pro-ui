## Findings (2)

### F-01 — Demo doesn't exercise `onCopySuccess` or `onCopyError` callbacks

- **Severity:** 🔹 Low
- **Dimension:** 4 — Demo + usage
- **Location:** `src/registry/components/marketing/share-bar-01/demo.tsx` (5 sub-tabs, none use `onCopySuccess` / `onCopyError`)
- **Observed:** The component exposes three analytics callbacks: `onShare(targetKind)`, `onCopySuccess()`, and `onCopyError(err)`. The demo's `Full + analytics` tab wires `onShare` to a `lastShared` state and displays the result, but neither `onCopySuccess` nor `onCopyError` is exercised. A consumer browsing `/components/share-bar-01` learns about `onShare` but not the copy-specific lifecycle hooks. The error-path callback is genuinely hard to demo (clipboard rarely fails in modern envs); the success-path callback is easy.
- **Suggested fix:** Either (a) extend `Full + analytics` to also wire `onCopySuccess={() => setLastShared("copy:success")}` (cheap; demonstrates the additional callback alongside `onShare`), or (b) add a sub-tab that subscribes to all three callbacks and shows a small log. Option (a) is the lower-friction fix.

### F-02 — `successResetMs` prop is used for both success AND error reset; name is misleading

- **Severity:** 🔹 Low
- **Dimension:** 1 — Procomp planning docs (API clarity)
- **Location:** `src/registry/components/marketing/share-bar-01/share-bar-01.tsx:91, 95` + `types.ts` (`successResetMs?: number`)
- **Observed:** The prop is named `successResetMs` and the JSDoc presumably describes it as "success state reset duration." But line 95 (`scheduleReset(successResetMs)` inside the `catch` block) means the same prop also controls the error-state reset window. Naming and behavior diverge — a consumer setting `successResetMs={5000}` thinking they're only extending the success-icon visibility is *also* extending the error-icon visibility. Documentation would lock the dual-purpose; renaming to `resetMs` (breaking) or splitting into `successResetMs` + `errorResetMs` (additive, breaking-on-default-only) is v0.2 territory.
- **Suggested fix:** v0.1.0 doc-only fix: update the prop's JSDoc to state "Resets both success AND error visual feedback states after this many ms." v0.2: consider renaming to `resetMs` or splitting. The current behavior is reasonable (one timer for one transient feedback state); the name is the issue.

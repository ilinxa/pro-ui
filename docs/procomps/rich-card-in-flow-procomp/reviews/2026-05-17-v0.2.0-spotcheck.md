### Rotating dimension

- **Chosen:** Public API
- **Why:** v0.2.0's headline change is a brand-new public surface вЂ” `PortEditorStrip` component + `PortEditorPermissions` + `PortField` types. Once a consumer wires it into their dialog, the prop shapes, callback shapes, and default behaviors become load-bearing for v0.3+ compatibility. The dynamicity-primacy memory + the F-cross-12 callback-shape lesson both apply: bar is "plausible consumer override," and "add it later" is a breaking change.
- **Verified?** **yes** (with one follow-up noted under F-02)
- **Notes:**
  - **`PortEditorStrip` props** (per `parts/port-editor-strip.tsx` + plan В§4):
    - `canvas: RichCardCanvasNode` вЂ” required; the node whose ports are being edited. Matches the same intersection type already exported by rcif v0.1.0 (extends `flow-canvas-01`'s NodeRecord + adds `data.tree: RichCardTree`). No widening, no breaking field changes.
    - `onChange: (canvas: RichCardCanvasNode) => void` вЂ” required; consumer-owned. Mirrors flow-canvas-01's existing `onChange` callback shape (full-object passback, not delta). Plan В§4 lock; consistent with the project's controlled-mode pattern memory.
    - `permissions?: PortEditorPermissions` вЂ” optional. Default = all fields allowed for all ports + add/remove allowed. Plan В§5 B2 lock.
    - `className?: string` вЂ” optional cosmetic override. Standard slot.
  - **`PortEditorPermissions` shape** (per `types.ts`):
    - `canAddPort?: (target) => boolean | PortField[]`
    - `canRemovePort?: (target, port) => boolean`
    - `canEditField?: (target, port, field: PortField) => boolean`
    - Three-callback design is **extensible without breaking** вЂ” adding new field values is non-breaking (consumers default to permissive on unknown fields, which the implementation honors via `?? true`).
  - **`PortField` union** = `"id" | "type" | "side" | "dir" | "multi" | "label"` (commit `d880860` B2-review): extracted as a top-level type so consumer `permissions` callbacks can take a typed parameter rather than a magic string. Per-field ports (Q-O4) would add `"field-ref"` вЂ” a non-breaking widening.
  - **Q3 atomic post-save lock** held: "both checked" in add-popover creates 2 atomic port rows with `{base}-in` / `{base}-out` ids; after save, rows are independent. No `groupId` field, no auto-grouping. Consumer can rebuild grouping by `base` prefix if they want.
  - **Q9 uncontrolled-by-design lock** held: `PortEditorStrip` re-reads `canvas.data.tree.ports` on every render вЂ” no `key={canvas.id}` remount required by consumers, no internal state for the port list. Add-popover holds its own draft state (controlled subform).
  - **Doc-port side-enforcement lock** (Q-O1 + Q4): editor-side ONLY. `flow-canvas-01` runtime stays neutral about `doc` ports; rcif's editor disables non-`bottom` side options when type=`doc` and auto-corrects on type change (port-editor-row.tsx:106 `handleTypeChange`). Important: this means consumers using `flow-canvas-01` WITHOUT the editor can still wire doc-ports to any side via their own UI вЂ” by design.
  - **shadcn dep growth (0 в†’ 7)** is the only consumer-visible installer behavior change. Documented in meta.ts + registry.json + procomp guide В§7 + commit message of B3 `3b3b769`. No surprise factor.

## Findings (max 5)

#### F-01 вЂ” shadcn primitive Radix в†’ Base UI divergence between producer and consumer-install

- **Severity:** рџ”ё Medium
- **Dimension:** Verification (fixed core) + Public API (rotating, indirectly вЂ” affects what callback shapes are safe to write)
- **Location:**
  - `src/registry/components/data/rich-card-in-flow/parts/port-editor-strip.tsx:118` (TooltipProvider)
  - `src/registry/components/data/rich-card-in-flow/parts/port-editor-row.tsx:106` + `:206` + `:229` (3 onValueChange handlers)
  - `src/registry/components/data/rich-card-in-flow/parts/port-editor-add-popover.tsx:149` + `:174` (2 onValueChange handlers)
- **Observed:** Producer's `src/components/ui/tooltip.tsx` + `select.tsx` import from `radix-ui` (Radix primitives вЂ” older shadcn era), but `shadcn@4.6.0 add` ships fresh `@base-ui/react/*` primitives to the consumer (current shadcn era, since the project's `components.json` declares `"style": "base-nova"`). Producer's tsc passed clean because producer's primitives still expose the Radix-shape APIs; consumer-side tsc failed because the fresh Base UI primitives have different props/signatures: `TooltipProvider` accepts `delay` (Base UI) but NOT `delayDuration` (Radix); `Select.onValueChange` is `(v: string) => void` on Radix but `(v: string | null, eventDetails: SelectRootChangeEventDetails) => void` on Base UI. This is a producer-side staleness вЂ” the project shifted to `base-nova` style but the primitive files in `src/components/ui/` were never re-added вЂ” and it manifests at every consumer install for every procomp that uses Tooltip or Select with non-trivial props.
- **Suggested fix:** **(closed in B4)** Three source-side fixes shipped in B4 to make rcif robust to BOTH primitive backends via parameter contravariance + dropping the `delayDuration` prop: (1) `<TooltipProvider>` no longer passes `delayDuration={300}` вЂ” accepts default delay on each backend (cosmetic UX divergence; consumer gets backend default, ~700ms on Radix or 0ms on Base UI); (2) 4 `onValueChange` handlers widened to accept `(v: string | null)` with null-short-circuit; (3) `handleTypeChange` function signature widened similarly. Re-smoke confirmed 0 rcif errors. **Broader systemic fix:** the producer-side primitives should be re-added under `base-nova` to eliminate the divergence at root (separate ticket; affects ALL procomps using shadcn primitives, not just rcif). **Promote to `F-cross-13` in `docs/reviews/sweep-tracker.md`** as a cross-cutting finding so future procomp authors know to write defensive callback signatures by default + can audit existing components for the same trap. Owner: assistant. Target version: tracker entry now; primitive refresh = standalone producer-hygiene task (no specific component version).

#### F-02 вЂ” Per-field ports (Q-O4) deferred to v0.3

- **Severity:** рџ”№ Low
- **Dimension:** Public API (rotating)
- **Location:** plan В§3 "Out of scope (v0.3+)" + procomp guide В§7 "Port editor (v0.2 addition)" в†’ end of section
- **Observed:** The user's original ask ("for fields and cards") included field-level ports. v0.2 ships card-level + subcard-level ports only. To add per-field ports without breaking, v0.3 will need: (a) a new port `kind`/`scope` discriminator (e.g., `"card" | "field"`), (b) a `fieldKey` reference for field-level ports, (c) a new value in the `PortField` union ("field-ref" or similar), and (d) edge-source / edge-target resolution that walks tree в†’ card-rcid в†’ field-key. Documented in plan В§3 + procomp guide.
- **Suggested fix:** Open v0.3 plan stage 1 description when user reopens. Plan must thread the discriminator change as a **non-breaking widening** (existing port records implicitly `kind: "card"` via the absence of the field) to preserve v0.2 consumer compatibility. Capture in MEMORY entry `project_active_component_queue.md` as the next-up rcif item. Owner: assistant + user. Target version: rcif v0.3.0.

#### F-03 вЂ” Custom port-type registration in PortEditorStrip's picker deferred to v0.3

- **Severity:** рџ”№ Low
- **Dimension:** Public API (rotating)
- **Location:** `parts/port-editor-add-popover.tsx` + `parts/port-editor-row.tsx` вЂ” both import `defaultPortTypes` from `flow-canvas-01/registries/port-type-registry` as a hardcoded source for the type-picker.
- **Observed:** Today, the type picker offers only `defaultPortTypes` (the 4 built-in types `data, control, error, doc`). Consumers who registered custom port types in `flow-canvas-01`'s `portTypes` prop (e.g., `event`, `streaming`) get NO way to pick them in the editor. The Q5-bis deferral in the plan acknowledges this вЂ” requires either (a) a shared context plumbing from flow-canvas-01 down to rcif's editor, or (b) a `getPortTypes(canvas)` helper export on flow-canvas-01 that consumers can pass into the strip via a new prop. Both paths are non-breaking extensions.
- **Suggested fix:** v0.3 plan should choose path (b) вЂ” explicit `portTypes?: PortTypeDef[]` prop on `PortEditorStrip` (default = `defaultPortTypes`). Consumer-control, no implicit context magic, matches the rest of rcif's API style. Owner: assistant. Target version: rcif v0.3.0 (alongside F-02; same plan).

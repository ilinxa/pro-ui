## Findings (max 5)

#### F-01 — F-S1 shadcn path-rewriter bugs surfaced by smoke; CLOSED pre-ship

- **Severity:** 🔸 Medium (closed)
- **Dimension:** Verification + Public API
- **Location:** [`src/registry/components/data/rich-card-in-flow/index.ts`](../../../../src/registry/components/data/rich-card-in-flow/index.ts), [`parts/rich-card-viewer.tsx`](../../../../src/registry/components/data/rich-card-in-flow/parts/rich-card-viewer.tsx), [`parts/subcard-block.tsx`](../../../../src/registry/components/data/rich-card-in-flow/parts/subcard-block.tsx), [`lib/*.ts`](../../../../src/registry/components/data/rich-card-in-flow/lib), [`dummy-data.ts`](../../../../src/registry/components/data/rich-card-in-flow/dummy-data.ts), [`types.ts`](../../../../src/registry/components/data/rich-card-in-flow/types.ts)
- **Observed:** shadcn 4.6.0's path rewriter has two distinct bugs that hit cross-procomp imports in this procomp:
  1. **`index.ts` cross-procomp re-exports get mis-rewritten.** Observed broken outputs include `@/components/data/rich-card/types` (kept stray `data/`) and `@/lib/update-node-data` (stripped most of path).
  2. **Same-category sibling `<other-slug>/types` imports get the CURRENT slug substituted for the target slug.** From rich-card-in-flow's `parts/`, `@/registry/components/data/flow-canvas-01/types` rewrote to `@/components/rich-card-in-flow/types` (substituted `rich-card-in-flow` for `flow-canvas-01`). NOT seen in json-form (cross-category forms→data); appears specific to same-category siblings.
- **Suggested fix:** **Closed in commit `86cdd05`.** Bug 1 → DROP cross-procomp re-exports from `index.ts`; consumers import directly from each procomp's barrel (`@ilinxa/rich-card`, `@ilinxa/flow-canvas-01`). Bug 2 → use RELATIVE paths for all cross-procomp imports in shipped source (`../flow-canvas-01/types`, `../../rich-card/types`, etc.). Relative paths bypass the alias rewriter and translate verbatim through producer→consumer tree (sibling-procomp relationships preserved at same depth). Documented in procomp guide §9.1 + plan F-V findings + this review for future maintainers. **Promote-to-cross-cutting candidate:** if a future same-category cross-procomp ship trips Bug 2 again, escalate to `sweep-tracker.md` as F-cross-13.

#### F-02 — `isCardLike` heuristic may over-trigger on incidental object shapes

- **Severity:** 🔹 Low (v0.2 candidate)
- **Dimension:** Public API + Component code internals
- **Location:** [`lib/enumerate-subcards.ts:28-36`](../../../../src/registry/components/data/rich-card-in-flow/lib/enumerate-subcards.ts)
- **Observed:** v0.1 heuristic considers a value "card-like" when it carries `__rcid` OR `__rcorder` OR `__rcmeta` OR an own `ports[]` array. A consumer-defined `RichCardJsonNode` with a non-card object value that *incidentally* carries `ports` (e.g. an embedded port-routing config object) or `__rcorder` (e.g. a non-card metadata blob) would render as a subcard outline. Doesn't break — just paints something unintended.
- **Suggested fix:** v0.2 tightening when rich-card ships a canonical "is-card" predicate (currently planned per rich-card v0.5 roadmap). Until then, the heuristic is conservative (requires AT LEAST one rich-card-meta field OR `ports`) and the contract is documented in the Stage 3 guide §6 + footgun §7.4-adjacent. Owner: assistant. Target version: v0.2.0.

#### F-03 — Rapid open/close per-mount cost (G3 in plan); documented trade-off

- **Severity:** 🔹 Low (v0.2 candidate)
- **Dimension:** Component code internals
- **Location:** [`demo.tsx:96`](../../../../src/registry/components/data/rich-card-in-flow/demo.tsx) (`key={editing.nodeId}`) + procomp guide §3 + plan §9
- **Observed:** Switching between editing nodes forces `<RichCard>` to remount via the `key` prop; Plate re-initializes its full editor state machine each open. At rapid-click-through speeds (5+ nodes in quick succession), the user perceives a noticeable delay between dialog open and content paint.
- **Suggested fix:** v0.2 perf polish path requires a `RichCardHandle.setTree(tree)` API on rich-card that doesn't exist today (would allow imperative tree-switching without remounting Plate). File a rich-card v0.5 PR if rapid-switching surfaces as a real consumer pain. Until then: trade-off documented in guide §3; the win (at-most-one editor mounted) outweighs the rapid-switch latency for the canonical use case (one-edit-at-a-time). Owner: assistant + user. Target version: blocked on rich-card v0.5.
